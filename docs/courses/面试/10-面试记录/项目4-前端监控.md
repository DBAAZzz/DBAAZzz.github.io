面试官提问：

### 白屏检测的具体方案是什么？采样检测 DOM 节点还是 MutationObserver？

白屏检测分两步：

时机：`load` 事件触发后（或加一个 delay，比如 load + 1s），此时页面应该已完成首次渲染
检测：用 `elementFromPoint` 对屏幕中心区域取若干个采样点（比如九宫格分布），判断这些点命中的元素是否都是 `body` / `html` / 挂载容器本身——如果是，说明没有实质内容渲染，判定白屏

为什么不用 `MutationObserver`
可以这样说：`MutationObserver` 在 SPA 下会持续触发，噪音大、回调开销高；而白屏本质是一个「快照判断」而非「变化监听」，`elementFromPoint` 在特定时机做一次点位采样，性能更可控，逻辑也更直接。

### 1/10 采样率在什么维度采样？用户维度还是请求维度？怎么保证采样的统计有效性？

是请求维度采样，不是用户维度。SDK 上报时成功接口 rate 设 10、失败接口设 1，每次请求独立执行一次 Math.random() < 1/rate，所以是对每个事件随机决策，不基于 userId 或 sid 做稳定分桶。

### SourceMap 解析服务是怎么部署的？线上环境的 SourceMap 怎么安全管理（不暴露给用户）？

构建时用 `hidden-source-map`，只生成文件不内联引用。构建产物上线后，CI 脚本把 SourceMap 按版本号移到内网专用目录，Nginx 不暴露这个路径。解析服务是 Express + PM2 部署，用 `source-map` 包做解析，错误上报时携带版本号，服务据此找对应文件还原堆栈位置。

### SSE 构建日志推送，连接断开怎么处理？有没有做断线重连和消息补偿？

当前用原生 EventSource 实现，SSE 协议本身支持 retry 字段，服务端可以下发重连间隔，断开后客户端会自动重连，这块是有的。

但消息补偿目前没有做——EventSource 支持 Last-Event-ID，重连时会把上次收到的 id 带给服务端，服务端理论上可以从这个 id 开始补发缺失的日志。

当前项目没有实现这个机制，主要是构建日志是实时流式输出，断线重连后重新订阅拿最新状态对用户影响不大，成本和收益不匹配。
