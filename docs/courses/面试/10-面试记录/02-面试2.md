## 小程序做了哪些性能优化

按编码/编译时与运行时两个维度整理。

### 一、编码/编译时优化（构建前可控）

这些优化在写代码和打包阶段就能确定效果，不依赖用户实际运行行为。

#### 1. 代码包体积治理

| 手段             | 说明                            |
| ---------------- | ------------------------------- |
| 清理无用代码     | 删除废弃页面、组件、样式        |
| Tree-Shaking     | JS/CSS 按需打包，剔除 dead code |
| 静态资源外置 CDN | 图片/视频不进主包，走网络加载   |
| 分包 & 独立分包  | 启动只下载主包，其余按需加载    |
| 部分页面 H5 化   | 包体受限时的降级方案            |

#### 2. 组件架构设计

- 合理拆分组件粒度：避免单个超大组件导致更新影响面过大。
- 但不过度拆分：组件过多会增加包体积和实例化开销。
- 骨架屏策略升级：从“整页骨架”改为“组件内独立骨架”，编码时就设计好谁先 ready 谁先展示。

#### 3. 图片资源编码策略

- 统一使用 WebP 格式。
- 按需裁剪降质，小图标用雪碧图或 iconfont。
- 图片标签加 `lazy-load`。

### 二、运行时优化（执行阶段生效）

这些优化在小程序实际运行过程中发挥作用，解决双线程通信、内存、渲染等实时问题。

#### 1. 启动阶段提速

| 手段             | 说明                                          | 效果                              |
| ---------------- | --------------------------------------------- | --------------------------------- |
| 本地缓存优先渲染 | 先用 Storage 缓存数据起屏，请求成功后增量覆盖 | 消除白屏等待                      |
| 冷启动数据预拉取 | `wx.getBackgroundFetchData` 在启动前取数      | 京喜：`2.5s -> 1s`（约 60% 提速） |
| 跳转前预请求     | `navigateTo` 前发起下一页主接口               | 减少 `onLoad` 后等待              |
| 分包预下载       | 进入 A 页时预下载可能访问的 B 分包            | 避免跳转白屏                      |

#### 2. 渲染性能

- 长列表回收：用 `IntersectionObserver` 监听可视区，不可见节点替换为占位元素。
- 图片懒加载：视口外图片延迟加载。
- 减少节点总量：控制页面总节点数和层级深度。

#### 3. 内存与稳定性

| 问题               | 应对                                                  |
| ------------------ | ----------------------------------------------------- |
| 内存泄漏           | `onHide` 清理定时器/监听器，`onShow` 按需恢复         |
| 内存预警           | 监听 `wx.onMemoryWarning` 上报，定位高风险页面        |
| 高频事件卡顿       | `onPageScroll` 加节流，回调内禁止重计算和大 `setData` |
| iOS WKWebView 回收 | 限制大图策略，长列表分段渲染/回收                     |

## 异常监控系统：如何检测白屏？弱网情况下上报丢失如何处理？什么是AOP切片

我一般按「可发现、可送达、可定位」三件事来回答这个题：白屏检测负责可发现，弱网可靠上报负责可送达，AOP切片负责低侵入采集和可定位。

### 1. 如何检测白屏

白屏本质是“用户视口内没有有效业务内容”，它不是标准 JS 异常，所以要主动探测。

检测策略（分层）：

- 第一层（低成本快速判定）：`load` 后延迟采样，检查根节点（如 `#app`）是否为空。
- 第二层（可视区多点采样）：在中心点 + 四角做 `elementFromPoint`，如果多次采样持续命中 `html/body` 或空容器，判定疑似白屏。
- 第三层（状态交叉验证）：结合骨架屏状态、路由状态、关键接口状态、关键资源加载结果（JS/CSS）做二次确认，降低误报。

误报治理：

- 排除“有意留白”页面：登录过渡页、权限页、纯 loading 页。
- 页面刚切换时不立刻判定：采用 N 次采样窗口（例如 1~3 秒内多次采样）。
- 框架级容器白屏和业务级模块白屏分开统计，避免问题混淆。

上报字段（用于排障）：

- 页面信息：URL、路由、页面可见状态、时间戳。
- 设备与网络：机型、系统、网络类型、是否弱网。
- 运行时上下文：最近的 JS 错误、Promise 错误、资源加载失败记录、长任务信息。
- 页面状态：首屏关键节点是否存在、接口返回状态、骨架屏是否超时未消失。

### 2. 弱网下上报丢失怎么处理

目标不是“100% 实时”，而是“最终可达 + 可去重 + 不阻塞主链路”。

可靠上报方案：

- 本地持久化队列：事件先落 `IndexedDB/localStorage`，每条数据带 `eventId`、重试次数、过期时间、优先级。
- 分级发送：高优先级（崩溃、白屏）即时发；低优先级（行为埋点）批量发。
- 发送通道兜底：优先 `sendBeacon`，其次 `fetch keepalive`，再降级到普通 `fetch/XHR`。
- 触发补偿时机：`online` 事件、页面恢复、应用下次启动、定时巡检时重放队列。
- 重试策略：指数退避 + 抖动（jitter），避免雪崩重试。
- 服务端幂等：按 `eventId` 去重，防止重试导致重复统计。
- 容量治理：队列设上限，按优先级淘汰低价值、过期事件，避免占满本地存储。

### 3. 什么是 AOP 切片（以及在监控里的作用）

AOP（面向切面编程）就是：不改业务代码主体，在关键入口统一“织入”采集逻辑。

在前端监控里的常见切点：

- 错误切点：`window.onerror`、`unhandledrejection`、`addEventListener('error', true)`。
- 请求切点：`fetch` / `XMLHttpRequest`（记录耗时、状态码、失败原因）。
- 路由切点：`history.pushState/replaceState`、`popstate`（记录页面跳转链路）。
- 性能切点：`PerformanceObserver`（LCP、CLS、LongTask）。

AOP 的价值：

- 低侵入：业务不需要到处手写埋点。
- 一致性：采集口径统一、字段统一。
- 可演进：监控策略升级只改一层切面。

AOP 的风险与治理：

- 风险：重复包裹、吞异常、性能开销、与第三方 SDK 冲突。
- 治理：给包装函数打标记防重复；保证原函数返回值和 `this` 不变；采样与限流；灰度启用切面策略。

### 总结（面试收口）

异常监控系统要解决三件事：

1. 能发现：用多点采样 + 状态交叉验证识别白屏。
2. 能送达：用本地队列 + 多通道 + 重试幂等保证弱网可达。
3. 能定位：用 AOP 在错误、请求、路由、性能关键链路统一采集，快速还原现场。

## 前端发布系统流程是怎么做的？支持复杂打包吗？支持灰度吗？

标准发布流程：

1. 代码合并触发 CI（Lint、Test、Build）。
2. 产物上传制品库/CDN，生成版本号与变更记录。
3. 发布系统执行环境部署（测试 -> 预发 -> 生产）。
4. 异常时一键回滚到稳定版本。

支持复杂打包：

- Monorepo + Turborepo：按依赖拓扑编排任务，支持增量构建和远程缓存。
- 多目标产物：按端（H5/小程序）、环境、租户输出差异化构建结果。
- 共享依赖治理：公共包抽离和缓存复用，降低重复构建成本。

支持灰度：

- 支持按比例、用户 ID、租户、地区、端版本等维度灰度。
- 灰度期间实时观测核心指标，超阈值自动熔断或回滚。
- 灰度策略与审批流打通，确保发布可控、可追溯。

## 如果让你设计一个灰度系统你会怎么做？

我会按「控制面 + 执行面 + 观测面」三层设计。

控制面：

- 规则原子化：条件（谁）+ 流量（多少）+ 版本（发哪个）。
- Workflow 编排：创建策略 -> 审批 -> 发布 -> 扩量 -> 全量/回滚。
- 版本与策略解耦：同一版本可挂多策略，便于快速调整。

执行面：

- 网关做规则匹配，使用一致性哈希保证同一用户稳定命中同一版本。
- 规则优先级：白名单 > 黑名单 > 比例灰度 > 默认全量。
- 提供 kill switch，必要时秒级切回稳定版本。

观测面：

- 实时看板：错误率、请求成功率、核心转化率、慢请求占比。
- 阈值报警：超阈值自动停止扩量或触发自动回滚。
- 全链路审计：记录策略变更人、时间和内容，方便复盘。

## 微前端是如何发布的？为什么要选择mirco-app？每个微前端框架的差异是什么？

微前端发布流程：

1. 子应用独立构建，产出静态资源和 `manifest`（版本、入口、依赖）。
2. 主应用从配置中心/注册中心拉取子应用版本映射。
3. 按路由或容器动态加载对应子应用，支持独立升级和回滚。
4. 先灰度子应用，指标稳定后再逐步扩量。

为什么选择 micro-app：

- 接入成本低，使用方式接近组件化，业务团队迁移心智小。
- 子应用改造成本低，能快速落地微前端拆分。
- JS/CSS 隔离能力够用，多实例支持较好。

框架差异（面试回答版）：

- `qiankun`：生态成熟、社区大，适合大型团队协作和复杂场景。
- `micro-app`：API 简单、接入快，适合低改造和快速上线。
- `wujie`：基于 iframe + WebComponent，隔离最强，适合高安全诉求。
- `single-spa`：最底层、最灵活，但需要自行补齐很多工程能力。
