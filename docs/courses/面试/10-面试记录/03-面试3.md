## SSR

### SSR渲染框架有哪些优势

SSR 的核心优势可以从 **SEO、首屏性能、用户体验** 三个维度来讲。

#### SEO 友好

CSR 返回的 HTML 是空壳，内容靠 JS 动态填充。搜索引擎爬虫虽然现在能执行 JS，但执行时机和效果不稳定，Googlebot 以外的爬虫几乎不执行 JS。SSR 直接返回包含完整内容的 HTML，爬虫直接可读。社交分享预览（OG 标签、Twitter Card）也是同理，爬虫不执行 JS，只有 SSR 才能让分享卡片正常展示。

#### 2首屏性能更好

CSR 的关键路径是：`下载 HTML → 下载 JS → 执行 JS → 发起接口请求 → 渲染内容`，用户在 JS 加载完成前看到的是白屏。SSR 是服务端直接拿数据渲染好 HTML 返回，浏览器收到响应就能开始渲染，FCP 和 LCP 指标更好，在**弱网和低端设备**上差异尤其明显。

#### 流式渲染进一步提升体验

现代框架（如 Next.js + React Server Components）支持流式 SSR，服务端可以先发送骨架屏 HTML，后续数据就绪后再通过流式方式补充内容，用户不需要等整个页面渲染完就能看到内容。

**当然也有代价（面试时主动提一下更加分）：**

- 服务端压力更大，需要做缓存策略（页面级缓存、CDN 缓存）
- TTFB 可能更长，因为要等服务端渲染完才响应
- `window`、`document` 等浏览器 API 在服务端不存在，需要做环境判断
- 开发复杂度更高，要处理 hydration、状态同步、数据预取等问题

**选型建议：**

- 内容类/文档类网站（SEO 强需求）→ SSR 首选
- 后台管理系统（无 SEO 诉求）→ CSR 够用
- 电商/营销落地页（SEO + 首屏速度）→ SSR 或 SSG
- 数据实时性强的交互页面 → CSR 或 SSR + CSR 混合

### SSR原理

**一句话概括：** SSR 的本质是把「虚拟 DOM → 真实 DOM」这一步从浏览器搬到服务端，只不过服务端没有 DOM API，所以变成了「虚拟 DOM → HTML 字符串」。

**核心流程分三步：**

1. **服务端渲染**：服务端执行组件渲染函数生成虚拟 DOM 树，然后通过 `renderToString` 将虚拟 DOM 序列化为 HTML 字符串，返回给浏览器。
2. **浏览器直出**：浏览器拿到 HTML 直接渲染出内容（这就是首屏快的原因），此时页面有内容但还不可交互。
3. **客户端 Hydration**：浏览器加载 JS 后，客户端再次生成虚拟 DOM 树，但**不会重新创建真实 DOM**，而是走 Hydration 流程——用客户端的虚拟 DOM 去"对接"已有的真实 DOM，在现有节点上挂载事件监听和响应式系统，让页面变得可交互。

**一个关键点：** Hydration 的前提是服务端和客户端两次生成的虚拟 DOM 结构**必须一致**，否则会出现 hydration mismatch，框架会被迫丢弃服务端 DOM 重新渲染，反而比 CSR 更慢。

**对比总结：** CSR 是「虚拟 DOM → 真实 DOM」全在浏览器完成；SSR 是服务端先做「虚拟 DOM → HTML 字符串」，客户端再通过 Hydration 把虚拟 DOM 和已有 DOM 关联起来。本质上虚拟 DOM 被渲染了两次，换来的是更快的首屏呈现。

### Websocket 有遇到哪些坑吗

**1. Nginx 没配置 Upgrade 导致握手失败**

WebSocket 建立连接依赖 HTTP Upgrade 协议。如果 Nginx 做了反向代理，默认不会转发 `Upgrade` 和 `Connection` 头，握手就会失败，客户端报 `101` 协议升级失败或直接连接拒绝。

```nginx
location /ws {
  proxy_pass http://backend;
  proxy_http_version 1.1;
  proxy_set_header Upgrade $http_upgrade;
  proxy_set_header Connection "upgrade";
  proxy_read_timeout 3600s; # 避免长连接被 Nginx 超时断开
}
```

**2. 心跳机制缺失导致连接假死**

网络中间设备（防火墙、NAT、负载均衡器）会主动断开一段时间内没有数据传输的 TCP 连接，但客户端和服务端感知不到，连接处于"假活"状态。消息发出去没有响应，也没有触发 `onclose`。

解决方式是自己实现心跳：客户端定时发 `ping`，服务端回 `pong`，超时没收到 `pong` 就主动重连。

```js
let heartbeatTimer
let ws

function connect() {
  ws = new WebSocket(url)

  ws.onopen = () => {
    startHeartbeat()
  }

  ws.onclose = () => {
    stopHeartbeat()
    setTimeout(connect, 3000) // 断线重连
  }

  ws.onmessage = (e) => {
    if (e.data === 'pong') resetHeartbeat()
  }
}

function startHeartbeat() {
  heartbeatTimer = setInterval(() => {
    ws.send('ping')
  }, 30000)
}
```

**3. 断线重连没做好，导致连接泄漏**

组件销毁时没有调用 `ws.close()`，重连逻辑在 `onclose` 里无条件触发，结果组件卸载后还在不断重连，内存里堆积了多个 WebSocket 实例。

解决方式：设一个 `shouldReconnect` 标志位，主动关闭时设为 `false`；组件卸载时先置 `false` 再 `close()`。

**4. 负载均衡下的粘性会话问题**

WebSocket 是长连接，同一个连接的消息必须打到同一台服务端实例。如果负载均衡按请求轮询，消息可能被转发到不同实例，导致状态不一致。

解决方式有两种：

- Nginx 配置 `ip_hash` 或 `sticky` 模块，确保同 IP 打到同一实例
- 服务端引入 Redis Pub/Sub，各实例共享消息，避免依赖粘性会话

**5. 跨域问题**

WebSocket 握手本质是一次 HTTP 请求，会受到同源策略约束，但和 CORS 又不完全一样——浏览器不会拦截 WebSocket 连接，但服务端可以主动校验 `Origin` 头拒绝。

实际上 WebSocket 不走 CORS，浏览器会带 `Origin` 但不做预检，所以要在服务端自己校验 `Origin` 白名单。
