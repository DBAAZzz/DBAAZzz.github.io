## taro、uni-app 等跨端框架的实现原理

跨端框架总体原理：

在小程序端，跨端框架基本都遵循“编译成小程序模版 + 运行时做数据/事件/生命周期桥接”这条路线，因为小程序渲染层是封闭的，必须走它的模版和setDate通道。

跨端框架的核心原理可以从三个层面来理解：编译时转换、运行时适配、渲染层差异

### 编译时转换

- 模板转换
- 语法适配
- 样式处理

uni-app 的编译时转换基于 Vue 编译器核心，通过三个阶段实现：解析→转换→代码生成。

核心流程：

```typescript
// 1. 解析：模板 → AST
const ast = baseParse(template, { ...options, parseMode: 'html' })

// 2. 转换：应用各种 transform
const [nodeTransforms, directiveTransforms] = getBaseTransformPreset({...})
const context = transform(ast, {
  nodeTransforms: [...nodeTransforms, ...options.nodeTransforms],
  directiveTransforms: {...directiveTransforms, ...options.directiveTransforms}
})

// 3. 代码生成：AST → 小程序代码
const result = generate(ast, options)
```

转换层次解析：

- 标签转换
- 指令转换
- 双端代码生成：JS render 函数（用于小程序逻辑层）、 WXML 模板（用于小程序视图层）

转换流程：

```bash
1. parse → AST
   Root
   └── Element(view)
       ├── Element(text) + v-if
       └── Element(button) + @click

2. transform → 修改 AST
   - transformTag: div → view
   - transformIf: 提取条件表达式
   - transformOn: click → bindtap
   - transformElement: 处理属性和插值

3. codegen → 双端输出
```

#### 样式处理

uni-app 的样式处理基于 PostCSS 插件体系，在编译时完成各种转换。主要涉及三个部分：单位转换、选择器转换、Scoped CSS。

```js
module.exports = {
  plugins: [
    uniPostcssPlugin(), // uni-app 专用插件
    require('autoprefixer')(), // 自动添加浏览器前缀
  ],
}
```

- 单位转换
- 标签选择器转换
- Scoped CSS 处理
- 深度选择器处理

### 运行时适配

Taro 3 的做法是在小程序端实现一套精简的 DOM/BOM API（document.createElement、appendChild 这些），让 React/Vue 的运行时以为自己在操作真实 DOM，实际上操作的是一个虚拟的数据结构，最后通过 setData 同步给小程序渲染层。

uni-app 则是基于 Vue 的响应式系统，在数据变化时通过自己的渲染层将 VNode 转换成各平台的渲染指令。

uni-app 通过一个精巧的桥接层将 Vue 实例和小程序 Page 连接起来。核心机制分为三个层次：**配置转换**、**实例桥接**和**生命周期映射**。

#### 配置转换：Vue → 小程序 Page

```ts
// packages/uni-mp-core/src/runtime/page.ts
function parsePage(vueOptions, parseOptions) {
  // 复用 parseComponent，因为小程序 Page 和 Component 配置结构类似
  const miniProgramPageOptions = parseComponent(vueOptions, {
    ...parseOptions,
    isPageInProject: true, // 标记为页面
  })

  // 特殊处理 onLoad 生命周期
  methods.onLoad = function (query) {
    // 保存页面参数
    this.options = query
    this.$page = { fullPath: route + query }

    // 调用 Vue 实例的 onLoad 钩子
    return this.$vm && this.$vm.$callHook(ON_LOAD, query)
  }

  // 初始化其他生命周期钩子
  initHooks(methods, PAGE_INIT_HOOKS)
  // ...
}
```

#### 实例桥接：双向引用

Vue 内部实例和小程序实例通过互相关联的方式实现桥接：

```ts
// packages/uni-mp-core/src/runtime/componentInstance.ts
export function initComponentInstance(instance, options) {
  initBaseInstance(instance, options)

  const ctx = instance.ctx

  // Vue 实例 -> 小程序实例
  ctx.$scope = options.mpInstance // 指向小程序 Page 实例
  ctx.$mpPlatform = __PLATFORM__

  // 小程序实例 -> Vue 实例（在 component.ts 中定义）
  mpInstance.$vm = instance.proxy // 小程序实例保存 Vue 实例代理

  // 转发小程序方法
  MP_METHODS.forEach((method) => {
    ctx[method] = function (...args) {
      const mpInstance = ctx.$scope
      return mpInstance[method]?.apply(mpInstance, args)
    }
  })
}
```

#### 生命周期映射

小程序生命周期通过统一的钩子机制映射到 Vue：

```ts
// packages/uni-mp-core/src/runtime/componentHooks.ts
function initHook(mpOptions, hook, excludes) {
  if (!hasOwn(mpOptions, hook)) {
    mpOptions[hook] = function (args) {
      // 核心：调用 Vue 实例的对应生命周期
      return this.$vm && this.$vm.$callHook(hook, args)
    }
  }
}

// packages/uni-mp-core/src/runtime/component.ts
const mpComponentOptions = {
  lifetimes: {
    attached() {
      // 创建 Vue 实例
      this.$vm = createVueInstance()
      // 调用 Vue onMounted
      this.$vm.$callHook('onMounted')
    },
    detached() {
      this.$vm.$callHook('onUnmounted')
    },
  },
  pageLifetimes: {
    show() {
      this.$vm.$callHook('onPageShow')
    },
    hide() {
      this.$vm.$callHook('onPageHide')
    },
    resize(size) {
      this.$vm.$callHook('onPageResize', size)
    },
  },
}
```

### 渲染层差异

- 小程序：框架生成的数据通过 setData 传给小程序原生渲染引擎
- H5：直接操作真实 DOM
- App（uni-app）：通过 weex 或 nvue 走原生渲染，或者 webview 渲染

## Nodejs 框架中间件原理

### 什么是中间件

中间件本质是运行在请求-响应链路上的拦截器函数，核心能力是：

- 在进入业务前做预处理（日志、鉴权、参数校验）
- 在业务后做后处理（耗时统计、统一响应）
- 决定请求流向（继续传递、提前返回、抛错）

常见签名：

- Express：`(req, res, next)`
- Koa：`(ctx, next)`（`next` 返回 Promise）

### Express 和 Koa 的执行模型差异

> 更准确地说：**Koa 是标准异步洋葱模型**，Express 更像**线性责任链**（可通过回调“跳转”到下一个）。

#### Express：线性分发（callback 链）

Express 内部维护一个 middleware stack，通过下标递增分发。`next()` 调用后会立刻进入后续中间件，本身不提供“等待后续完成再回来”的语义。

```js
// 简化版 express 分发逻辑
function dispatch(i) {
  const fn = stack[i]
  if (!fn) return
  fn(req, res, (err) => {
    if (err) return handleError(err)
    dispatch(i + 1)
  })
}
dispatch(0)
```

#### Koa：Promise 组合（异步洋葱）

Koa 使用 `koa-compose` 把中间件组合成 Promise 链，`await next()` 会等待内层执行完，再回到外层，天然形成“进 -> 出”的洋葱结构。

```js
// 简化版 koa-compose
const compose = (mws) => (ctx) => {
  let index = -1
  const dispatch = (i) => {
    if (i <= index)
      return Promise.reject(new Error('next() called multiple times'))
    index = i
    const fn = mws[i]
    if (!fn) return Promise.resolve()
    return Promise.resolve(fn(ctx, () => dispatch(i + 1)))
  }
  return dispatch(0)
}
```

```js
// Koa 典型“前后置”写法
app.use(async (ctx, next) => {
  const start = Date.now()
  await next()
  console.log(`耗时: ${Date.now() - start}ms`)
})
```

### 错误处理机制差异

- Express：通过 `next(err)` 进入错误中间件（四参签名：`(err, req, res, next)`）
- Koa：通常在最外层 `try/catch + await next()` 统一兜底

```js
// Koa 全局错误兜底
app.use(async (ctx, next) => {
  try {
    await next()
  } catch (err) {
    ctx.status = err.status || 500
    ctx.body = { message: err.message || 'Internal Server Error' }
    ctx.app.emit('error', err, ctx)
  }
})
```

### 面试可直接回答（30 秒版本）

Node.js 中间件本质是责任链。Express 是基于 `next()` 的线性分发模型，适合直观串联功能；Koa 通过 Promise 组合实现异步洋葱模型，`await next()` 能做统一前后置控制，错误处理也更集中。两者都能完成日志、鉴权、限流等能力，但在异步流程控制和可组合性上，Koa 更“函数式”。

### 常见追问/易错点

- Koa 里忘记 `await next()`，会导致后续中间件不执行或时序错乱
- Koa 中间件里多次调用 `next()` 会触发异常
- Express 在响应已发送后继续写入，容易出现 `headers already sent`
- 中间件顺序很关键：一般是日志/trace -> 错误兜底 -> bodyParser -> 鉴权 -> 路由 -> 404

## Webpack、Vite、Rollup 打包工具对比

### 术语先说清（面试加分点）

严格来说它们是**构建工具/打包器**，不是业务框架。  
面试里先纠正术语，再展开原理，通常会加分。

### 一句话定位

- Webpack：面向应用的全能型打包器，生态最成熟，能力边界最广
- Vite：开发期极快的构建工具（基于原生 ESM），生产构建默认走 Rollup
- Rollup：面向库构建的打包器，Tree Shaking 和产物纯净度表现突出

### 底层原理差异

#### Webpack：先构建依赖图，再统一打包

核心流程：Entry -> 解析依赖 -> Loader 转换 -> Plugin 扩展 -> Chunk 生成 -> 输出产物

- Loader：把非 JS 资源转成模块（如 ts/css/vue）
- Plugin：基于 Compiler/Compilation 生命周期做能力扩展
- HMR：基于模块依赖图做热更新，但链路相对更重

#### Vite：开发不打包，按需编译；生产再打包

开发阶段：

- 利用浏览器原生 ESM，请求到哪个模块就编译哪个模块
- 第三方依赖预构建（通常由 esbuild 完成）以减少请求和格式转换成本
- HMR 基于模块边界精准失效，更新路径更短

生产阶段：

- 统一走 Rollup 做打包、分包、压缩和产物优化

#### Rollup：基于 ESM 静态分析做“干净打包”

- 天然擅长库模式（多格式输出：ESM/CJS/UMD）
- Tree Shaking 更激进，结合 Scope Hoisting 产物更扁平
- 对应用开发期能力（如开箱即用 dev server 体验）不如 Vite 完整

### 对比表（面试高频维度）

| 维度         | Webpack                | Vite                      | Rollup            |
| ------------ | ---------------------- | ------------------------- | ----------------- |
| 开发启动速度 | 中等（通常需要先构建） | 快（原生 ESM + 按需编译） | 中等（偏构建器）  |
| HMR 体验     | 可用但链路较重         | 快且精准                  | 依赖上层工具补齐  |
| 生产构建能力 | 强（应用场景成熟）     | 强（底层用 Rollup）       | 强（尤其库构建）  |
| 配置复杂度   | 相对高                 | 相对低                    | 中等              |
| 生态与兼容   | 最成熟                 | 快速发展、主流完善        | 插件生态聚焦打包  |
| 更适合       | 大型应用、复杂历史项目 | 现代 Web 应用开发         | SDK/组件库/工具库 |

### 选型建议（可直接答）

- **如果是大型应用且历史包袱重**：优先 Webpack（兼容和可定制能力强）
- **如果是新项目追求开发效率**：优先 Vite（启动快、HMR 快、心智负担低）
- **如果是发 npm 包/组件库**：优先 Rollup（产物更干净、格式输出灵活）

### 高频追问与答法

1. **为什么 Vite 开发快？**  
   因为开发期不做全量打包，而是基于浏览器 ESM 按需编译，依赖再做预构建优化。

2. **Vite 都这么快了，为什么生产还要打包？**  
   生产环境要考虑请求数、缓存策略、代码分割、兼容和压缩，所以仍需要 bundling。

3. **Tree Shaking 生效前提是什么？**  
   依赖 ESM 静态结构（`import/export`）和无副作用标记（如 `sideEffects`）。

4. **React/Vue 项目能直接用 Rollup 吗？**  
   能。做“库项目”（组件库、SDK）很常见；做“应用项目”也能做，但要自己补 dev server、HMR、路由分包等工程能力，通常不如 Vite/Next/Nuxt 省心。

5. **Webpack 能做库构建，为啥实际较少作为首选？**  
   不是没人用，而是相对 Rollup/Vite 更少。主要因为 Webpack 更偏应用打包，库场景下配置更重、产物控制（多格式输出/纯净度）成本更高；同时社区在库构建上更常用 Rollup、Vite library mode、tsup 等方案。

### 误区澄清（面试可主动说）

- “Rollup 只能打包原生 JS”是误区：React/Vue 都可通过插件链支持（TS/JSX/SFC）
- “Webpack 不能打库”是误区：能打，但在库场景下通常不是最省成本的首选
- 选型核心不是“谁更强”，而是“应用优先还是库优先”

### 深挖追问：Rollup 是否被 tsc / unbuild 替代？

结论可以这样说：**Rollup 在“使用入口”上被弱化，但在“底层打包能力”上没有被替代。**

- `tsc`：更适合 bundleless 场景（逐文件输出、类型声明生成），不是完整 bundler
- `unbuild`：更像工程化封装层，帮你减少配置成本，不等于抛弃 Rollup 思路
- 现状：很多团队不再手写 `rollup.config`，而是通过 Vite library mode / unbuild 间接使用打包能力

### 什么时候直接用 Rollup（面试可落地）

- 需要精细控制产物：多入口、多格式（ESM/CJS/UMD）、`preserveModules`、复杂 `external`
- 对包体积和 Tree Shaking 结果非常敏感（SDK、组件库、工具库）
- 需要复杂插件链与构建钩子（自定义 transform、产物后处理）
- 项目已有成熟 Rollup 体系，迁移收益低于维护成本

### 3 句选型法（可直接背）

1. 纯 TS 库、可不打包：优先 `tsc`
2. 常规库、追求省心：优先 `unbuild` / Vite library mode
3. 复杂库、追求极致可控：直接上 Rollup

### 面试 30 秒回答模板

三者本质都是构建工具：Webpack 偏“全能应用打包器”，能力最全但配置偏重；Vite 偏“开发体验优先”，开发期基于原生 ESM 按需编译，启动和热更新很快，生产构建默认走 Rollup；Rollup 偏“库打包器”，ESM 静态分析和 Tree Shaking 更突出。实际选型上，复杂应用偏 Webpack，新项目偏 Vite，SDK/组件库偏 Rollup。

## webpack5对比webpack4做了哪些更新

### 一句话概括

Webpack 5 是一次大版本迭代，核心改进集中在**持久化缓存**、**更彻底的 Tree Shaking**、**模块连邦**和**移除 Node polyfill**。

### 核心更新点

#### 1. 持久化缓存（构建性能提升最明显）

```js
// webpack.config.js
module.exports = {
  cache: {
    type: 'filesystem', // 开启文件系统缓存
    buildDependencies: {
      config: [__filename], // 配置文件变化时失效缓存
    },
  },
}
```

- Webpack 4：缓存只在内存中，每次启动都要重新构建
- Webpack 5：支持持久化到磁盘，二次构建速度提升 **70%+**

#### 2. 更好的长期缓存

```js
optimization: {
  moduleIds: 'deterministic',  // 确定性模块 ID
  chunkIds: 'deterministic',   // 确定性 chunk ID
}
```

- `[contenthash]` 现在基于文件**真实内容**计算
- 只有代码真正变化才会改变 hash，避免无效缓存失效

#### 3. Tree Shaking 增强

```js
// Webpack 5 支持嵌套 Tree Shaking
import { utils } from './module'
utils.foo() // 只有 foo 被打包，utils.bar 会被移除

// 支持 CommonJS 的部分 Tree Shaking
const { pick } = require('lodash')
```

- 嵌套导出分析（innerGraph）
- CommonJS 模块部分支持
- 副作用自动推断

#### 4. 模块联合（Module Federation）

```js
// 远程应用暴露模块
new ModuleFederationPlugin({
  name: 'app1',
  exposes: {
    './Button': './src/Button',
  },
})

// 主应用消费
new ModuleFederationPlugin({
  remotes: {
    app1: 'app1@http://localhost:3001/remoteEntry.js',
  },
})
```

- **微前端核心能力**：多个独立构建在运行时共享模块
- 无需发 npm 包，直接远程加载

#### 5. 原生资产模块（Asset Modules）

```js
// Webpack 4 需要 file-loader / url-loader
// Webpack 5 内置支持
module: {
  rules: [
    {
      test: /\.(png|jpg|gif)$/,
      type: 'asset', // 自动选择 inline 或 resource
      parser: {
        dataUrlCondition: { maxSize: 8 * 1024 },
      },
    },
  ]
}
```

#### 6. 移除 Node.js polyfill（破坏性变更）

```js
// Webpack 4 自动注入 crypto、buffer 等 polyfill
// Webpack 5 需要手动处理
resolve: {
  fallback: {
    crypto: require.resolve('crypto-browserify'),
    buffer: require.resolve('buffer/'),
    // 或者设为 false 表示不需要
    fs: false,
  }
}
```

- 减少了不必要的包体积
- 升级时最常见的报错来源

### 对比表

| 维度           | Webpack 4           | Webpack 5                 |
| -------------- | ------------------- | ------------------------- |
| 缓存           | 仅内存缓存          | 支持持久化文件缓存        |
| 长期缓存       | hash 不稳定         | 确定性 ID + 真实内容 hash |
| Tree Shaking   | 仅 ESM 顶层         | 支持嵌套分析 + 部分 CJS   |
| 资源处理       | 需要 file-loader 等 | 内置 Asset Modules        |
| Node polyfill  | 自动注入            | 默认移除，需手动配置      |
| 微前端         | 无原生支持          | Module Federation         |
| 最低 Node 版本 | 6+                  | 10.13+                    |

### 升级注意事项

1. **Node polyfill 报错**：最常见，按需配置 `resolve.fallback`
2. **插件不兼容**：部分插件 API 变化，需升级到支持 v5 的版本
3. **配置项变化**：如 `NamedModulesPlugin` → `optimization.moduleIds: 'named'`
4. **loader 配置**：资产类 loader 可替换为 Asset Modules

### 面试 30 秒回答

Webpack 5 相比 4 的核心变化：一是**持久化缓存**，构建结果可以缓存到磁盘，二次构建快很多；二是**长期缓存优化**，用确定性 ID 和真实内容 hash，避免无效缓存失效；三是**Tree Shaking 增强**，支持嵌套分析和部分 CJS；四是**模块联合**，微前端场景下多应用可以运行时共享模块；五是**移除自动 Node polyfill**，减小包体积但升级时需要手动处理。

### 高频追问

1. **升级 Webpack 5 遇到最多的问题是什么？**
   Node polyfill 报错。Webpack 5 不再自动注入 crypto、buffer 等，需要在 `resolve.fallback` 手动配置。

2. **持久化缓存怎么开启？**
   配置 `cache: { type: 'filesystem' }`，建议加上 `buildDependencies` 指定配置文件依赖。

3. **Module Federation 和 npm 包的区别？**
   npm 包是构建时依赖，需要发版和安装；MF 是运行时依赖，远程模块可以独立部署和更新，适合微前端。

4. **为什么要移除 Node polyfill？**
   很多 polyfill（如 crypto）体积大且浏览器端很少用到，自动注入会无端增加包体积。

## 是如何搭建MCP架构的

### 什么是 MCP

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，让大模型以**统一标准**调用外部工具和数据源。可以类比为"AI 时代的 USB 接口"——不同厂商的设备（工具）只要遵循同一协议，就能被任意支持 MCP 的 AI 应用使用。

### 核心架构

```text
┌─────────────────────────────────────────────────────────┐
│                      MCP Host                           │
│  （Claude Desktop / VS Code / 自研 AI 应用）             │
│                         │                               │
│              MCP Client（SDK 内置）                      │
└─────────────┬───────────────────────────┬───────────────┘
              │ stdio / HTTP+SSE          │
┌─────────────▼───────────┐   ┌───────────▼───────────────┐
│      MCP Server A       │   │      MCP Server B         │
│  （如：文件系统工具）     │   │  （如：数据库查询工具）    │
└─────────────────────────┘   └───────────────────────────┘
```

**三层角色：**

| 角色   | 职责                            | 示例                     |
| ------ | ------------------------------- | ------------------------ |
| Host   | AI 应用宿主，管理多个 Client    | Claude Desktop、Cursor   |
| Client | 与 Server 建立 1:1 连接         | SDK 内置，开发者无需实现 |
| Server | 暴露具体能力（Tools/Resources） | 自己开发的工具服务       |

### 三大核心能力

```typescript
// 1. Tools - 让模型执行操作（类似 POST）
server.tool('search_docs', { query: z.string() }, async ({ query }) => {
  const results = await searchDatabase(query)
  return { content: [{ type: 'text', text: JSON.stringify(results) }] }
})

// 2. Resources - 暴露数据给模型（类似 GET）
server.resource('config://app', async () => {
  return { contents: [{ uri: 'config://app', text: readConfig() }] }
})

// 3. Prompts - 预定义交互模板
server.prompt('code_review', { code: z.string() }, ({ code }) => ({
  messages: [{ role: 'user', content: `请审查以下代码：\n${code}` }],
}))
```

### 传输方式对比

| 传输方式   | 适用场景         | 优点           | 缺点             |
| ---------- | ---------------- | -------------- | ---------------- |
| stdio      | 本地工具         | 简单、安全     | 仅限本地         |
| HTTP + SSE | 远程服务、多用户 | 可部署、可扩展 | 需处理认证和安全 |

### MCP 面试 30 秒回答

MCP 是 Anthropic 推出的开放协议，让 AI 模型以统一标准调用外部工具。

架构分三层：Host 是 AI 应用宿主，Client 负责通信，Server 暴露具体能力。

搭建时用官方 SDK 创建 McpServer，通过 `server.tool()` 注册工具、`server.resource()` 暴露数据，最后选择 stdio 或 HTTP 传输启动。本地工具用 stdio 简单安全，远程服务用 HTTP+SSE 支持多用户。

### MCP 高频追问

1. **MCP 和 Function Calling 有什么区别？**

   Function Calling 是模型厂商各自的实现，格式不统一；MCP 是跨厂商的开放协议，一次开发到处运行，且支持 Resources 和 Prompts 等更丰富的能力。

2. **stdio 和 HTTP 传输怎么选？**

   本地开发工具、CLI 集成用 stdio，简单安全；需要远程部署、支持多用户、做成 SaaS 服务用 HTTP+SSE。

3. **如何保证 MCP Server 的安全性？**
   - 最小权限原则：只暴露必要的工具
   - 参数校验：用 zod 严格校验输入
   - 远程部署时：加认证（API Key / OAuth）、限流、日志审计

4. **实际项目中用 MCP 做了什么？**

   （根据实际经验回答，例如：）搭建了内部知识库查询工具，让 AI 助手能直接搜索公司文档；或者集成了 Jira/GitLab API，让 AI 能帮忙创建任务和查看 MR 状态。

## 无界、qiankun、micro-app微前端框架是如何实现隔离的

面试场景下回答要点：先讲"隔离维度"，再点名三者的"核心手段"，最后补"优缺点/追问点"。

### 隔离维度（统一口径）

- **JS 隔离**：防止全局变量/原型污染，避免不同子应用互相干扰。
- **样式隔离**：避免 CSS 选择器冲突和样式泄漏。
- **DOM 隔离**：限制子应用只能操作自己的 DOM。
- **路由隔离**：主/子应用路由切换互不打架。
- **资源隔离**：避免脚本/样式重复加载、缓存冲突或污染。

### qiankun 的隔离方式（基于 single-spa）

#### JS 沙箱实现

qiankun 提供三种沙箱模式：

**1. LegacySandbox（单实例代理沙箱）**

```js
// 简化版实现原理
class LegacySandbox {
  constructor() {
    this.addedPropsMap = new Map() // 记录新增属性
    this.modifiedPropsMap = new Map() // 记录修改属性的原值
    this.currentUpdatedPropsMap = new Map() // 记录本次更新的属性

    const fakeWindow = Object.create(null)
    this.proxy = new Proxy(fakeWindow, {
      get: (target, prop) => {
        return prop in target ? target[prop] : window[prop]
      },
      set: (target, prop, value) => {
        const originalValue = window[prop]
        if (!window.hasOwnProperty(prop)) {
          this.addedPropsMap.set(prop, value)
        } else if (!this.modifiedPropsMap.has(prop)) {
          this.modifiedPropsMap.set(prop, originalValue)
        }
        this.currentUpdatedPropsMap.set(prop, value)
        window[prop] = value // 直接修改真实 window
        return true
      },
    })
  }

  // 激活时恢复之前的修改
  active() {
    this.currentUpdatedPropsMap.forEach((v, p) => (window[p] = v))
  }

  // 失活时回滚
  inactive() {
    this.modifiedPropsMap.forEach((v, p) => (window[p] = v))
    this.addedPropsMap.forEach((_, p) => delete window[p])
  }
}
```

**2. ProxySandbox（多实例代理沙箱，默认模式）**

```js
// 简化版实现原理
class ProxySandbox {
  constructor() {
    const fakeWindow = Object.create(null)

    this.proxy = new Proxy(fakeWindow, {
      get: (target, prop) => {
        // 优先从 fakeWindow 读取，否则从真实 window 读取
        return prop in target ? target[prop] : window[prop]
      },
      set: (target, prop, value) => {
        // 所有写操作都落到 fakeWindow，不污染真实 window
        target[prop] = value
        return true
      },
      has: (target, prop) => {
        return prop in target || prop in window
      },
    })
  }
}
```

**关键区别**：

- LegacySandbox：修改真实 window，通过记录变更实现回滚，**仅支持单实例**
- ProxySandbox：所有修改落到 fakeWindow，**支持多实例并行**

**3. SnapshotSandbox（快照沙箱，兼容不支持 Proxy 的浏览器）**

```js
class SnapshotSandbox {
  constructor() {
    this.windowSnapshot = {}
    this.modifyPropsMap = {}
  }

  active() {
    // 激活：记录当前 window 快照
    this.windowSnapshot = {}
    for (const prop in window) {
      if (window.hasOwnProperty(prop)) {
        this.windowSnapshot[prop] = window[prop]
      }
    }
    // 恢复上次的修改
    Object.keys(this.modifyPropsMap).forEach((p) => {
      window[p] = this.modifyPropsMap[p]
    })
  }

  inactive() {
    // 失活：记录变更并恢复快照
    this.modifyPropsMap = {}
    for (const prop in window) {
      if (window.hasOwnProperty(prop)) {
        if (window[prop] !== this.windowSnapshot[prop]) {
          this.modifyPropsMap[prop] = window[prop]
          window[prop] = this.windowSnapshot[prop]
        }
      }
    }
  }
}
```

function rewriteStyle(styleText, appName) {
  // 解析 CSS 规则，为每个选择器添加前缀
  return styleText.replace(/([^{}]+)\{/g, (match, selector) => {
    const prefix = `div[data-qiankun="${appName}"]`

### micro-app 的隔离方式（类 WebComponent 思路）

#### 整体架构

micro-app 使用 CustomElement + Proxy 沙箱的组合：

```html
<!-- 主应用使用 -->
<micro-app
  name="app1"
  url="http://localhost:3001/"
></micro-app>
```

#### JS 沙箱实现

```js
// micro-app 的沙箱核心（简化版）
class Sandbox {
  constructor(appName) {
    this.microWindow = {} // 子应用专属的 window 代理对象
    this.injectedKeys = new Set() // 记录注入的属性

    // 绑定全局对象到沙箱
    this.microWindow.window = this.microWindow
    this.microWindow.self = this.microWindow
    this.microWindow.globalThis = this.microWindow

    this.proxyWindow = new Proxy(this.microWindow, {
      get: (target, key) => {
        // 特殊属性直接返回
        if (key === 'window' || key === 'self') return this.proxyWindow
        // 优先从 microWindow 取，否则从真实 window 取
        if (Reflect.has(target, key)) return Reflect.get(target, key)
        const rawValue = Reflect.get(window, key)
        // 函数需要绑定到原始 window
        return typeof rawValue === 'function' ? rawValue.bind(window) : rawValue
      },
      set: (target, key, value) => {
        this.injectedKeys.add(key)
        return Reflect.set(target, key, value)
      },
    })
  }

  // 执行子应用脚本
  execScript(code) {
    const execFunc = new Function('window', 'self', `with(window){${code}}`)
    execFunc.call(this.proxyWindow, this.proxyWindow, this.proxyWindow)
  }
}
```

#### 样式隔离实现

**默认模式：选择器前缀改写**

```js
// micro-app 样式处理核心逻辑
function scopedCSS(styleText, appName) {
  const prefix = `micro-app[name=${appName}]`

  return styleText.replace(
    /([^{}]+)\{([^{}]+)\}/g,
    (match, selector, rules) => {
      // 处理特殊选择器
      if (selector.includes(':root')) {
        selector = selector.replace(':root', prefix)
      }
      // 处理 body/html 选择器
      if (/^(body|html)/.test(selector.trim())) {
        selector = selector.replace(/^(body|html)/, prefix)
      }
      // 普通选择器添加前缀
      const scopedSelector = selector
        .split(',')
        .map((s) => `${prefix} ${s.trim()}`)
        .join(',')

      return `${scopedSelector}{${rules}}`
    },
  )
}
```

**disableScopecss 模式**：关闭样式隔离

### 无界（Wujie）的隔离方式（iframe + WebComponent）

#### 核心架构：JS 与 UI 分离

无界的独特之处在于将 JS 运行环境和 UI 渲染分离：

```text
┌─────────────────────────────────────────┐
│              主应用                      │
│  ┌───────────────────────────────────┐  │
│  │     WebComponent 容器              │  │
│  │  （子应用 UI 渲染在这里）          │  │
│  └───────────────────────────────────┘  │
│                    ↑ DOM 代理           │
│  ┌───────────────────────────────────┐  │
│  │     隐藏的 iframe                  │  │
│  │  （子应用 JS 在这里执行）          │  │
│  └───────────────────────────────────┘  │
└─────────────────────────────────────────┘
```

#### JS 隔离：iframe 天然沙箱

```js
// 无界创建 iframe 的简化逻辑
function createIframe(url, appName) {
  const iframe = document.createElement('iframe')
  // 设置空白页，避免跨域问题
  iframe.src = 'about:blank'
  iframe.style.display = 'none'
  document.body.appendChild(iframe)

  const iframeWindow = iframe.contentWindow

  // iframe 内的 window 天然隔离
  // 子应用的所有全局变量都在 iframe 内，不会污染主应用
  return iframeWindow
}
```

**iframe 沙箱的优势**：

- 天然隔离：全局变量、原型链、定时器等完全独立
- 无需 Proxy：浏览器原生支持，兼容性最好
- 安全性高：同源策略保护

#### DOM 代理：渲染到主应用

子应用的 DOM 操作需要代理到主应用的 WebComponent 容器：

```js
// 无界 DOM 代理核心逻辑（简化版）
function patchDocument(iframeWindow, shadowRoot) {
  // 代理 document 操作到 shadowRoot
  Object.defineProperty(iframeWindow, 'document', {
    get() {
      return new Proxy(iframeWindow.document, {
        get(target, prop) {
          // body/head 等指向 shadowRoot 内的对应元素
          if (prop === 'body') return shadowRoot.body
          if (prop === 'head') return shadowRoot.head

          // DOM 查询方法代理到 shadowRoot
          if (prop === 'querySelector') {
            return (selector) => shadowRoot.querySelector(selector)
          }
          if (prop === 'querySelectorAll') {
            return (selector) => shadowRoot.querySelectorAll(selector)
          }

          return Reflect.get(target, prop)
        },
      })
    },
  })
}
```

### 面试可用的一句话总结

- **qiankun**：Proxy 沙箱 + CSS 作用域/Shadow DOM，偏"运行时改写"，生态最成熟。
- **micro-app**：WebComponent 容器 + Proxy 沙箱 + CSS 选择器改写，偏"组件化封装"，接入最简单。

### 常见追问与风险点

- **样式隔离的缺陷**：选择器改写会影响第三方库（如 antd 的全局样式），Shadow DOM 会屏蔽全局样式（如字体、CSS 变量）。
- **JS 沙箱的边界**：需要处理 `setInterval`/事件监听/全局副作用回收；原型链污染（如 `Array.prototype.xxx = ...`）较难完全隔离。
- **iframe 的代价**：通信成本、内存占用和跨域限制，但隔离最强。

- **追求最强隔离**：无界（iframe 天然隔离，适合对安全性要求高的场景）。

- **追求轻量易用**：micro-app（接入成本低、API 简单，类似使用组件）。

### 高频追问答题点（可背）

**1. 为什么要做 JS 沙箱？**

防止子应用污染主应用/其他子应用的全局变量和原型链。例如子应用设置 `window.Vue = Vue2`，会影响使用 Vue3 的主应用。

**2. Proxy 沙箱和快照沙箱的区别？**

- Proxy 沙箱：通过 `Proxy` 拦截，支持多实例，修改不影响真实 window
- 快照沙箱：激活时记录 window 快照，失活时恢复，仅支持单实例，兼容老浏览器

**3. 为什么样式隔离不总是完美？**

- 选择器改写：对 `:root`、`body`、复杂选择器、第三方库样式处理不稳定
- Shadow DOM：会屏蔽全局样式（字体、CSS 变量、reset 样式），第三方弹窗可能渲染到 Shadow DOM 外

**4. 为什么 iframe 也有人不用？**

- 通信（`postMessage`）成本高，需序列化
- 内存占用大，每个 iframe 都是独立的 browsing context
- 跨域调试麻烦，需要配置 CORS
- 首屏性能略差（需要创建完整的文档环境）

**5. 如何保证卸载干净？**

统一记录副作用（计时器、事件、DOM 插入、全局变量），卸载时集中回收：

```js
// 伪代码
class SideEffectCollector {
  intervals = []
  timeouts = []
  listeners = []

  clear() {
    this.intervals.forEach(clearInterval)
    this.timeouts.forEach(clearTimeout)
    this.listeners.forEach(([el, type, fn]) => el.removeEventListener(type, fn))
  }
}
```

**6. 无界为什么要把 JS 和 UI 分开？**

- JS 在 iframe：获得最强的 JS 隔离能力
- UI 在 Shadow DOM：避免 iframe 的布局限制，支持弹窗等需要脱离容器的 UI
- 通过 DOM 代理连接两者：子应用以为在操作自己的 document，实际操作的是主应用的 Shadow DOM

## 为啥要用 monorepo，monorepo 解决了什么问题

**一句话定义**：把多个强相关的项目/包放在同一个仓库里，用统一的依赖、工具链和流程管理，适合“多包协作 + 高频联动变更”的场景。

**面试答题思路（先价值，后代价）**：

**1. 解决的核心问题**

- **跨包联动难**：多仓库改动需要多次 PR、版本对齐、手动发布，容易错配。monorepo 支持**原子提交**，一次提交改完所有相关包。
- **依赖一致性差**：多仓库容易出现不同版本依赖、重复安装。monorepo 通常只有一个 lockfile，**版本一致、依赖可去重**。
- **代码复用成本高**：共享工具库、组件库在多仓库需要单独发布与维护。monorepo 能让**共享包就地依赖**，开发体验更顺。
- **工程规范难统一**：ESLint/TSConfig/构建脚本、测试基建容易分裂。monorepo 支持**统一工程规范与脚手架**。
- **协作效率低**：跨团队协作需要在多个仓库里切换上下文。monorepo **降低上下文切换成本**，定位问题更快。

**2. 典型适用场景**

- 多个前端应用 + 共享组件库/工具库
- BFF/Node 服务 + 共享中间件/SDK
- 需要统一发布节奏/一致规范的中大型团队

**3. 代价与风险（要能说出来）**

- **仓库变大**：克隆/拉取/索引成本增加，需要工具支持（如稀疏检出、增量构建）。
- **工具链复杂**：需要 workspace、依赖边界、构建缓存等机制，否则会“更乱”。
- **权限与隔离**：代码都在一起，权限粒度更粗，需要配合 CODEOWNERS、CI 规则。
- **构建压力**：全量构建会慢，必须做 **affected build/test**。

**4. 落地方式（点到即可）**

- **包管理**：`pnpm/yarn/npm workspaces` 统一依赖与 hoisting
- **构建/任务编排**：Nx / Turborepo 做缓存、增量、任务依赖图
- **版本管理**：固定版本（全仓同版）或独立版本（changesets）

**简洁答法示例**：

“monorepo 主要解决多仓库在依赖一致性、跨包协作、工程规范统一上的问题。它让跨包改动可以原子提交、共享包可以就地依赖，同时配合增量构建提升 CI 效率。但代价是仓库变大、工具链复杂，所以更适合多包高频联动的团队。”
