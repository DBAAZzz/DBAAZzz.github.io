## JavaScript 内存模型概述

JavaScript 引擎使用栈（Stack）和堆（Heap）两种数据结构来管理内存。理解两者的区别和使用场景是掌握 JavaScript 内存管理的基础。

---

## 栈内存 (Stack)

### 特点

- **数据结构**：后进先出（LIFO, Last In First Out）
- **存储内容**：
  - 基本类型（Primitive Types）：`number`、`string`、`boolean`、`null`、`undefined`、`symbol`、`bigint`
  - 执行上下文（Execution Context）
  - 对象的引用地址（指针）
- **内存分配**：编译时确定，自动分配和释放
- **访问速度**：极快，CPU 可通过寄存器直接访问
- **空间限制**：固定大小，通常较小（约 1MB，不同浏览器可能不同）
- **生命周期**：跟随执行上下文，函数执行完毕后自动释放

### 栈溢出（Stack Overflow）

当函数递归调用层级过深或创建过多局部变量时，会导致栈溢出：

```javascript
function recursion() {
  recursion(); // 无限递归，最终导致 Maximum call stack size exceeded
}
```

---

## 堆内存 (Heap)

### 特点

- **数据结构**：无序的动态内存空间
- **存储内容**：
  - 引用类型（Reference Types）：`Object`、`Array`、`Function`、`Date`、`RegExp` 等
  - 闭包捕获的变量
- **内存分配**：运行时动态分配
- **访问速度**：较慢，需要通过引用地址间接访问
- **空间限制**：动态且较大（通常受限于系统可用内存）
- **生命周期**：由垃圾回收器（GC）管理，当对象不可达时被回收

### 垃圾回收机制

现代 JavaScript 引擎（如 V8）主要使用以下算法：

1. **标记清除（Mark-and-Sweep）**：标记所有可达对象，清除未标记的对象
2. **分代收集（Generational Collection）**：
   - 新生代（Young Generation）：存储生命周期短的对象，使用 Scavenge 算法
   - 老生代（Old Generation）：存储生命周期长的对象，使用标记清除和标记整理
3. **增量标记（Incremental Marking）**：避免长时间的 GC 停顿

---

## 内存分配机制对比

| 特性         | 栈内存（Stack）                  | 堆内存（Heap）            |
| :----------- | :------------------------------- | :------------------------ |
| **分配方式** | 编译时确定，自动分配             | 运行时动态分配            |
| **存储数据** | 基本类型值、引用地址、执行上下文 | 引用类型对象              |
| **内存大小** | 固定且较小（~1MB）               | 动态且较大（受系统限制）  |
| **访问效率** | O(1)，直接访问                   | O(1)，通过引用间接访问    |
| **内存释放** | 自动，执行上下文销毁时释放       | GC 管理，对象不可达时回收 |
| **内存碎片** | 无碎片，连续分配                 | 可能产生碎片，需要整理    |
| **线程安全** | 每个线程独立                     | 共享（需考虑并发问题）    |
| **典型错误** | Stack Overflow（栈溢出）         | Memory Leak（内存泄漏）   |

## 数据存储示例

### 基本类型的存储

```javascript
let a = 10;
let b = a;
b = 20;
console.log(a); // 10
```

**内存分析**：

- `a` 和 `b` 都在栈上存储，值独立
- 修改 `b` 不影响 `a`

### 引用类型的存储

```javascript
let obj1 = { name: "Alice" };
let obj2 = obj1;
obj2.name = "Bob";
console.log(obj1.name); // 'Bob'
```

**内存分析**：

- 对象 `{ name: 'Alice' }` 存储在堆中
- `obj1` 和 `obj2` 在栈中存储同一个堆地址
- 修改 `obj2.name` 实际修改的是堆中的对象，`obj1` 也会受影响

### 内存布局详解

```javascript
function execute() {
  let age = 30; // 基本类型
  let name = "Alice"; // 基本类型
  let user = {
    // 引用类型
    id: 1,
    email: "alice@example.com",
  };
  let anotherUser = user; // 浅拷贝引用
}

execute();
```

**内存布局**：

```
栈内存（Stack）                        堆内存（Heap）
┌────────────────────────┐          ┌──────────────────────────────┐
│ execute() 执行上下文    │          │                              │
├────────────────────────┤          │  0x1001A:                    │
│ age: 30                │          │  {                           │
│ name: 'Alice'          │          │    id: 1,                    │
│ user: 0x1001A          ├──────────►    email: 'alice@...'        │
│ anotherUser: 0x1001A   ├──────────►  }                           │
└────────────────────────┘          └──────────────────────────────┘
        ↓
  函数执行完毕，栈帧弹出                 等待 GC 回收（引用计数为 0）
```

## 高级特性

### 1. 闭包与堆内存

闭包会导致外层函数的变量保存在堆中，不会随执行上下文销毁：

```javascript
function createCounter() {
  let count = 0; // count 被闭包捕获，存储在堆中
  return function () {
    return ++count;
  };
}

const counter = createCounter();
console.log(counter()); // 1
console.log(counter()); // 2
```

**内存分析**：

- `count` 原本应在 `createCounter` 执行完毕后销毁
- 由于返回的函数引用了 `count`，`count` 被保存在堆中的闭包作用域对象中

### 2. Class 静态属性的存储

**静态属性存储在堆内存中**。

```javascript
class User {
  static userCount = 0; // 静态属性
  constructor(name) {
    this.name = name; // 实例属性
    User.userCount++;
  }
}
```

**原理**：

- `class User` 本质是构造函数对象，存储在堆中
- 静态属性 `userCount` 作为 `User` 函数对象的属性，存储在堆中
- 实例属性 `name` 存储在每个实例对象中（也在堆中）

```javascript
// 等价于
function User(name) {
  this.name = name;
  User.userCount++;
}
User.userCount = 0; // 附加在函数对象上，存储在堆中
```

**访问路径**：

- `User.userCount`：栈中保存 `User` 函数对象的引用 → 堆中的 `User` 对象 → `userCount` 属性
- `instance.name`：栈中保存实例的引用 → 堆中的实例对象 → `name` 属性

## 性能优化建议

### 1. 避免内存泄漏

```javascript
// ❌ 不当使用：全局变量导致内存泄漏
window.cache = [];
function addData(data) {
  window.cache.push(data); // cache 永远不会被回收
}

// ✅ 正确使用：及时清理
let cache = [];
function addData(data) {
  cache.push(data);
}
function clearCache() {
  cache = null; // 让 GC 能够回收
}
```

### 2. 减少堆分配

```javascript
// ❌ 频繁创建对象
for (let i = 0; i < 10000; i++) {
  const obj = { value: i };
  process(obj);
}

// ✅ 对象复用
const reusableObj = { value: 0 };
for (let i = 0; i < 10000; i++) {
  reusableObj.value = i;
  process(reusableObj);
}
```

### 3. 注意闭包陷阱

```javascript
// ❌ 闭包持有大对象
function process(largeArray) {
  return function () {
    console.log(largeArray.length); // 整个 largeArray 被闭包持有
  };
}

// ✅ 只保留必要数据
function process(largeArray) {
  const length = largeArray.length; // 只保存需要的值
  return function () {
    console.log(length);
  };
}
```

## 面试常见问题

### Q1: 为什么基本类型比引用类型访问更快？

**A**: 基本类型存储在栈中，CPU 可以直接通过栈指针访问，且栈内存是连续的，缓存命中率高。引用类型需要先从栈中读取引用地址，再根据地址访问堆中的数据，多了一次间接寻址。

### Q2: 字符串是基本类型还是引用类型？

**A**: JavaScript 中的字符串是基本类型，但在 V8 引擎的实现中，短字符串（≤15 字节）直接存储在栈或对象内部，长字符串存储在堆中，通过引用访问。这是引擎的优化，不影响字符串的基本类型语义（不可变性、值传递）。

### Q3: 如何判断是否发生了内存泄漏？

**A**:

- 使用 Chrome DevTools 的 Memory 面板进行 Heap Snapshot 对比
- 监控 Performance Monitor 中的 JS heap size
- 使用 `performance.memory` API 监控内存使用
- 常见内存泄漏场景：
  - 未清理的定时器
  - 未解绑的事件监听器
  - 闭包中持有的大对象
  - 全局变量滥用
  - DOM 引用未清理

### Q4: 垃圾回收会影响性能吗？

**A**: 会。GC 在执行时会产生"Stop-The-World"停顿，阻塞 JavaScript 执行。现代引擎通过增量标记、并发标记、延迟清理等技术减少停顿时间。建议：

- 减少对象创建频率
- 使用对象池复用对象
- 避免创建大量短生命周期对象
- 在关键路径上避免触发 GC（如动画执行时）

---

## 总结

- **栈内存**：存储基本类型和引用地址，生命周期短，自动管理，访问快速
- **堆内存**：存储引用类型对象，生命周期由 GC 管理，访问较慢但空间大
- **引用传递**：对象赋值时传递的是堆地址，修改会影响所有引用
- **内存优化**：避免内存泄漏、减少堆分配、注意闭包使用
- **理解内存模型**是编写高性能 JavaScript 代码的基础
