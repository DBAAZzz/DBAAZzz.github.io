## 基础类型与引用类型：

### JS 基础类型（如字符串）为什么能调用方法（包装对象）？

JS 的基础类型（`string`、`number`、`boolean`）本身是没有方法的，但当你访问它们的属性或方法时，JS 引擎会**自动装箱**——临时把基础值包装成对应的包装对象（`String`、`Number`、`Boolean`），调用完方法后再立即销毁。

```js
'hello'.toUpperCase()
// 等价于：
new String('hello').toUpperCase() // 临时对象，用完即销毁
```

这也是为什么给字符串赋值属性不会报错，但也不会保留——因为每次访问都是新的临时对象：

```js
const s = 'hello'
s.foo = 1 // 临时包装对象上设置，立刻销毁
console.log(s.foo) // undefined
```

`null` 和 `undefined` 没有对应包装对象，所以访问属性会直接报错。

### 如何判断 this 指向？箭头函数与普通函数的区别？

**普通函数的 this 由调用方式决定（运行时绑定）：**

| 调用方式                                          | this 指向                          |
| ------------------------------------------------- | ---------------------------------- |
| 直接调用 `fn()`                                   | 全局对象（严格模式下 `undefined`） |
| 方法调用 `obj.fn()`                               | `obj`                              |
| `new fn()`                                        | 新创建的实例对象                   |
| `fn.call(ctx)` / `fn.apply(ctx)` / `fn.bind(ctx)` | 指定的 `ctx`                       |

**箭头函数没有自己的 this**，它的 this 是定义时所在词法作用域的 this，不会被 `call/apply/bind` 改变，也不能作为构造函数。

```js
const obj = {
  name: 'A',
  fn() {
    const arrow = () => console.log(this.name) // this 是 obj
    arrow()
  },
}
obj.fn() // 'A'
```

**记忆口诀：普通函数看调用，箭头函数看定义。**

### 实现一个深拷贝函数（处理循环引用、复杂数据结构）。

核心思路：递归 + `WeakMap` 记录已拷贝对象（处理循环引用）。

```js
function deepClone(target, map = new WeakMap()) {
  // 基础类型直接返回
  if (target === null || typeof target !== 'object') return target

  // 处理循环引用
  if (map.has(target)) return map.get(target)

  // 处理特殊类型
  if (target instanceof Date) return new Date(target)
  if (target instanceof RegExp) return new RegExp(target)

  // 创建对应类型的空对象（保留原型链）
  const clone = Array.isArray(target) ? [] : {}
  map.set(target, clone)

  // 递归拷贝
  for (const key of Reflect.ownKeys(target)) {
    clone[key] = deepClone(target[key], map)
  }

  return clone
}
```

**关键点：**

- `WeakMap` 而不是 `Map`：key 是弱引用，不影响垃圾回收
- `Reflect.ownKeys` 而不是 `Object.keys`：能拿到 Symbol 类型的 key
- 特殊对象（Date、RegExp）需单独处理，直接递归会变成空对象 `{}`
- 函数不需要拷贝，直接返回引用

## 原型与闭包：

### 谈谈对原型链的理解。

每个对象都有一个内部属性 `[[Prototype]]`（可通过 `__proto__` 或 `Object.getPrototypeOf` 访问），指向它的**原型对象**。当访问一个属性时，如果对象自身没有，就会沿着原型链向上查找，直到找到或到达 `null` 为止。

```text
实例对象 → 构造函数.prototype → Object.prototype → null
```

关键规则：

- 函数都有 `prototype` 属性，`new` 出来的实例的 `__proto__` 指向该 `prototype`
- `Object.prototype.__proto__` 是 `null`，这是原型链的终点
- 方法查找走原型链，属性赋值不走原型链（只在自身创建）

```js
function Person(name) {
  this.name = name
}
Person.prototype.sayHi = function () {
  console.log(this.name)
}

const p = new Person('A')
p.sayHi() // 自身没有，往原型链找到 Person.prototype.sayHi
```

**原型链也是实现继承的基础。**

### 什么是闭包？它的应用场景和潜在问题（内存泄漏）是什么？

**什么是闭包：** 函数在定义时捕获了其外部作用域的变量，即使外部函数已经执行完毕，这些变量依然被内部函数持有，这个内部函数就是闭包。

```js
function counter() {
  let count = 0
  return function () {
    return ++count
  }
}
const add = counter()
add() // 1
add() // 2  — count 没有被回收
```

**应用场景：**

- **数据私有化**：模块模式，对外只暴露接口，内部变量不可直接访问
- **函数工厂**：根据参数生成定制化函数（如 `makeAdder(5)`）
- **缓存/记忆化**：闭包保存计算结果，避免重复计算
- **防抖、节流**：闭包维护 timer 状态

**内存泄漏风险：** 闭包持有对外部变量的引用，导致这些变量无法被 GC 回收。常见场景：

- 事件监听器里的闭包引用了大对象，但监听器没有被移除
- 全局变量持有闭包，导致整条引用链无法回收

**排查方式：** Chrome DevTools → Memory → Heap Snapshot，观察对象是否持续增长。

## 异步编程 (核心)：

### Promise.all、Promise.allSettled 的原理与手写实现。

**区别：**

- `Promise.all`：任意一个 reject 就整体 reject，全部 resolve 才 resolve
- `Promise.allSettled`：等所有 Promise 落定（无论成功失败），返回每个结果的状态描述

**手写 Promise.all：**

```js
function promiseAll(promises) {
  return new Promise((resolve, reject) => {
    const results = []
    let count = 0
    const total = promises.length

    if (total === 0) return resolve([])

    promises.forEach((p, i) => {
      Promise.resolve(p)
        .then((val) => {
          results[i] = val
          count++
          if (count === total) resolve(results)
        })
        .catch(reject) // 任意一个失败就直接 reject
    })
  })
}
```

**手写 Promise.allSettled：**

```js
function promiseAllSettled(promises) {
  return Promise.all(
    promises.map((p) =>
      Promise.resolve(p)
        .then((value) => ({ status: 'fulfilled', value }))
        .catch((reason) => ({ status: 'rejected', reason })),
    ),
  )
}
```

**注意：** `forEach` 遍历时要用 `results[i] = val` 保证顺序，不能用 `push`。

### async/await 的执行机制？如果 await 一个普通值会怎样？

**执行机制：** `async/await` 是 Generator + Promise 的语法糖。

执行到 `await` 时，函数暂停，把 `await` 后面的值包装成 Promise，等 Promise resolve 后，将结果作为 `await` 表达式的值，然后以**微任务**的形式恢复函数执行。

```js
async function fn() {
  console.log(1)
  const result = await fetch(...)  // 暂停，注册微任务
  console.log(2)  // Promise resolve 后，以微任务恢复
}
fn()
console.log(3)
// 输出顺序：1 → 3 → 2
```

**await 普通值：** 会被 `Promise.resolve()` 包装，相当于 `await Promise.resolve(value)`。函数依然会**暂停一个微任务**，然后立即拿到该值继续执行。

```js
async function fn() {
  const x = await 42 // 等同于 await Promise.resolve(42)
  console.log(x) // 42
}
```

### 浏览器与 Node.js 的事件循环 (Event Loop) 机制有什么区别？

**浏览器事件循环：**

每轮循环（每个宏任务）结束后，清空所有微任务队列，然后可能进行 UI 渲染，再执行下一个宏任务。结构相对简单：宏任务 → 微任务 → 渲染 → 宏任务…

**Node.js 事件循环：**

基于 libuv，有明确的**6个阶段**，按顺序轮转：

```text
timers → pending callbacks → idle/prepare → poll → check → close callbacks
```

- **timers**：执行 `setTimeout`、`setInterval` 回调
- **poll**：获取 I/O 事件，执行 I/O 回调（核心阶段，会在此等待）
- **check**：执行 `setImmediate` 回调
- **close callbacks**：关闭事件（如 `socket.on('close')`）

**Node.js 的微任务：** `process.nextTick` 的优先级高于 `Promise`，会在每个阶段切换时优先清空。

**核心区别：**

|          | 浏览器                   | Node.js                            |
| -------- | ------------------------ | ---------------------------------- |
| 架构     | 单队列轮转               | 多阶段轮转（6 个阶段）             |
| 独有 API | -                        | `setImmediate`、`process.nextTick` |
| 微任务   | Promise → queueMicrotask | nextTick > Promise                 |
| UI 渲染  | 每轮宏任务后可能渲染     | 无                                 |

### 微任务（Microtask）与宏任务（Macrotask）的执行顺序？UI 渲染属于哪种？

**常见分类：**

| 类型   | 代表 API                                                                         |
| ------ | -------------------------------------------------------------------------------- |
| 宏任务 | `setTimeout`、`setInterval`、`MessageChannel`、I/O、`setImmediate`（Node）       |
| 微任务 | `Promise.then`、`MutationObserver`、`queueMicrotask`、`process.nextTick`（Node） |

**执行顺序：**

1. 执行当前宏任务（同步代码本身就是第一个宏任务）
2. 清空微任务队列（全部执行完，包括执行过程中新增的微任务）
3. 浏览器判断是否需要 UI 渲染，渲染完再进行下一个宏任务
4. 取下一个宏任务，重复上述流程

**UI 渲染：** 既不是宏任务也不是微任务，它发生在**微任务清空之后、下一个宏任务之前**（浏览器自行决定是否渲染）。

**典型面试题顺序：**

```js
console.log(1)
setTimeout(() => console.log(2), 0)
Promise.resolve().then(() => console.log(3))
console.log(4)
// 输出：1 → 4 → 3 → 2
```

### 如何实现一个异步任务调度器/并发控制器？

控制最多同时执行 N 个异步任务，多余的任务排队等待。

```js
class Scheduler {
  constructor(limit) {
    this.limit = limit // 最大并发数
    this.running = 0 // 当前运行中的任务数
    this.queue = [] // 等待队列
  }

  add(task) {
    return new Promise((resolve, reject) => {
      this.queue.push({ task, resolve, reject })
      this.run()
    })
  }

  run() {
    while (this.running < this.limit && this.queue.length > 0) {
      const { task, resolve, reject } = this.queue.shift()
      this.running++
      Promise.resolve(task())
        .then(resolve)
        .catch(reject)
        .finally(() => {
          this.running--
          this.run()
        })
    }
  }
}

// 使用：
const scheduler = new Scheduler(2)
const delay = (ms) => new Promise((r) => setTimeout(r, ms))

scheduler.add(() => delay(1000).then(() => console.log('task1')))
scheduler.add(() => delay(500).then(() => console.log('task2')))
scheduler.add(() => delay(300).then(() => console.log('task3')))
// 最多同时执行 2 个任务，task3 等 task1 或 task2 完成后再执行
```

## TypeScript：

### TS 的内置工具类型（如 Omit、Pick）的用法？

| 工具类型         | 作用                         | 示例                                       |
| ---------------- | ---------------------------- | ------------------------------------------ |
| `Partial<T>`     | 所有属性变可选               | `Partial<User>`                            |
| `Required<T>`    | 所有属性变必填               | `Required<User>`                           |
| `Readonly<T>`    | 所有属性变只读               | `Readonly<Config>`                         |
| `Pick<T, K>`     | 从 T 中挑选指定属性          | `Pick<User, 'id' \| 'name'>`               |
| `Omit<T, K>`     | 从 T 中排除指定属性          | `Omit<User, 'password'>`                   |
| `Record<K, V>`   | 构造键值类型映射             | `Record<string, number>`                   |
| `Exclude<T, U>`  | 从联合类型 T 中排除 U        | `Exclude<'a'\|'b'\|'c', 'a'>` → `'b'\|'c'` |
| `Extract<T, U>`  | 从 T 中提取可赋值给 U 的类型 | `Extract<'a'\|'b', 'a'\|'c'>` → `'a'`      |
| `NonNullable<T>` | 排除 null 和 undefined       | `NonNullable<string \| null>` → `string`   |
| `ReturnType<T>`  | 提取函数返回值类型           | `ReturnType<typeof fn>`                    |
| `Parameters<T>`  | 提取函数参数类型（元组）     | `Parameters<typeof fn>`                    |

### 如何在不导出函数声明的情况下提取其入参类型？

使用 `Parameters<T>` 工具类型配合 `typeof`：

```ts
function createUser(name: string, age: number, role: 'admin' | 'user') {
  // ...
}

// 提取参数类型元组
type CreateUserParams = Parameters<typeof createUser>
// [name: string, age: number, role: 'admin' | 'user']

// 提取某一个参数
type RoleType = Parameters<typeof createUser>[2]
// 'admin' | 'user'
```

如果是异步函数需要提取返回值：

```ts
async function fetchUser() {
  return { id: 1, name: 'A' }
}

type UserData = Awaited<ReturnType<typeof fetchUser>>
// { id: number; name: string }
```

### Vue 3 中如何获取组件的类型提示（useTemplateRef 等）？

**获取 DOM 元素：**

```ts
const inputRef = useTemplateRef<HTMLInputElement>('myInput')
// 或者
const inputRef = ref<HTMLInputElement | null>(null)
```

**获取子组件实例类型：**

```ts
import MyComp from './MyComp.vue'
import { useTemplateRef } from 'vue'

// useTemplateRef 会自动推断组件实例类型
const myComp = useTemplateRef<typeof MyComp>('myComp')

// 或者用 InstanceType
type MyCompInstance = InstanceType<typeof MyComp>
const myComp = ref<MyCompInstance | null>(null)
```

**子组件用 defineExpose 暴露方法后，父组件可以调用：**

```ts
// MyComp.vue
defineExpose({ reset, validate })

// Parent.vue
const comp = useTemplateRef('myComp')
comp.value?.reset() // 有完整类型提示
```

## 内存管理：

### 内存泄漏常见原因及排查手段（Node.js 端与浏览器端）。

**常见原因：**

1. **全局变量**：意外创建全局变量（`this.foo = ...` 或未用 `var/let/const` 声明），永远不会被 GC
2. **未清理的定时器**：`setInterval` 没有 `clearInterval`，回调持续引用外部对象
3. **未移除的事件监听**：`addEventListener` 后忘记 `removeEventListener`，且回调闭包引用了大对象
4. **闭包引用**：闭包意外持有大对象的引用，导致无法回收
5. **DOM 引用**：JS 变量持有已从 DOM 树移除的节点引用，节点无法被 GC
6. **Vue/React 中**：组件卸载后没有清理 watch、定时器、全局事件监听

**排查手段：**

**浏览器端（Chrome DevTools）：**

- Memory 面板 → **Heap Snapshot**：对比两个时间点的内存快照，找新增且未释放的对象
- Memory 面板 → **Allocation instrumentation on timeline**：录制一段时间内的内存分配，找到持续增长的对象
- Performance 面板：观察 JS Heap 折线，正常应呈锯齿形（有 GC），持续上升说明有泄漏

**Node.js 端：**

- `process.memoryUsage()` 定期打印，观察 `heapUsed` 趋势
- `--inspect` 启动，配合 Chrome DevTools 远程调试
- `heapdump` / `v8-profiler-next` 生成堆快照，用 Chrome DevTools 分析

### 如何避免在编码中产生内存泄漏？

**编码规范层面：**

1. **及时清理定时器**：在组件卸载（`onUnmounted`）或函数退出时 `clearTimeout/clearInterval`
2. **及时移除事件监听**：`addEventListener` 和 `removeEventListener` 成对出现，避免匿名函数（匿名函数无法 removeEventListener）
3. **使用 WeakMap/WeakSet 存储 DOM 引用或对象引用**：弱引用不阻止 GC
4. **避免闭包过度持有**：不需要的大对象在使用完后手动置 `null`
5. **Vue 中规范：** `watch`、`watchEffect` 会返回 stop 函数，如果在组件外部创建需手动停止；全局事件总线订阅在 `onUnmounted` 中取消

**框架层面：**

- Vue 3 的 `onUnmounted` 钩子里统一清理副作用
- React 的 `useEffect` 返回清理函数
- 使用 `AbortController` 取消 fetch 请求，避免组件卸载后回调仍在执行并操作已卸载的组件状态
