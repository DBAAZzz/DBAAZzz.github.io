## 常见攻击与防护：

### CSRF (跨站请求伪造)：原理、Token 机制、Samesite Cookie 策略。

**背诵版：**

- CSRF 的原理：用户登录了 A 网站，浏览器保存了 Cookie。然后用户访问了恶意网站 B，B 的页面里偷偷向 A 发了一个请求（比如转账），浏览器会**自动带上 A 的 Cookie**，A 的服务器以为是用户自己操作的，就执行了
- 关键点：攻击者**不需要拿到 Cookie**，只需要利用浏览器会自动携带 Cookie 这个机制
- 防护手段三个：
  1. **CSRF Token**：服务端生成一个随机 Token 写入页面（表单隐藏字段或 meta 标签），请求时必须带上这个 Token。恶意网站拿不到这个 Token，所以伪造的请求会被拒绝
  2. **SameSite Cookie**：设置 Cookie 的 `SameSite` 属性。`Strict` 表示跨站请求**完全不带 Cookie**（最安全但影响体验，比如从外链点进来也不带）；`Lax` 是默认值，只有导航到目标网站的 GET 请求才带 Cookie（兼顾安全和体验）；`None` 必须配合 `Secure`（HTTPS）使用，等于没限制
  3. **检查 Referer/Origin**：服务端校验请求头的 Referer 或 Origin 是否来自合法域名，但 Referer 可能被篡改或为空，不够可靠，一般作为辅助手段

```
// 典型攻击场景
// 用户登录了 bank.com，恶意网站里有：
<img src="https://bank.com/transfer?to=hacker&amount=10000" />
// 浏览器请求这个 URL 时会自动带上 bank.com 的 Cookie
```

### XSS (跨站脚本攻击)：存储型/反射型/DOM 型区别？转义 (Encode) 与过滤 (Filter) 的权衡。

**背诵版：**

- XSS 就是攻击者把**恶意脚本注入**到网页中，在其他用户的浏览器里执行，可以偷 Cookie、劫持会话、篡改页面
- 三种类型：
  1. **存储型**：恶意脚本被**存到数据库**了（比如评论区提交了 `<script>alert(1)</script>`），所有访问这个页面的用户都会中招。危害最大
  2. **反射型**：恶意脚本在 **URL 参数**里（比如 `?search=<script>...`），服务端把参数原样拼到 HTML 返回。需要诱导用户点击恶意链接才能触发
  3. **DOM 型**：纯前端的问题，JS 直接把用户输入写到 DOM 里（比如 `innerHTML = location.hash`），不经过服务端。攻击者构造恶意 URL 即可
- 防护核心——**转义 vs 过滤**：
  - **转义（Encode）**：把 `<`、`>`、`&`、`"`、`'` 等特殊字符转成 HTML 实体（`&lt;`、`&gt;`），脚本就不会被执行。**优先用转义**，简单可靠不会误杀
  - **过滤（Filter）**：用白名单/黑名单过滤危险标签和属性。适合富文本场景（用户需要输入 HTML），但实现复杂，容易被绕过
  - 权衡：**普通文本用转义，富文本用过滤（白名单模式）**，两者结合使用
- 其他防护：设置 `HttpOnly` Cookie（JS 读不到）、启用 CSP（限制脚本来源）、避免使用 `innerHTML`（用 `textContent` 代替）

### CSP (内容安全策略)：如何配置及解决什么问题？

**背诵版：**

- CSP 解决的问题：**防止 XSS 和数据注入攻击**。原理是通过白名单机制，告诉浏览器只允许加载和执行**指定来源**的资源
- 配置方式两种：
  1. HTTP 响应头：`Content-Security-Policy: 策略指令`
  2. HTML meta 标签：`<meta http-equiv="Content-Security-Policy" content="策略指令">`
- 常用指令（背几个核心的就行）：
  - `default-src 'self'`：默认只允许加载同源资源
  - `script-src 'self' https://cdn.example.com`：JS 只能从同源和指定 CDN 加载
  - `style-src 'self' 'unsafe-inline'`：允许同源样式和内联样式
  - `img-src *`：图片可以从任意来源加载
  - `connect-src 'self'`：AJAX/WebSocket 只能连同源
- CSP 能做到的事：**禁止内联脚本执行**（除非明确允许 `'unsafe-inline'`）、**限制外部脚本来源**、**阻止 eval 执行**、**报告违规行为**（`report-uri` 指令）
- 实际使用：先用 `Content-Security-Policy-Report-Only` 头只报告不拦截，观察一段时间确认没问题后再改成正式的 `Content-Security-Policy`

```
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com; style-src 'self' 'unsafe-inline'; img-src *; report-uri /csp-report
```

## HTTPS/TLS：

### HTTPS 的完整握手流程（TLS 1.2/1.3）。

**背诵版：**

- HTTPS = HTTP + TLS，TLS 在 TCP 连接建立后、HTTP 通信前进行**加密握手**

**TLS 1.2 握手（2-RTT）：**

1. **Client Hello**：客户端发送支持的 TLS 版本、加密套件列表、一个客户端随机数
2. **Server Hello**：服务端选定 TLS 版本和加密套件，返回服务端随机数 + 服务端证书
3. **客户端验证证书**：用 CA 公钥验证证书合法性，取出服务端公钥
4. **客户端生成预主密钥**：用服务端公钥加密一个随机数（Pre-Master Secret）发给服务端
5. **双方生成会话密钥**：用客户端随机数 + 服务端随机数 + 预主密钥，各自计算出相同的**对称密钥**
6. **Finished**：双方用对称密钥加密一个确认消息，握手完成，后续用对称加密通信

**TLS 1.3 握手（1-RTT）：**

- 客户端在 Client Hello 时就**直接带上了密钥协商参数**（DH 公钥），服务端收到后一步到位算出密钥
- 省掉了一个来回，从 2-RTT 降到了 **1-RTT**
- 还支持 **0-RTT 恢复**（复用之前的会话），适合重复连接场景
- 废弃了不安全的加密算法（RSA 密钥交换、RC4、DES 等），只保留前向安全的算法

**记忆口诀：1.2 两个来回，1.3 一个来回。**

### 对称加密与非对称加密在 HTTPS 中的作用。

**背诵版：**

- **非对称加密**（RSA/ECDHE）：用在**握手阶段**，负责安全地交换密钥。公钥加密、私钥解密，安全但**慢**
- **对称加密**（AES）：用在**通信阶段**，负责加密实际传输的数据。同一个密钥加解密，**快**
- HTTPS 的设计是**用非对称加密来交换对称密钥，然后用对称密钥加密数据**，兼顾了安全和性能
- 为什么不全用非对称加密？因为太慢了，RSA 加解密的性能是 AES 的几百倍差距
- 为什么不直接用对称加密？因为密钥没法安全传输——你要是明文传密钥，中间人一截获就完了
- 所以两者**互补**：非对称加密解决密钥分发问题，对称加密解决性能问题

### 中间人攻击 (MITM) 的原理及抓包工具（Charles/Fiddler）的实现原理。

**背诵版：**

- 中间人攻击的原理：攻击者在客户端和服务端之间**拦截通信**，对双方伪装身份——对客户端假装自己是服务端，对服务端假装自己是客户端，从而可以**窃取和篡改数据**
- HTTPS 下为什么通常安全？因为有**证书验证**——客户端会验证服务端证书是否由可信 CA 签发，如果证书不对浏览器会报警
- 抓包工具（Charles/Fiddler）的原理——**就是合法的中间人**：
  1. 工具自己生成一个**根证书**，用户手动安装并信任这个根证书
  2. 客户端发起 HTTPS 请求时，工具拦截，**用自己的根证书签发一个假的服务端证书**返回给客户端
  3. 客户端验证证书——因为用户已经信任了工具的根证书，所以验证通过
  4. 工具再用真正的服务端证书与服务端建立连接
  5. 这样工具就成了中间人，**两端分别维护一个 TLS 连接**，可以看到所有明文数据
- 所以抓包 HTTPS 的前提是**用户主动安装并信任了抓包工具的根证书**，没装证书的话浏览器会报证书错误
- 防御手段：**证书固定（Certificate Pinning）**——App 内置服务端证书的指纹，不信任系统证书链，这样即使装了抓包证书也无法抓包（常见于银行类 App）

## 网络协议 (HTTP/TCP)：

### HTTP 1.1 vs 2.0 (多路复用) vs 3.0 (QUIC/UDP)。

**背诵版：**

**HTTP 1.1：**

- 默认开启**持久连接**（Keep-Alive），一个 TCP 连接可以发多个请求，但必须**排队**——前一个响应完了才能发下一个
- 核心问题是**队头阻塞**：前面的请求慢了，后面的全部卡住
- 浏览器的应对：对同一域名开 **6-8 个 TCP 连接**并发，但连接数有上限

**HTTP 2.0：**

- **多路复用**：一个 TCP 连接上可以同时发送多个请求和响应，以二进制帧（Frame）为单位交错传输，**彻底解决了应用层的队头阻塞**
- **头部压缩**（HPACK）：用静态表 + 动态表 + 哈夫曼编码压缩重复的 Header，减少开销
- **服务端推送**：服务端可以主动向客户端推送资源（如推 CSS、JS），不用等客户端请求
- **二进制分帧**：数据以二进制帧传输，解析更高效
- 但还有问题：TCP 层的队头阻塞没解决——一个 TCP 包丢了，后面所有帧都要等重传

**HTTP 3.0：**

- 底层把 TCP 换成了 **QUIC（基于 UDP）**，彻底解决了 TCP 的队头阻塞
- QUIC 自己实现了可靠传输、流量控制、拥塞控制，丢包只影响**单个流**，不影响其他流
- **0-RTT 连接建立**：之前连过的服务器可以在第一个数据包就携带应用数据，速度极快
- TLS 1.3 **内建**在 QUIC 里，握手和加密合并在一起，更快更安全
- **连接迁移**：用连接 ID 标识连接（不是用 IP+端口），WiFi 切到 4G 不用重新握手

**一句话总结：1.1 排队等，2.0 一条路多辆车但路断了全堵，3.0 换成了多条独立车道。**

### 状态码 301, 302, 304 的精确含义。

**背诵版：**

- **301 Moved Permanently（永久重定向）**：资源**永久**搬到了新地址。浏览器会**缓存**这个重定向，下次直接访问新地址，不再请求旧地址。搜索引擎会把权重转移到新地址。典型场景：网站换域名
- **302 Found（临时重定向）**：资源**临时**在另一个地址。浏览器**不缓存**，每次还是先请求旧地址再被重定向。典型场景：登录后跳转、A/B 测试
- **304 Not Modified（未修改）**：服务端告诉客户端，你之前缓存的资源**没有变**，直接用缓存就行，不用重新下载。是**协商缓存**命中的结果
  - 流程：客户端带着 `If-Modified-Since`（时间戳）或 `If-None-Match`（ETag）请求服务端，服务端对比后发现没变，返回 304（不返回 body）
- 区分记忆：**301 永久搬家**（浏览器记住新地址）、**302 临时借住**（每次还去老地址问）、**304 没变化用缓存**

**补充几个常考的：**

- **200**：成功
- **204**：成功但没有内容返回（常用于 DELETE 请求或预检请求响应）
- **400**：请求参数有误（客户端的错）
- **401**：未认证（没登录）
- **403**：已认证但没权限
- **404**：资源不存在
- **500**：服务端内部错误
- **502**：网关错误（后端服务挂了）
- **503**：服务暂时不可用（过载或维护中）

### TCP 三次握手过程？如果最后一次 ACK 丢了会发生什么？

**背诵版：**

**三次握手过程：**

1. **第一次握手（SYN）**：客户端发送 `SYN=1, seq=x`，进入 `SYN_SENT` 状态。意思是"我要建连接了"
2. **第二次握手（SYN+ACK）**：服务端收到后发送 `SYN=1, ACK=1, seq=y, ack=x+1`，进入 `SYN_RCVD` 状态。意思是"我收到了，我也要建连接"
3. **第三次握手（ACK）**：客户端发送 `ACK=1, seq=x+1, ack=y+1`，进入 `ESTABLISHED` 状态。服务端收到后也进入 `ESTABLISHED`。意思是"我确认了，开始通信吧"

**为什么是三次不是两次？**

- 两次握手的话，服务端发完 SYN+ACK 就认为连接建立了，但如果客户端的第一个 SYN 是**过期的重发包**（网络延迟导致），服务端会白白建立一个无用连接，浪费资源
- 三次握手让客户端有机会告诉服务端"这个 SYN 是有效的"，避免历史连接的干扰

**如果最后一次 ACK 丢了：**

- 服务端一直在 `SYN_RCVD` 状态，等待 ACK。超时后会**重传 SYN+ACK**（通常重试 5 次，间隔依次翻倍）
- 客户端以为连接已建立（`ESTABLISHED`），开始发数据。服务端收到数据包后发现连接还没建立，会回复 **RST（重置）**，客户端收到后知道连接失败，重新发起握手
- 如果重传都失败了，服务端最终**放弃连接**，释放资源

## 跨域通信：

### CORS 的预检请求 (Options) 触发时机？

**背诵版：**

- 浏览器把跨域请求分为**简单请求**和**非简单请求**，只有非简单请求才会触发预检（OPTIONS）
- **简单请求的条件**（全部满足才算简单）：
  1. 方法是 `GET`、`HEAD` 或 `POST`
  2. Header 只有 `Accept`、`Accept-Language`、`Content-Language`、`Content-Type`
  3. `Content-Type` 只能是 `text/plain`、`multipart/form-data`、`application/x-www-form-urlencoded`
- 不满足以上任一条件就是**非简单请求**，浏览器会先发一个 **OPTIONS 预检请求**，问服务端"我能不能用这个方法/这些头来访问你"
- 常见触发预检的场景：`Content-Type: application/json`、自定义 Header（如 `Authorization`、`X-Token`）、`PUT`/`DELETE` 方法
- 预检请求的响应头：
  - `Access-Control-Allow-Origin`：允许的源
  - `Access-Control-Allow-Methods`：允许的方法
  - `Access-Control-Allow-Headers`：允许的自定义头
  - `Access-Control-Max-Age`：预检结果的**缓存时间**（秒），缓存期内不会重复预检
- 优化：设置 `Access-Control-Max-Age` 缓存预检结果，避免每次都发 OPTIONS

### 两个跨域 iframe 之间如何通信？

**背诵版：**

- 核心方案是 **`postMessage`**，这是浏览器提供的**跨源通信 API**，安全且标准

```js
// iframe A 发送消息
const iframeB = document.getElementById('iframeB').contentWindow
iframeB.postMessage({ type: 'hello', data: '你好' }, 'https://b.com')

// iframe B 接收消息
window.addEventListener('message', (event) => {
  // 安全校验：一定要检查 origin！
  if (event.origin !== 'https://a.com') return
  console.log(event.data) // { type: 'hello', data: '你好' }
})
```

- 使用要点三个：
  1. **一定要校验 `event.origin`**，否则任何页面都能给你发消息，有安全风险
  2. 第二个参数指定**目标源**（`targetOrigin`），用 `*` 表示不限制，但生产环境一定要写具体域名
  3. 数据会被**结构化克隆**（类似深拷贝），不能传函数和 DOM 节点

- 其他通信方式（了解即可）：
  - **同域 iframe**：直接通过 `parent.window` / `iframe.contentWindow` 互相访问
  - **url hash**：父页面改 iframe 的 `src` hash，iframe 监听 `hashchange`。单向、容量小
  - **SharedWorker**：同源的多个页面/iframe 共享一个 Worker 通信
  - **BroadcastChannel**：同源页面之间的广播通信，API 更简单

### Node.js 层面如何处理跨域？

**背诵版：**

- 本质就是在响应头里加上 CORS 相关的头，告诉浏览器"这个跨域请求我允许"

**方式一：手动设置响应头**

```js
app.use((req, res, next) => {
  res.setHeader('Access-Control-Allow-Origin', 'https://example.com') // 允许的源，不要用 *（有安全风险且不支持带 Cookie）
  res.setHeader('Access-Control-Allow-Methods', 'GET, POST, PUT, DELETE')
  res.setHeader('Access-Control-Allow-Headers', 'Content-Type, Authorization')
  res.setHeader('Access-Control-Allow-Credentials', 'true') // 允许带 Cookie
  res.setHeader('Access-Control-Max-Age', '86400') // 预检缓存 24 小时

  // 预检请求直接返回
  if (req.method === 'OPTIONS') {
    return res.sendStatus(204)
  }
  next()
})
```

**方式二：用 cors 中间件（Express）**

```js
const cors = require('cors')
app.use(
  cors({
    origin: 'https://example.com',
    credentials: true,
  }),
)
```

**方式三：反向代理（Nginx / Node 代理）**

```
// 原理：前端请求同源的 Node 服务，Node 服务再转发给目标 API
// 因为服务端之间通信没有跨域限制（跨域是浏览器的安全策略）

// Vite 开发环境配置代理
export default {
  server: {
    proxy: {
      '/api': {
        target: 'https://api.example.com',
        changeOrigin: true,
        rewrite: (path) => path.replace(/^\/api/, ''),
      },
    },
  },
}
```

- 三种方式怎么选：
  - **开发环境**：用 Vite/Webpack 的 devServer proxy 最方便
  - **生产环境**：Nginx 反向代理最常用，或者后端直接设置 CORS 头
  - 注意：`Access-Control-Allow-Origin` 不能设为 `*` 同时 `Allow-Credentials` 为 `true`，必须指定具体域名
