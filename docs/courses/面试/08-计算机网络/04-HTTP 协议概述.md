## 计算机网络体系结构

在介绍 HTTP 之前，我们需要了解网络分层模型，主要包括 **OSI 七层参考模型** 和 **TCP/IP 五层模型**。

| 层级 | OSI 七层模型           | TCP/IP 五层模型 | 对应协议/设备                     |
| :--- | :--------------------- | :-------------- | :-------------------------------- |
| 7    | 应用层 (Application)   | 应用层          | **HTTP**， FTP， SMTP， DNS       |
| 6    | 表示层 (Presentation)  | 应用层          | JPEG， ASCII， 加密/解密          |
| 5    | 会话层 (Session)       | 应用层          | RPC， SQL                         |
| 4    | 传输层 (Transport)     | 传输层          | **TCP**， **UDP**                 |
| 3    | 网络层 (Network)       | 网络层          | **IP**， ICMP， ARP， 路由器      |
| 2    | 数据链路层 (Data Link) | 数据链路层      | Ethernet， MAC， 交换机， 网桥    |
| 1    | 物理层 (Physical)      | 物理层          | 物理介质， 网卡， 中继器， 集线器 |

## HTTP 协议概述

### 基本概念

**HTTP** (Hyper Text Transfer Protocol，超文本传输协议) 是一个基于 **TCP/IP** 通信协议来传递数据（HTML 文件、图片文件、查询结果等）的应用层协议。

它采用 **客户端-服务器 (Client-Server)** 架构模式。

### 协议特点

1.  **无连接**：
    - 限制每次连接只处理一个请求。服务器处理完客户的请求，并收到客户的应答后，即断开连接。
    - _注：HTTP/1.1 默认开启 Keep-Alive 长连接，在一定程度上解决了这个问题。_
2.  **无状态**：
    - 协议对于事务处理没有记忆能力。缺少状态意味着如果后续处理需要前面的信息，必须重传，这可能导致数据量增大。
    - _解决方案：Cookie， Session， Token 等机制。_
3.  **简单快速**：
    - 客户向服务器请求服务时，只需传送请求方法和路径。由于 HTTP 协议简单，使得 HTTP 服务器的程序规模小，通信速度很快。
4.  **灵活**：
    - HTTP 允许传输任意类型的数据对象。正在传输的类型由 `Content-Type` 加以标记。

## HTTP 报文结构

### 请求报文 (Request)

请求报文由以下四部分组成：

1.  **请求行 (Request Line)**
    - 包含：请求方法 (Method)、请求 URL、HTTP 版本号。
    - 示例：`GET /index.html HTTP/1.1`
2.  **请求头 (Request Headers)**
    - 用来说明服务器要使用的附加信息（键值对）。
    - 示例：`User-Agent`、`Accept`、`Content-Type`、`Connection`。
3.  **空行**
    - 用于分割请求头与请求体。
4.  **请求体 (Request Body)**
    - POST 等方法传递的数据实体。

### 响应报文 (Response)

响应报文由以下四部分组成：

1.  **状态行 (Status Line)**
    - 包含：HTTP 版本号、状态码 (Status Code)、状态消息 (Reason Phrase)。
    - 示例：`HTTP/1.1 200 OK`
2.  **响应头 (Response Headers)**
    - 说明客户端要使用的一些附加信息。
    - 示例：`Content-Type`、`Date`、`Content-Length`、`Server`。
3.  **空行**
    - 用于分割响应头与响应体。
4.  **响应体 (Response Body)**
    - 服务器返回给客户端的文本信息（HTML， JSON， 图片二进制等）。

## HTTP 方法 (Methods)

### 常见方法

| 方法        | 描述                                                                            |
| :---------- | :------------------------------------------------------------------------------ |
| **GET**     | 获取资源。请求指定的页面信息，并返回实体主体。                                  |
| **POST**    | 传输资源。向指定资源提交数据进行处理请求（例如提交表单或者上传文件）。          |
| **PUT**     | 更新资源。从客户端向服务器传送的数据取代指定的文档的内容。                      |
| **DELETE**  | 删除资源。请求服务器删除指定的页面。                                            |
| **HEAD**    | 获取报文首部。类似于 GET 请求，只不过返回的响应中没有具体的内容，用于获取报头。 |
| **OPTIONS** | 允许客户端查看服务器的性能，常用于 CORS 预检请求。                              |

### GET 和 POST 的区别 (重点)

| 特性         | GET                                              | POST                                                                                            |
| :----------- | :----------------------------------------------- | :---------------------------------------------------------------------------------------------- |
| **数据位置** | 数据拼接到 URL 后 (Query String)                 | 数据放在 Request Body 中                                                                        |
| **数据大小** | 受浏览器和服务器对 URL 长度的限制 (一般 2KB~8KB) | 理论上无限制                                                                                    |
| **缓存**     | 能被缓存                                         | 默认不缓存                                                                                      |
| **历史记录** | 参数保留在浏览器历史中                           | 参数不会保留                                                                                    |
| **安全性**   | 较差，数据暴露在 URL 中                          | 相对较好，但抓包仍可见 (需 HTTPS 保证安全)                                                      |
| **编码**     | 只能进行 URL 编码                                | 支持多种编码 (form-data， json， etc.)                                                          |
| **TCP 包**   | 产生一个 TCP 数据包 (Header 和 Data 一起发送)    | (大部分浏览器) 产生两个 TCP 数据包: 先发 Header 响应 100 Continue，再发 Data (Firefox 只发一次) |
| **幂等性**   | **幂等** (多次请求结果一致)                      | **不幂等** (多次请求可能产生副作用，如重复扣款)                                                 |

**副作用与幂等性说明：**

- **副作用**：指对服务器上的资源做改变。搜索是无副作用的，注册是有副作用的。
- **幂等**：指发送 M 和 N 次请求（两者不相同且都大于 1），服务器上资源的状态一致。GET 多用于无副作用、幂等的场景；POST 多用于副作用、不幂等的场景。

## 常见 HTTP 状态码

HTTP 状态码由**三位数字**组成，第一位定义了响应的类别。

| 状态码 | 类别             | 描述                       |
| :----- | :--------------- | :------------------------- |
| 1xx    | 信息性状态码     | 接收的请求正在处理         |
| 2xx    | 成功状态码       | 请求正常处理完毕           |
| 3xx    | 重定向状态码     | 需要进行附加操作以完成请求 |
| 4xx    | 客户端错误状态码 | 服务器无法处理请求         |
| 5xx    | 服务器错误状态码 | 服务器处理请求出错         |

### 1xx 信息性状态码

- **100 Continue**：客户端应继续其请求（通常用于大文件上传前的预检）。
- **101 Switching Protocols**：服务器正在切换协议（如 HTTP 升级到 WebSocket）。

### 2xx 成功状态码

- **200 OK**：请求成功，响应报文包含实体。
- **204 No Content**：请求成功，但响应报文不含实体的主体部分。
- **205 Reset Content**：请求成功，不含实体，要求请求方重置内容。
- **206 Partial Content**：进行范围请求（断点续传时常见）。

### 3xx 重定向状态码

- **301 Moved Permanently**：永久性重定向。资源已被分配了新的 URL，搜索引擎会更新索引。
- **302 Found**：临时性重定向。资源临时被分配了新的 URL。
- **303 See Other**：表示资源存在着另一个 URL，应使用 GET 方法获取资源。
- **304 Not Modified**：未修改。客户端有缓冲的文档并发出了一个条件性的请求（If-Modified-Since），服务器告诉客户端，原来缓冲的文档还可以继续使用。
- **307 Temporary Redirect**：临时重定向，和 302 类似，但要求客户端**保持请求方法不变**向新的地址发出请求（即 POST 重定向后仍为 POST）。

### 4xx 客户端错误状态码

- **401 Unauthorized**：需要身份验证（通常返回 `WWW-Authenticate` 头）。
- **403 Forbidden**：服务器理解请求但拒绝执行（权限不足）。
- **404 Not Found**：请求的资源不存在。
- **405 Method Not Allowed**：请求方法不被允许（如只支持 GET 却发送了 POST）。
- **408 Request Timeout**：客户端请求超时。
- **429 Too Many Requests**：请求过于频繁，触发限流。

### 5xx 服务器错误状态码

- **500 Internal Server Error**：服务器端在执行请求时发生了错误。
- **501 Not Implemented**：服务器不支持当前请求所需要的某个功能。
- **502 Bad Gateway**：作为网关或者代理工作的服务器尝试执行请求时，从上游服务器接收到无效的响应。
- **503 Service Unavailable**：服务器暂时处于超负载或正在停机维护，无法处理请求。
- **504 Gateway Timeout**：网关超时。

**502 vs 504 的区别：**

- **502**：上游服务器返回了错误的响应（如崩溃、返回格式错误）。
- **504**：上游服务器根本没响应（超时）。

## HTTP 进阶特性

### HTTP 持久连接 (Keep-Alive)

- HTTP/1.0 默认是短连接，每个请求/应答都需要建立一个 TCP 连接。
- **HTTP/1.1 默认开启 Keep-Alive**。
- **原理**：使客户端到服务器端的连接持续有效，即不关闭底层的 TCP 连接，当出现对服务器的后继请求时，直接复用已建立的连接，避免了 TCP 握手和挥手的开销。
- 头部设置：`Connection: keep-alive`

### HTTP 管线化 (Pipelining)

- **概念**：管线化机制允许客户端在未收到上一个请求的响应前，就发送下一个请求。
- **限制**：
  - 只有 GET 和 HEAD 请求可以进行管线化。
  - 必须在持久连接即 Keep-Alive 下才能支持。
  - 服务端必须按照请求的顺序依次返回响应（队头阻塞问题仍在应用层存在）。
  - 因为各种兼容性和代理问题，现代浏览器默认通常不开启管线化，而是使用 HTTP/2 多路复用代替。

### HTTP 数据协商

客户端和服务器通过头部信息协商数据的格式、压缩方式、语言等。

#### 请求头 (Client -> Server)

- `Accept`: 申明想要的数据格式 (`image/webp`， `application/json`).
- `Accept-Encoding`: 告诉服务端支持的压缩方式 (`gzip`， `deflate`， `br`).
- `Accept-Language`: 描述语言信息 (`zh-CN`).
- `User-Agent`: 描述客户端浏览器相关信息。

#### 响应头 (Server -> Client)

- `Content-Type`: 实际返回的数据格式 (`text/html; charset=utf-8`).
- `Content-Encoding`: 实际使用的压缩方式 (`gzip`).
- `Content-Language`: 实际返回的语言。

#### Content-Type 常见类型 (Form 表单)

- `application/x-www-form-urlencoded`: 默认格式，键值对拼接 `key=value&key=value`。
- `multipart/form-data`: 用于**文件上传**。请求体被 `boundary` 分隔成多个部分，每部分有自己的 `Content-Disposition`。
- `text/plain`: 纯文本。
- `application/json`: JSON 格式字符串 (Ajax/Fetch 常用)。

## 安全与策略

### HTTP Redirect 重定向机制

- **301 (永久)**: 浏览器会缓存重定向结果，下次直接访问新地址，不访问原地址。
- **302 (临时)**: 浏览器每次都会先访问原地址，确认是否需要跳转。

### CSP (Content Security Policy 内容安全策略)

CSP 是一个额外的安全层，用于检测和削弱某些特定类型的攻击，包括跨站脚本 (XSS) 和数据注入攻击等。

- **作用**：
  - 限制资源获取（例如只允许加载本域脚本）。
  - 报告资源获取越权。
- **示例**：
  - `Content-Security-Policy: default-src 'self'` (只允许同源资源)
  - `Content-Security-Policy: img-src *` (允许任意源加载图片)
