# 微前端框架深度对比

## 概述

微前端是一种将单体前端应用拆分为多个可独立开发、部署和运行的小型应用的架构模式。本文深度对比主流微前端框架的技术原理、优缺点和适用场景。

---

## 一、框架对比总览

| 框架            | 技术方案              | JS 隔离               | CSS 隔离              | 通信机制          | 生态成熟度 | 学习成本 |
| --------------- | --------------------- | --------------------- | --------------------- | ----------------- | ---------- | -------- |
| **qiankun**     | single-spa + 沙箱     | Proxy 沙箱            | Shadow DOM/动态样式表 | 内置通信          | ⭐⭐⭐⭐⭐ | 中       |
| **micro-app**   | WebComponent          | WebComponent 天然隔离 | Shadow DOM            | 数据绑定          | ⭐⭐⭐⭐   | 低       |
| **无界(wujie)** | iframe + WebComponent | iframe 天然隔离       | iframe 天然隔离       | props/window 通信 | ⭐⭐⭐     | 中低     |
| **single-spa**  | 路由劫持              | 无（需自行实现）      | 无（需自行实现）      | 自定义            | ⭐⭐⭐⭐⭐ | 高       |

## 二、各框架深度解析

### 1. qiankun（阿里系，最流行）

#### 技术原理

```javascript
// qiankun 核心原理示意
// 1. 基于 single-spa 封装
// 2. import-html-entry 加载子应用
// 3. Proxy 沙箱隔离全局变量

class ProxySandbox {
  constructor() {
    const fakeWindow = Object.create(null);
    this.proxy = new Proxy(fakeWindow, {
      get(target, prop) {
        return prop in target ? target[prop] : window[prop];
      },
      set(target, prop, value) {
        target[prop] = value;
        return true;
      },
    });
  }
}
```

**隔离机制：**

- **JS 隔离**：快照沙箱（单例）→ LegacySandbox（Proxy 单例）→ ProxySandbox（Proxy 多例）
- **CSS 隔离**：
  - 动态样式表：运行时增删 `<style>` 标签
  - Shadow DOM：实验性，样式完全隔离但可能影响弹窗等
  - 样式前缀：通过 `strictStyleIsolation` 配置

**优点：**

- 开箱即用，配置简单，文档完善
- 阿里生态支持（antd、umi 等），社区活跃
- 支持多种沙箱模式，兼容性好
- 预加载、资源缓存等性能优化

**缺点：**

- 依赖 single-spa，概念较多
- CSS 隔离不完美，Shadow DOM 有坑
- 子应用需要改造（导出生命周期）
- 沙箱有性能开销

**适用场景：**

- 大型企业级应用，多团队协作
- 需要快速落地微前端方案
- 技术栈以 React/Vue 为主

---

### 2. micro-app（京东，最简单）

#### 技术原理

```javascript
// micro-app 核心原理
// 基于 WebComponent 封装
class MicroApp extends HTMLElement {
  connectedCallback() {
    // 1. 创建 iframe 式隔离环境（不是真 iframe）
    // 2. Shadow DOM 隔离样式
    // 3. 数据绑定通信
    this.loadApp(this.getAttribute("url"));
  }
}

customElements.define("micro-app", MicroApp);
```

**隔离机制：**

- **JS 隔离**：类 iframe 沙箱（通过 with + Proxy 实现纯净的 window 上下文）
- **CSS 隔离**：Shadow DOM 天然隔离（可配置关闭）

**优点：**

- **API 极简**：像使用 `<iframe>` 一样简单
  ```html
  <micro-app name="app1" url="http://localhost:3000/"></micro-app>
  ```
- 子应用**零改造**或少量改造
- JS/CSS 隔离完美，天然支持多实例
- 性能优秀，接近原生

**缺点：**

- 社区相对较小
- WebComponent 兼容性要求（IE 不支持）
- 某些场景下 Shadow DOM 样式穿透复杂
- 调试相对困难

**适用场景：**

- 中小型项目，快速接入
- 子应用不希望大改造
- 对隔离性要求高

### 3. 无界 wujie（腾讯，隔离最强）

#### 技术原理

```javascript
// wujie 核心原理
// iframe(JS隔离) + WebComponent(渲染容器)
// 1. iframe 运行 JS，天然隔离
// 2. WebComponent 渲染 DOM
// 3. 通过劫持 document、appendChild 等实现 DOM 同步

// 伪代码
const iframe = document.createElement("iframe");
const container = new WebComponent();

// iframe 执行 JS
iframe.contentWindow.eval(appCode);

// 劫持 DOM 操作，同步到 container
iframe.contentWindow.document.appendChild = (node) => {
  container.shadowRoot.appendChild(node);
};
```

**隔离机制：**

- **JS 隔离**：iframe 天然隔离（最强）
- **CSS 隔离**：WebComponent + Shadow DOM
- **路由同步**：劫持 iframe 的 history，同步到主应用

**优点：**

- **隔离性最强**：真实 iframe 环境，不会有沙箱逃逸问题
- 子应用**零改造**
- 支持多实例、预加载、keep-alive
- 性能优秀（渲染在主文档流）

**缺点：**

- iframe 通信复杂度（虽然框架已封装）
- 调试困难（跨 iframe）
- 弹窗、遮罩层定位问题（需要额外处理）
- 相对较新，生态不如 qiankun

**适用场景：**

- 对隔离性要求极高（安全、稳定性）
- 子应用完全不可控（第三方应用）
- 老旧应用接入

---

### 4. single-spa（鼻祖，最底层）

#### 技术原理

```javascript
// single-spa 核心：路由劫持 + 生命周期管理
import { registerApplication, start } from "single-spa";

registerApplication({
  name: "app1",
  app: () => import("./app1.js"), // 返回生命周期函数
  activeWhen: "/app1",
  customProps: {
    /* 传递给子应用的数据 */
  },
});

// 子应用需导出生命周期
export async function bootstrap(props) {
  /* 初始化 */
}
export async function mount(props) {
  /* 挂载 */
}
export async function unmount(props) {
  /* 卸载 */
}

start();
```

**隔离机制：**

- **无内置隔离**，需要自己实现沙箱
- 仅提供应用注册、路由劫持、生命周期管理

**优点：**

- 最灵活，可定制性强
- 框架无关，理论上支持所有技术栈
- 社区成熟，qiankun 等都基于它

**缺点：**

- **学习成本最高**，需要理解底层原理
- **需要大量自行实现**：沙箱、样式隔离、通信等
- 子应用改造成本高

**适用场景：**

- 需要高度定制的复杂场景
- 技术团队成熟，有微前端经验
- 作为底层框架二次封装

## 三、核心技术对比

### 3.1 JS 隔离方案对比

| 方案             | 原理                                           | 优点               | 缺点                | 使用框架           |
| ---------------- | ---------------------------------------------- | ------------------ | ------------------- | ------------------ |
| **快照沙箱**     | 记录 window 快照，mount 时恢复，unmount 时还原 | 简单易实现         | 无法多实例、性能差  | qiankun（旧版）    |
| **Proxy 沙箱**   | 拦截 window 操作，记录修改                     | 支持多实例、性能好 | 兼容性（IE 不支持） | qiankun、micro-app |
| **iframe 沙箱**  | 真实 iframe 环境                               | 隔离最彻底         | 通信复杂、内存占用  | wujie              |
| **with + Proxy** | 构造纯净 window 上下文                         | 隔离完美、性能好   | 实现复杂            | micro-app          |

### 3.2 CSS 隔离方案对比

| 方案             | 原理                 | 优点         | 缺点                         |
| ---------------- | -------------------- | ------------ | ---------------------------- |
| **动态样式表**   | 运行时增删 `<style>` | 简单         | 污染风险、卸载不彻底         |
| **Shadow DOM**   | WebComponent 隔离    | 完美隔离     | 弹窗等全局组件失效、调试困难 |
| **CSS Modules**  | 编译时加前缀         | 无运行时开销 | 需要构建支持                 |
| **BEM/命名空间** | 约定前缀             | 兼容性好     | 人工维护、易出错             |

### 3.3 通信方案对比

```javascript
// qiankun - 内置通信
import { initGlobalState } from "qiankun";
const actions = initGlobalState({ user: "admin" });
actions.onGlobalStateChange((state) => console.log(state));

// micro-app - 数据绑定
<micro-app name="app1" data={this.state.data}></micro-app>;
window.microApp.getData(); // 子应用获取

// wujie - props 传递
<WujieReact name="app1" props={{ user: "admin" }}></WujieReact>;

// single-spa - 自定义
// 通过 customProps 传递，或使用发布订阅模式
```

## 四、实际落地建议

### 4.1 技术选型决策树

```
是否需要极致隔离（安全/第三方应用）？
├─ 是 → 无界 wujie
└─ 否 →
    是否追求简单易用？
    ├─ 是 → micro-app
    └─ 否 →
        是否需要成熟生态/大型项目？
        ├─ 是 → qiankun
        └─ 否 →
            是否需要高度定制？
            ├─ 是 → single-spa
            └─ 否 → qiankun/micro-app
```

### 4.2 常见问题与解决方案

#### 问题 1：样式污染

```javascript
// qiankun 方案
sandbox: {
  strictStyleIsolation: true, // Shadow DOM（可能有兼容问题）
  // 或
  experimentalStyleIsolation: true // 动态添加前缀
}

// micro-app 方案（默认 Shadow DOM）
<micro-app shadowDOM></micro-app>

// 人工约定：BEM 命名规范
.app1-header { }
.app2-header { }
```

#### 问题 2：全局状态共享

```javascript
// 方案1：主应用提供全局状态管理（推荐）
// 主应用
window.__GLOBAL_STATE__ = new Vuex.Store({
  /* ... */
});

// 子应用访问
const store = window.__GLOBAL_STATE__;

// 方案2：使用框架内置通信
// qiankun
import { initGlobalState } from "qiankun";

// 方案3：自定义事件总线
class EventBus {
  /* 发布订阅模式 */
}
window.__EVENT_BUS__ = new EventBus();
```

#### 问题 3：静态资源 404

```javascript
// 原因：子应用相对路径基于主应用域名
// 解决：使用绝对路径或配置 publicPath

// Webpack 配置
module.exports = {
  output: {
    publicPath: window.__POWERED_BY_QIANKUN__ ? "/child-app/" : "/",
  },
};

// Vite 配置
export default defineConfig({
  base: window.__POWERED_BY_QIANKUN__ ? "/child-app/" : "/",
});
```

#### 问题 4：路由冲突

```javascript
// 主应用 history 模式，子应用 hash 模式（最简单）
// 或都使用 history，但子应用加前缀

// qiankun 注册

  {
    name: "app1",
    entry: "//localhost:3000",
    container: "#container",
    activeRule: "/app1", // 激活路径
  },
]);

// 子应用路由
const router = new VueRouter({
  mode: "history",
  base: window.__POWERED_BY_QIANKUN__ ? "/app1" : "/",
  routes: [
    /* ... */
  ],
});
```

## 五、面试回答参考

### 问题 1：介绍微前端及其解决的问题

**参考回答：**

> 微前端是将单体前端应用拆分为多个可独立开发、部署和运行的小型应用的架构模式，类似于后端微服务。
>
> **解决的核心问题：**
>
> 1. **技术栈无关**：不同团队可以使用 React、Vue、Angular 等不同框架
> 2. **独立部署**：各子应用独立发布，不影响其他应用
> 3. **团队自治**：大型项目拆分给多个团队并行开发
> 4. **增量升级**：老项目逐步迁移，而不是推倒重来
> 5. **代码隔离**：避免全局变量污染、样式冲突
>
> **我在项目中的实践：**（结合自己经历）
> 我们之前有个 5 年的老 Vue2 项目，新需求想用 Vue3 + TS 开发。使用 qiankun 实现微前端后，主应用保持 Vue2，新模块用 Vue3 独立开发，通过路由切换加载。这样既避免了整体重构风险，又能使用新技术栈，还实现了多团队并行开发。

---

### 问题 2：qiankun 的沙箱隔离原理

**参考回答：**

> qiankun 有三种沙箱模式，逐步演进：
>
> **1. 快照沙箱（SnapshotSandbox）**
>
> ```javascript
> class SnapshotSandbox {
>   mount() {
>     // 记录当前 window 状态
>     this.snapshot = {};
>     for (const prop in window) {
>       this.snapshot[prop] = window[prop];
>     }
>     // 恢复上次修改
>     Object.keys(this.modifyPropsMap).forEach((p) => {
>       window[p] = this.modifyPropsMap[p];
>     });
>   }
>   unmount() {
>     // 记录修改，恢复原始 window
>     this.modifyPropsMap = {};
>     for (const prop in window) {
>       if (window[prop] !== this.snapshot[prop]) {
>         this.modifyPropsMap[prop] = window[prop];
>         window[prop] = this.snapshot[prop];
>       }
>     }
>   }
> }
> ```
>
> - 缺点：只能单实例，diff window 性能差
>
> **2. LegacySandbox（Proxy 单例）**
>
> - 用 Proxy 拦截 window 操作，记录新增/修改的属性
> - 缺点：仍然只能单实例
>
> **3. ProxySandbox（Proxy 多例，当前默认）**
>
> ```javascript
> class ProxySandbox {
>   constructor() {
>     const fakeWindow = Object.create(null);
>     this.proxy = new Proxy(fakeWindow, {
>       get(target, prop) {
>         // 优先取自己的，没有再取 window 的
>         return prop in target ? target[prop] : window[prop];
>       },
>       set(target, prop, value) {
>         // 所有修改都记录在 fakeWindow
>         target[prop] = value;
>         return true;
>       },
>     });
>   }
> }
> ```
>
> - 优点：支持多实例，每个子应用有独立的 fakeWindow
> - 子应用的 JS 运行在 `with(proxy) { code }` 中
>
> **核心思路：** 通过 Proxy 劫持全局变量访问，让子应用以为在操作 window，实际操作的是独立对象，达到隔离效果。

### 问题 3：微前端的 CSS 隔离方案及优缺点

**参考回答：**

> CSS 隔离是微前端的难点，主流方案有：
>
> **1. Shadow DOM（qiankun/micro-app 支持）**
>
> - 原理：WebComponent 的 Shadow DOM 天然隔离
> - 优点：完美隔离，子应用样式不会泄漏
> - 缺点：
>   - 弹窗、抽屉等挂载到 body 的组件样式丢失（因为渲染在 Shadow 外）
>   - 主应用无法影响子应用样式（有时需要主题统一）
>   - 调试困难
>
> **2. 动态样式表（qiankun 默认）**
>
> - 原理：加载子应用时插入 `<style>`，卸载时删除
> - 优点：兼容性好，不影响弹窗等组件
> - 缺点：卸载不彻底可能残留样式，存在污染风险
>
> **3. CSS Modules / Scoped CSS（构建时）**
>
> - 原理：编译时给类名加哈希/前缀
> - 优点：无运行时开销，天然隔离
> - 缺点：需要构建支持，对第三方组件库无效
>
> **4. 命名空间约定（BEM 等）**
>
> - 原理：人工约定前缀，如 `.app1-header {}`
> - 优点：简单、兼容性好
> - 缺点：靠约定，容易出错
>
> **实际选择：**
>
> - 一般项目：动态样式表 + BEM 规范
> - 隔离要求高：Shadow DOM + 手动处理弹窗（挂到 Shadow 内）
> - 新项目：CSS Modules

### 问题 4：qiankun、micro-app、wujie 如何选择

**参考回答：**

> **qiankun - 大型企业级，成熟稳定**
>
> - 适合：多团队协作、需要成熟生态（antd、umi 等）
> - 优点：社区最活跃、文档完善、阿里系背书
> - 缺点：配置相对复杂、子应用需要改造
>
> **micro-app - 简单易用，快速落地**
>
> - 适合：中小型项目、快速接入
> - 优点：API 最简单（像用 iframe）、子应用零改造、隔离完美
> - 缺点：社区较小、WebComponent 不兼容 IE
>
> **wujie - 隔离最强，适合不可控场景**
>
> - 适合：第三方应用接入、对安全隔离要求极高
> - 优点：iframe 天然隔离、子应用零改造、keep-alive 性能好
> - 缺点：相对较新、调试复杂、弹窗定位需处理
>
> **我的选择经验：**
>
> - 我们公司有多个老项目需要整合，用的 **micro-app**，因为老项目不想大改，micro-app 零改造接入成本最低，而且隔离性也够用
> - 如果是新项目，技术团队成熟，我会选 **qiankun**，生态好，遇到问题容易找到解决方案
> - 如果要接入完全不可控的第三方应用，会选 **wujie**，iframe 隔离最安全

### 问题 5：微前端的性能优化

**参考回答：**

> **1. 预加载（Prefetch）**
>
> ```javascript
> // qiankun 支持
> import { prefetchApps } from "qiankun";
> prefetchApps([{ name: "app1", entry: "//localhost:3000" }]);
> ```
>
> - 在浏览器空闲时提前加载子应用资源
>
> **2. 缓存策略**
>
> - 子应用静态资源设置长缓存（hash 文件名）
> - qiankun 会缓存已加载的子应用 HTML/JS
>
> **3. keep-alive（保活）**
>
> ```javascript
> // wujie 支持
> <WujieReact name="app1" alive={true}></WujieReact>
> ```
>
> - 子应用切换时不销毁，保留状态
> - 适合频繁切换的场景
>
> **4. 按需加载**
>
> - 路由懒加载，避免一次性加载所有子应用
>
> ```javascript
> registerMicroApps([
>   {
>     name: "app1",
>     entry: "//localhost:3000",
>     activeRule: "/app1", // 只在访问 /app1 时加载
>   },
> ]);
> ```
>
> **5. 共享依赖（Webpack Module Federation）**
>
> ```javascript
> // 主应用和子应用共享 React/Vue 等大库
> module.exports = {
>   plugins: [
>     new ModuleFederationPlugin({
>       shared: ["react", "react-dom"],
>     }),
>   ],
> };
> ```
>
> **6. 子应用打包优化**
>
> - 代码分割、Tree Shaking
> - 压缩、Gzip
> - CDN 加速
>
> **实际效果：**
> 我们项目用了预加载 + 缓存 + 共享依赖后，子应用首次加载从 3s 降到 1.2s，切换应用从 800ms 降到 200ms。

### 问题 6：微前端的挑战和坑

**参考回答：**

> **1. 首屏性能**
>
> - 问题：加载主应用 + 子应用，资源请求多
> - 解决：预加载、SSR、骨架屏
>
> **2. 样式污染**
>
> - 问题：全局样式、第三方组件库样式冲突
> - 解决：Shadow DOM（注意弹窗）、CSS Modules、BEM 规范
>
> **3. 全局变量冲突**
>
> - 问题：多个子应用都修改 `window.xxx`
> - 解决：沙箱隔离 + 约定不使用全局变量
>
> **4. 通信复杂**
>
> - 问题：主应用和子应用、子应用间通信
> - 解决：使用框架提供的通信机制，避免直接访问 window
>
> **5. 调试困难**
>
> - 问题：多个应用同时运行，错误栈复杂
> - 解决：Source Map、错误边界、日志系统
>
> **6. 子应用改造成本**
>
> - 问题：老项目需要导出生命周期、处理 publicPath
> - 解决：选择改造成本低的框架（micro-app/wujie）
>
> **7. 重复依赖**
>
> - 问题：每个子应用都打包 React，浪费资源
> - 解决：Webpack Module Federation 共享依赖
>
> **我踩过的坑：**
> 最大的坑是弹窗样式丢失（用了 Shadow DOM）。后来改成动态样式表 + BEM 规范，弹窗手动挂载到子应用容器内解决的。

### 问题 7：如何从 0 到 1 落地微前端

**参考回答：**

> **第一步：评估是否需要微前端**
>
> - 单体应用是否足够大（多团队、多技术栈）
> - 是否有独立部署需求
> - 是否有老项目改造需求
> - **不要为了微前端而微前端**，中小项目可能不适合
>
> **第二步：技术选型**
>
> - 根据团队技术栈、项目规模、隔离需求选择框架
> - 我的建议：先做 POC（Proof of Concept），验证关键场景
>
> **第三步：主应用搭建**
>
> ```javascript
> // 以 qiankun 为例
> import { registerMicroApps, start } from "qiankun";
>
> registerMicroApps([
>   {
>     name: "app1",
>     entry: "//localhost:3000",
>     container: "#subapp-container",
>     activeRule: "/app1",
>   },
> ]);
>
> start();
> ```
>
> **第四步：子应用改造**
>
> ```javascript
> // 子应用入口文件
> let instance = null;
>
> function render(props = {}) {
>   const { container } = props;
>   instance = new Vue({
>     router,
>     store,
>     render: (h) => h(App),
>   }).$mount(container ? container.querySelector("#app") : "#app");
> }
>
> // 独立运行
> if (!window.__POWERED_BY_QIANKUN__) {
>   render();
> }
>
> // 生命周期
> export async function bootstrap() {}
> export async function mount(props) {
>   render(props);
> }
> export async function unmount() {
>   instance.$destroy();
> }
> ```
>
> **第五步：解决通信、样式隔离等问题**
>
> - 通信：使用框架提供的机制或自定义事件总线
> - 样式：选择合适的隔离方案
> - 路由：主应用拦截路由，分发给子应用
>
> **第六步：规范制定**
>
> - 子应用开发规范（命名、打包）
> - 通信协议
> - 部署流程
>
> **第七步：监控与优化**
>
> - 性能监控（加载时间、切换时间）
> - 错误监控（Sentry 等）
> - 持续优化

## 六、总结

### 6.1 快速决策

- **快速落地** → micro-app
- **大型项目/成熟生态** → qiankun
- **极致隔离/第三方应用** → wujie
- **高度定制** → single-spa

### 6.2 微前端并非银弹

**适合场景：**

- 大型应用，多团队协作
- 技术栈迁移
- 老项目改造

**不适合场景：**

- 小型项目（增加复杂度）
- 单团队单技术栈（monorepo 更好）
- 追求极致性能（单体更快）

### 6.3 关键点

1. **隔离性**：JS/CSS 隔离是核心难点
2. **通信**：选择合适的通信方案，避免过度耦合
3. **性能**：预加载、缓存、共享依赖
4. **规范**：制定开发、部署规范
5. **监控**：性能、错误监控不可少
