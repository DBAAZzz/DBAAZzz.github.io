## 一、Node.js 基础

### 1.1 核心概念

**Q1: 什么是 Node.js?它有哪些特点?**

<details>
<summary>查看答案</summary>

Node.js 是一个基于 Chrome V8 引擎的 JavaScript 运行时环境,主要特点:

- **事件驱动**: 基于事件循环机制处理异步操作
- **非阻塞 I/O**: I/O 操作不会阻塞主线程
- **单线程**: 主线程是单线程,但通过事件循环和线程池处理并发
- **跨平台**: 支持 Windows、Linux、macOS 等
- **高性能**: V8 引擎提供快速的 JavaScript 执行
- **轻量高效**: 适合 I/O 密集型应用

**深入理解:**

1. **V8 引擎优化机制:**
   - JIT(即时编译): 热代码路径被编译为机器码
   - Hidden Classes: 优化对象属性访问
   - Inline Caching: 缓存属性查找结果
   - Garbage Collection: 分代回收(新生代 Scavenge + 老生代 Mark-Sweep/Mark-Compact)

2. **libuv 底层架构:**
   - Event Loop: 事件循环调度器
   - Thread Pool: 默认 4 个线程处理文件 I/O、DNS 查询、crypto 等
   - 平台抽象层: 封装不同操作系统的 I/O 多路复用(Linux epoll、macOS kqueue、Windows IOCP)

3. **适用场景边界:**
   - ✓ 适合: I/O 密集(API 网关、实时应用、数据流处理)
   - ✗ 不适合: CPU 密集(需配合 Worker Threads 或微服务拆分)
   - 内存限制: 默认堆内存 ~1.4GB(64 位),可通过 `--max-old-space-size` 调整

</details>

**Q2: Node.js 为什么采用单线程模型?有什么优缺点?**

<details>
<summary>查看答案</summary>

**设计理念:**

- 避免多线程上下文切换的开销
- 避免线程同步和死锁问题
- 简化编程模型,符合 JavaScript 语言特性
- Ryan Dahl 受 Nginx 事件驱动架构启发

**优点:**

- 无需线程锁,避免死锁和竞态条件
- 内存占用低(单线程栈 ~1MB vs 多线程栈 N×1MB)
- 上下文切换开销小,适合高并发 I/O

**缺点与解决方案:**

| 问题               | 影响                     | 解决方案                                   |
| ------------------ | ------------------------ | ------------------------------------------ |
| 无法利用多核       | CPU 利用率低             | cluster 模块 / PM2                         |
| CPU 密集任务阻塞   | 事件循环饥饿,请求延迟高  | Worker Threads / 子进程 / 微服务拆分       |
| 单点故障           | 进程崩溃导致服务不可用   | 进程守护 + 健康检查 + 自动重启             |
| 内存泄漏无法隔离   | 影响整个应用             | 内存监控 + 定期重启 + 分布式架构           |

**本质理解:**

- "单线程" 指的是 **JavaScript 执行线程单一**
- Node.js 底层 libuv 线程池(默认 4 线程)处理文件 I/O、DNS、crypto
- 实际是 **单线程事件循环 + 多线程 I/O** 的混合模型
- 适合 I/O 密集场景的核心原因: I/O 操作委托给操作系统/线程池,主线程只负责调度

</details>

**Q3: CommonJS 和 ES Modules 有什么区别?**

<details>
<summary>查看答案</summary>

| 特性         | CommonJS               | ES Modules          |
| ------------ | ---------------------- | ------------------- |
| 加载方式     | 运行时同步加载         | 编译时静态分析      |
| 加载结果     | 值的拷贝(浅拷贝)       | 值的动态绑定(引用)  |
| 语法         | require/module.exports | import/export       |
| this 指向    | exports 对象           | undefined           |
| 循环引用     | 返回未完成的 exports   | 动态绑定,更安全     |
| 顶层 await   | 不支持                 | 支持                |
| Tree Shaking | 不支持                 | 支持                |
| 条件加载     | 支持(运行时)           | 不支持(需动态导入)  |

```javascript
// CommonJS - 值拷贝
// counter.js
let count = 0
module.exports = {
  count,
  increment: () => ++count,
}

// main.js
const counter = require('./counter')
console.log(counter.count) // 0
counter.increment()
console.log(counter.count) // 仍然是 0 (拷贝值未更新)

// ES Modules - 动态绑定
// counter.mjs
export let count = 0
export const increment = () => ++count

// main.mjs
import { count, increment } from './counter.mjs'
console.log(count) // 0
increment()
console.log(count) // 1 (动态绑定,实时更新)
```

**深层原理:**

1. **加载时机差异:**
   - CommonJS: `require()` 同步执行,可在任意位置调用,运行时解析
   - ESM: `import` 在模块解析阶段提升,静态结构便于编译优化

2. **循环依赖处理:**
   - CommonJS: 返回当前已执行部分的 exports(不完整对象)
   - ESM: 通过动态绑定解决,但仍需注意初始化顺序

3. **Node.js 中的互操作:**
   ```javascript
   // ESM 中加载 CommonJS
   import cjs from './cjs-module.js' // default 导入整个 module.exports

   // CommonJS 中加载 ESM (需异步)
   const esm = await import('./esm-module.mjs')

   // package.json 配置
   {
     "type": "module",  // 所有 .js 视为 ESM
     "exports": {
       "import": "./index.mjs",
       "require": "./index.cjs"
     }
   }
   ```

4. **性能影响:**
   - ESM 静态分析支持 Tree Shaking,可减少 30-50% 打包体积
   - CommonJS 缓存机制(`require.cache`)提升重复加载性能
   - ESM 异步特性避免阻塞,但首次加载可能稍慢

</details>

**Q5: process.nextTick() 和 setImmediate() 有什么区别?**

<details>
<summary>查看答案</summary>

```javascript
// process.nextTick(): 在当前执行栈结束后立即执行
// setImmediate(): 在事件循环的 check 阶段执行

console.log('1')

setImmediate(() => {
  console.log('2')
})

process.nextTick(() => {
  console.log('3')
})

console.log('4')

// 输出: 1 4 3 2
```

**核心区别:**

| 特性           | process.nextTick()             | setImmediate()              |
| -------------- | ------------------------------ | --------------------------- |
| 执行时机       | 当前阶段结束后,下一阶段开始前  | check 阶段                  |
| 队列类型       | 微任务队列(nextTickQueue)      | 宏任务队列                  |
| 优先级         | 最高(高于 Promise)             | 低于所有微任务              |
| 递归行为       | 可能导致事件循环饥饿           | 不会阻塞事件循环            |
| I/O 回调后执行 | 总是在 I/O 之前                | 在 I/O 之后                 |

**执行顺序详解:**

```javascript
const fs = require('fs')

fs.readFile(__filename, () => {
  console.log('I/O callback')

  setTimeout(() => console.log('timer'), 0)
  setImmediate(() => console.log('immediate'))
  process.nextTick(() => console.log('nextTick'))
  Promise.resolve().then(() => console.log('promise'))
})

// 输出顺序: I/O callback → nextTick → promise → immediate → timer
// 原因: 在 I/O 回调中,setImmediate 优先于 setTimeout(因为已经进入 poll 阶段)
```

**使用场景:**

1. **process.nextTick():**
   - 让用户有机会注册错误处理器(在抛出错误前)
   - 递归操作分批处理(但需控制深度)
   - 确保回调在构造函数完成后执行

   ```javascript
   // 实际应用:EventEmitter 中确保事件监听器已注册
   class MyEmitter extends EventEmitter {
     constructor() {
       super()
       process.nextTick(() => {
         this.emit('ready') // 确保外部已调用 .on('ready')
       })
     }
   }
   ```

2. **setImmediate():**
   - 长时间运行的任务分片
   - 避免阻塞 I/O 操作
   - 递归调用时不会饥饿事件循环

   ```javascript
   // 处理大数组不阻塞 I/O
   function processArray(arr, callback) {
     const chunk = arr.splice(0, 100)
     // 处理 chunk...
     if (arr.length > 0) {
       setImmediate(() => processArray(arr, callback)) // 让出事件循环
     } else {
       callback()
     }
   }
   ```

**底层原理:**

- `process.nextTick()` 使用内部 `nextTickQueue`,在 C++ 层面每个事件循环阶段转换前检查
- `setImmediate()` 使用 libuv 的 check handles,在事件循环的固定阶段执行
- Node.js 限制 `nextTickQueue` 递归深度(`process.maxTickDepth` 已废弃,现在无限制但不推荐滥用)

</details>

## 二、事件循环机制

**Q6: 详细描述 Node.js 的事件循环机制**

<details>
<summary>查看答案</summary>

Node.js 事件循环分为 6 个阶段:

```
   ┌───────────────────────────┐
┌─>│           timers          │ 执行 setTimeout/setInterval 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks     │ 执行延迟到下一个循环的 I/O 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │ 内部使用
│  └─────────────┬─────────────┘      ┌───────────────┐
│  ┌─────────────┴─────────────┐      │   incoming:   │
│  │           poll            │<─────┤  connections, │
│  └─────────────┬─────────────┘      │   data, etc.  │
│  ┌─────────────┴─────────────┐      └───────────────┘
│  │           check           │ 执行 setImmediate 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
└──┤      close callbacks      │ 执行 close 事件回调
   └───────────────────────────┘
```

**各阶段详解:**

1. **timers 阶段:**
   - 执行 `setTimeout()` 和 `setInterval()` 的回调
   - 由 poll 阶段控制何时执行
   - 时间精度受限于系统和当前执行代码的影响

2. **pending callbacks 阶段:**
   - 执行推迟到下一个循环迭代的 I/O 回调
   - 例如 TCP 错误回调(ECONNREFUSED)

3. **idle, prepare 阶段:**
   - 仅内部使用,准备工作

4. **poll 阶段(核心):**
   - 检索新的 I/O 事件,执行 I/O 回调(除 close、timers、setImmediate)
   - 如果 poll 队列不为空:同步执行所有回调直到队列清空或达到系统限制
   - 如果 poll 队列为空:
     - 若有 `setImmediate()` 回调,结束 poll 进入 check 阶段
     - 若无,等待回调被添加并立即执行
     - 若有到期的 timer,回到 timers 阶段

5. **check 阶段:**
   - 执行 `setImmediate()` 回调
   - `setImmediate()` 实际上是特殊的 timer,在 poll 阶段空闲后立即执行

6. **close callbacks 阶段:**
   - 执行关闭回调,如 `socket.on('close', ...)`

**微任务队列执行时机:**

```javascript
// 每个阶段之间都会清空微任务队列
// 执行顺序: nextTickQueue → Promise microtask queue

┌─────────────┐
│   Phase 1   │
└──────┬──────┘
       │
       ├──> process.nextTick 队列
       │
       ├──> Promise 微任务队列
       │
┌──────▼──────┐
│   Phase 2   │
└──────┬──────┘
       │
       ├──> process.nextTick 队列
       │
       └──> Promise 微任务队列
```

**关键理解点:**

1. **为什么 setTimeout(fn, 0) 不一定比 setImmediate 快?**
   ```javascript
   // 非 I/O 回调中:顺序不确定(取决于进程性能)
   setTimeout(() => console.log('timeout'), 0)
   setImmediate(() => console.log('immediate'))

   // I/O 回调中:setImmediate 总是先执行
   fs.readFile(__filename, () => {
     setTimeout(() => console.log('timeout'), 0)
     setImmediate(() => console.log('immediate'))
     // 输出: immediate → timeout (已在 poll 阶段,直接进 check)
   })
   ```

2. **poll 阶段的阻塞计算:**
   ```javascript
   // poll 阶段会计算等待时间
   const timeout = Math.min(
     最近的 timer 到期时间,
     系统相关的时间限制
   )
   ```

3. **浏览器 vs Node.js 事件循环:**
   - 浏览器:每执行一个宏任务后清空微任务队列
   - Node.js:每个阶段结束后清空微任务队列(批量处理宏任务)

**实战应用:**

```javascript
// 利用 setImmediate 拆分 CPU 密集任务
function processLargeArray(array) {
  const chunk = 100
  let index = 0

  function process() {
    const end = Math.min(index + chunk, array.length)
    for (let i = index; i < end; i++) {
      // 处理 array[i]
    }
    index = end
    if (index < array.length) {
      setImmediate(process) // 让出事件循环,不阻塞 I/O
    }
  }
  process()
}
```

</details>

**Q8: 什么是事件循环饥饿?如何避免?**

<details>
<summary>查看答案</summary>

**事件循环饥饿:**
当 process.nextTick() 或其他微任务不断产生新的微任务,导致事件循环无法进入下一个阶段。

```javascript
// 错误示例 - 导致饥饿
function loop() {
  process.nextTick(loop)
}
loop()
// setTimeout 永远不会执行
setTimeout(() => console.log('never run'), 0)
```

**避免方法:**

1. 限制递归调用深度
2. 使用 setImmediate() 替代 process.nextTick()
3. 将任务分批处理
4. 监控事件循环延迟

```javascript
// 正确示例
function loop(count = 0) {
  if (count < 1000) {
    setImmediate(() => loop(count + 1))
  }
}
loop()
```

</details>

## 三、模块系统

**Q9: require() 的加载机制是怎样的?**

<details>
<summary>查看答案</summary>

**加载顺序:**

1. 如果是核心模块(如 fs、http),直接返回
2. 如果以 `/` 开头,按绝对路径查找
3. 如果以 `./` 或 `../` 开头,按相对路径查找
4. 否则,在 node_modules 中查找

**文件查找规则:**

1. 如果有明确后缀,直接加载
2. 否则依次尝试: .js → .json → .node
3. 如果是目录,查找 package.json 的 main 字段
4. 否则查找 index.js → index.json → index.node

**缓存机制:**

- 模块加载后会缓存在 require.cache 中
- 再次 require 直接返回缓存
- 可通过 delete require.cache[path] 清除缓存

```javascript
// 第一次加载
const mod1 = require('./module')
// 第二次直接从缓存返回
const mod2 = require('./module')
console.log(mod1 === mod2) // true
```

</details>

**Q10: module.exports 和 exports 有什么区别?**

<details>
<summary>查看答案</summary>

```javascript
// 初始状态
exports = module.exports = {}

// 正确用法 1
module.exports = {
  name: 'test',
}

// 正确用法 2
exports.name = 'test'

// 错误用法 - exports 指向被改变,不再指向 module.exports
exports = {
  name: 'test',
}

// 正确理解
console.log(exports === module.exports) // 初始时 true
```

**原则:**

- 最终导出的是 `module.exports`
- `exports` 只是 `module.exports` 的引用
- 不要重新赋值 `exports`,否则会断开引用
- 导出单个值时用 `module.exports`,导出多个属性时用 `exports`

</details>

**Q11: 如何处理循环引用?**

<details>
<summary>查看答案</summary>

```javascript
// a.js
console.log('a starting')
exports.done = false
const b = require('./b.js')
console.log('in a, b.done =', b.done)
exports.done = true
console.log('a done')

// b.js
console.log('b starting')
exports.done = false
const a = require('./a.js')
console.log('in b, a.done =', a.done)
exports.done = true
console.log('b done')

// main.js
const a = require('./a.js')
const b = require('./b.js')
console.log('in main, a.done =', a.done, ', b.done =', b.done)

/* 输出:
a starting
b starting
in b, a.done = false  // a 还未完成加载
b done
in a, b.done = true
a done
in main, a.done = true , b.done = true
*/
```

**处理建议:**

1. 重构代码,避免循环依赖
2. 延迟 require(在函数内部引用)
3. 使用依赖注入
4. 将共同依赖提取到第三个模块

</details>

## 四、核心模块深入

### 4.1 Stream 流

**Q12: Stream 有哪几种类型?各自的应用场景是什么?**

<details>
<summary>查看答案</summary>

**四种类型:**

1. **Readable(可读流)**: fs.createReadStream、http.IncomingMessage
2. **Writable(可写流)**: fs.createWriteStream、http.ServerResponse
3. **Duplex(双工流)**: net.Socket、TCP socket
4. **Transform(转换流)**: zlib.createGzip、crypto.createCipher

```javascript
const fs = require('fs')
const zlib = require('zlib')

// 读取文件 -> 压缩 -> 写入文件
fs.createReadStream('input.txt')
  .pipe(zlib.createGzip())
  .pipe(fs.createWriteStream('input.txt.gz'))
```

**深入理解:**

1. **Stream 的两种模式:**
   - **Flowing Mode(流动模式)**: 数据自动读取,通过事件传递
     - 触发方式: `pipe()`、监听 `data` 事件、调用 `resume()`
   - **Paused Mode(暂停模式)**: 需手动调用 `read()` 读取数据
     - 默认模式,可通过 `pause()` 切换

2. **内部缓冲机制:**
   ```javascript
   // highWaterMark: 缓冲区阈值(默认 16KB)
   const readable = fs.createReadStream('file.txt', {
     highWaterMark: 64 * 1024, // 64KB
   })

   // 当缓冲区超过 highWaterMark,readable._readableState.length 增长
   // 背压机制会暂停读取,直到消费者处理完数据
   ```

3. **对象模式 vs 普通模式:**
   ```javascript
   // 普通模式: 处理 Buffer/String,highWaterMark 单位为字节
   const normalStream = new Readable({ highWaterMark: 1024 })

   // 对象模式: 处理任意 JS 对象,highWaterMark 单位为对象数量
   const objectStream = new Readable({
     objectMode: true,
   ```

**Q16: Buffer 的内存分配机制是怎样的?**

<details>
<summary>查看答案</summary>

**分配策略(Slab 分配器):**

1. **小对象(<4KB)**: 使用预分配的内存池(8KB Slab)
2. **大对象(≥4KB)**: 直接从 C++ 层分配内存,绕过 V8 堆

```javascript
// 小对象 - 从 8KB 内存池分配
const buf1 = Buffer.alloc(100) // 从当前 Slab 切分 100 字节

// 大对象 - 直接分配(不经过内存池)
const buf2 = Buffer.alloc(8192) // 超过阈值,独立分配
```

**内存池工作原理:**

```javascript
// Node.js 内部维护一个 8KB 的 poolSize
Buffer.poolSize = 8 * 1024 // 8KB

// 分配流程:
// 1. 首次分配小 Buffer 时,创建 8KB 的 Slab
// 2. 从 Slab 切分所需大小
// 3. 记录偏移量,下次从剩余空间继续分配
// 4. Slab 空间不足时,创建新 Slab(旧 Slab 等待 GC)

let poolOffset = 0
let allocPool = Buffer.allocUnsafe(Buffer.poolSize)

function allocate(size) {
  if (size > Buffer.poolSize >>> 1) {
    // 大于 4KB,直接分配
    return Buffer.allocUnsafe(size)
  }
  if (size > Buffer.poolSize - poolOffset) {
    // 当前 Slab 空间不足,创建新 Slab
    allocPool = Buffer.allocUnsafe(Buffer.poolSize)
    poolOffset = 0
  }
  const buf = allocPool.slice(poolOffset, poolOffset + size)
  poolOffset += size
  return buf
}
```

**allocUnsafe vs alloc vs allocUnsafeSlow:**

| 方法               | 内存池 | 初始化 | 性能     | 安全性 | 使用场景                       |
| ------------------ | ------ | ------ | -------- | ------ | ------------------------------ |
| alloc()            | ✓      | 填充 0 | 中等     | 安全   | 通用场景                       |
| allocUnsafe()      | ✓      | 不初始化 | 最快     | 不安全 | 立即填充数据,性能敏感          |
| allocUnsafeSlow()  | ✗      | 不初始化 | 较慢     | 不安全 | 不希望影响内存池,短期使用      |

```javascript
// 1. Buffer.alloc() - 安全但慢
const safe = Buffer.alloc(10) // <Buffer 00 00 00 00 00 00 00 00 00 00>

// 2. Buffer.allocUnsafe() - 快但可能包含旧数据
const unsafe = Buffer.allocUnsafe(10) // <Buffer 可能是任意值>
unsafe.fill(0) // 使用前务必初始化

// 3. Buffer.allocUnsafeSlow() - 不使用内存池
const slow = Buffer.allocUnsafeSlow(10) // 直接从系统分配

// 性能对比(分配 1000 万次 100 字节):
// allocUnsafe: ~100ms
// alloc: ~300ms (多了 fill(0) 操作)
```

**内存管理最佳实践:**

```javascript
// 1. 重用 Buffer(避免频繁分配)
class BufferPool {
  constructor(size, count) {
    this.pool = Array.from({ length: count }, () => Buffer.alloc(size))
    this.available = [...this.pool]
  }
  acquire() {
    return this.available.pop() || Buffer.alloc(this.size)
  }
  release(buf) {
    buf.fill(0) // 清空数据
    this.available.push(buf)
  }
}

// 2. 大文件使用 Stream(避免一次性分配大 Buffer)
fs.createReadStream('large.file', { highWaterMark: 64 * 1024 })

// 3. Buffer 切片注意事项
const parent = Buffer.from('Hello World')
const child = parent.slice(0, 5) // 共享底层内存!
child[0] = 0x48 // 会修改 parent

// 安全做法:复制而非切片
const copy = Buffer.from(parent.slice(0, 5))

// 4. 及时释放引用
let bigBuffer = Buffer.alloc(100 * 1024 * 1024) // 100MB
processData(bigBuffer)
bigBuffer = null // 让 GC 回收
```

**底层原理:**

- Buffer 基于 V8 的 `Uint8Array`,但内存分配在 **V8 堆外**(外部内存)
- 优势: 不受 V8 堆大小限制(默认 ~1.4GB),减轻 GC 压力
- 劣势: 需要手动管理,不当使用可能导致内存泄漏
- `process.memoryUsage()`:
  - `heapUsed`: V8 堆内存
  - `external`: Buffer 等外部内存
  - `arrayBuffers`: ArrayBuffer 和 SharedArrayBuffer 占用

**性能陷阱:**

```javascript
// 陷阱 1: 频繁创建小 Buffer
for (let i = 0; i < 1000000; i++) {
  const buf = Buffer.alloc(10) // 每次分配,GC 压力大
}

// 优化: 重用 Buffer
const reusable = Buffer.alloc(10)
for (let i = 0; i < 1000000; i++) {
  reusable.fill(i) // 重用同一个 Buffer
}

// 陷阱 2: 大 Buffer 拼接
let result = Buffer.alloc(0)
for (const chunk of chunks) {
  result = Buffer.concat([result, chunk]) // O(n²) 复杂度!
}

// 优化: 先收集再一次性拼接
const chunks = []
// ... 收集 chunks
const result = Buffer.concat(chunks)
```

```javascript
        console.log('所有进程已重启')
        return
```

```
V8 Heap Memory
├── 新生代(New Space) - 32MB(64位)/16MB(32位)
│   ├── From Space (Semi-space)
│   └── To Space (Semi-space)
├── 老生代(Old Space) - 默认 1.4GB(64位)/700MB(32位)
│   ├── Old Pointer Space (指针对象)
│   └── Old Data Space (数据对象)
├── Large Object Space - 大对象(>256KB)
├── Code Space - JIT 编译后的代码
└── Map Space - 隐藏类和元信息
```

**垃圾回收算法详解:**

**1. Scavenge(新生代) - 副垃圾回收器(Minor GC)**

```
初始状态:
┌─────────────┬─────────────┐
│ From Space  │  To Space   │
│   (使用中)   │   (空闲)    │
└─────────────┴─────────────┘

GC 执行:
1. 检查 From Space 中的对象
2. 存活对象 → 复制到 To Space
3. 清空 From Space
4. 交换 From 和 To 指针

特点:
- 采用 Cheney 算法(标记-复制)
- Stop-The-World(暂停应用)
- 速度极快(1-10ms)
- 空间换时间(50%空间利用率)
```

**2. Mark-Sweep & Mark-Compact(老生代) - 主垃圾回收器(Major GC)**

```javascript
// Mark-Sweep(标记清除):
// 1. 标记阶段: 从根对象(全局、栈)遍历,标记可达对象
// 2. 清除阶段: 回收未标记对象
// 缺点: 产生内存碎片

// Mark-Compact(标记整理):
// 1. 标记存活对象
// 2. 整理阶段: 将存活对象移动到一端,清理另一端
// 3. 更新指针引用
// 优点: 无碎片,缺点: 速度慢(需移动对象)

// V8 策略: 优先使用 Mark-Sweep,碎片化严重时触发 Mark-Compact
```

**3. 增量标记(Incremental Marking)**

```
传统 GC: 一次性完成(Stop-The-World 100ms+)
┌──────────────────────────────┐
│   ████████████████████████   │ GC 暂停应用
└──────────────────────────────┘

增量标记: 分片执行(多次 5-10ms)
┌───┐   ┌───┐   ┌───┐   ┌───┐
│ █ │   │ █ │   │ █ │   │ █ │ 交替执行
└───┘   └───┘   └───┘   └───┘
  ↕       ↕       ↕       ↕
 应用    应用    应用    应用

优势: 减少单次暂停时间,提升响应性
实现: 三色标记法(白-灰-黑)
```

**4. 并发标记(Concurrent Marking)**

```
主线程:  ████████████████████████ (应用继续运行)
GC 线程: ──── ████████ ──── (后台标记)

Orinoco GC(V8 新版):
- 并发标记: GC 在独立线程执行,不阻塞主线程
- 并行清除: 多个 GC 线程同时清除
- 惰性清除: 延迟清除,按需进行
```

**1. 注入攻击防御(Critical):**

```javascript
  username: req.body.username, // { $ne: null } 绕过认证

// ✅ 类型验证和白名单

if (typeof req.body.username !== 'string') {
  throw new Error('Invalid input')
}

// ✅ 使用 mongoose schema(自动验证)

// 攻击: userFile = "file.txt; rm -rf /"

// ✅ 使用参数数组(不经过 shell)

  // 安全: 参数不会被 shell 解析
```

## 八、框架相关

**Q24: Express 和 Koa 有什么区别?**

<details>
<summary>查看答案</summary>

**核心差异:**

```javascript
// Express - 回调风格
app.get('/user/:id', (req, res, next) => {
  User.findById(req.params.id, (err, user) => {
    if (err) return next(err)
    res.json(user)
  })
})

// Koa - async/await
app.use(async (ctx, next) => {
  try {
    const user = await User.findById(ctx.params.id)
    ctx.body = user
  } catch (err) {
    ctx.throw(500, err)
  }
})
```

**详细对比:**

| 特性           | Express                       | Koa                           |
| -------------- | ----------------------------- | ----------------------------- |
| 发布时间       | 2010                          | 2013(Express 原班人马)        |
| 异步处理       | 回调/Promise                  | async/await(原生支持)         |
| 错误处理       | 需手动 next(err)              | try-catch 或 ctx.throw()      |
| 中间件执行     | 线性(顺序执行)                | 洋葱模型(可回溯)              |
| 内置功能       | 路由、模板、静态文件          | 极简内核(需中间件)            |
| 响应处理       | res.send()/json()/render()    | ctx.body = (自动序列化)       |
| 请求对象       | req, res, next                | ctx(封装 request/response)    |
| 路由           | 内置                          | 需 @koa/router                |
| 体积           | ~200KB                        | ~50KB                         |
| 性能           | 基准                          | 略快 5-15%(更轻量)            |
| 错误传播       | 需在每个中间件手动 next(err)  | 自动冒泡到顶层                |
| Context 绑定   | this 指向全局                 | this 指向 ctx                 |

**1. 中间件执行模式对比:**

```javascript
// Express: 线性执行(单向)
app.use((req, res, next) => {
  console.log('1')
  next()
  console.log('2') // 立即执行,不等待后续中间件
})

app.use((req, res, next) => {
  console.log('3')
  res.send('Done')
})
// 输出: 1 3 2

// Koa: 洋葱模型(双向)
app.use(async (ctx, next) => {
  console.log('1')
  await next() // 等待后续中间件完成
  console.log('6')
})

app.use(async (ctx, next) => {
  console.log('2')
  await next()
  console.log('5')
})

app.use(async (ctx, next) => {
  console.log('3')
  ctx.body = 'Hello'
  console.log('4')
})
// 输出: 1 2 3 4 5 6
```

**洋葱模型的优势:**

```javascript
// 场景 1: 统计请求耗时
app.use(async (ctx, next) => {
  const start = Date.now()
  await next() // 执行后续中间件
  const duration = Date.now() - start
  ctx.set('X-Response-Time', `${duration}ms`)
  console.log(`${ctx.method} ${ctx.url} - ${duration}ms`)
})

// 场景 2: 数据库事务(确保响应后再提交)
app.use(async (ctx, next) => {
  const trx = await db.transaction()
  ctx.trx = trx
  try {
    await next() // 业务逻辑
    await trx.commit() // 响应成功后提交
  } catch (err) {
    await trx.rollback() // 出错回滚
    throw err
  }
})

// 场景 3: 后置处理响应
app.use(async (ctx, next) => {
  await next()
  // 可以在这里修改响应
  if (ctx.body && typeof ctx.body === 'object') {
    ctx.body = {
      success: true,
      data: ctx.body,
      timestamp: Date.now(),
    }
  }
})
```

**2. 错误处理对比:**

```javascript
// Express: 需要每个中间件手动传递错误
app.get('/user/:id', (req, res, next) => {
  User.findById(req.params.id, (err, user) => {
    if (err) return next(err) // 必须手动传递
    res.json(user)
  })
})

// 集中错误处理(必须放在最后)
app.use((err, req, res, next) => {
  console.error(err.stack)
  res.status(500).send('Something broke!')
})

// Koa: 错误自动冒泡
app.use(async (ctx, next) => {
  const user = await User.findById(ctx.params.id) // 错误自动向上冒泡
  ctx.body = user
})

// 顶层错误处理(自动捕获所有错误)
app.on('error', (err, ctx) => {
  console.error('Server error', err, ctx)
})

// 或使用中间件
app.use(async (ctx, next) => {
  try {
    await next()
  } catch (err) {
    ctx.status = err.status || 500
    ctx.body = { error: err.message }
    ctx.app.emit('error', err, ctx)
  }
})
```

**3. Context 对象的优势:**

```javascript
// Express: req 和 res 分离
app.get('/users', (req, res) => {
  const page = req.query.page
  const userAgent = req.get('User-Agent')
  res.set('X-Custom-Header', 'value')
  res.json({ users: [] })
})

// Koa: ctx 统一封装,更简洁
app.use(async (ctx) => {
  const page = ctx.query.page // 等价于 ctx.request.query.page
  const userAgent = ctx.get('User-Agent') // 代理到 request
  ctx.set('X-Custom-Header', 'value') // 代理到 response
  ctx.body = { users: [] } // 等价于 ctx.response.body
  // ctx 提供了便捷方法,自动代理到 request 或 response
})

// Koa 的 ctx 还提供了实用方法
ctx.assert(ctx.state.user, 401, 'User not authenticated')
ctx.throw(400, 'Invalid input')
ctx.redirect('/login')
```

**选择建议:**

**选择 Express 如果:**
- 需要快速开发原型
- 团队熟悉传统 Node.js 开发
- 需要丰富的第三方中间件
- 项目不复杂,不需要精细控制中间件执行
- 需要模板引擎、静态文件等开箱即用功能

**选择 Koa 如果:**
- 追求现代化、优雅的代码风格
- 需要洋葱模型(如统计耗时、事务管理)

```javascript
// Express 风格中间件
class ExpressLikeApp {
  constructor() {
    this.middlewares = []
  }

  use(middleware) {
    this.middlewares.push(middleware)
  }

  handle(req, res) {
    let index = 0

    const next = (err) => {
      if (err) {
        // 错误处理
        res.statusCode = 500
        res.end(err.message)
        return
      }

      if (index >= this.middlewares.length) {
        res.statusCode = 404
        res.end('Not Found')
        return
      }

      const middleware = this.middlewares[index++]
      try {
        middleware(req, res, next)
      } catch (err) {
        next(err)
      }
    }

    next()
  }
}

// Koa 风格中间件(洋葱模型)
class KoaLikeApp {
  constructor() {
    this.middlewares = []
  }

  use(middleware) {
    this.middlewares.push(middleware)
  }

  compose(middlewares) {
    return function (context) {
      let index = -1

      function dispatch(i) {
        if (i <= index) {
          return Promise.reject(new Error('next() called multiple times'))
        }
        index = i

        if (i >= middlewares.length) {
          return Promise.resolve()
        }

        const middleware = middlewares[i]
        try {
          return Promise.resolve(middleware(context, () => dispatch(i + 1)))
        } catch (err) {
          return Promise.reject(err)
        }
      }

      return dispatch(0)
    }
  }

  async handle(ctx) {
    const fn = this.compose(this.middlewares)
    try {
      await fn(ctx)
    } catch (err) {
      console.error(err)
      ctx.status = 500
      ctx.body = 'Internal Server Error'
    }
  }
}

// 使用示例
const app = new KoaLikeApp()

app.use(async (ctx, next) => {
  console.log('middleware 1 start')
  await next()
  console.log('middleware 1 end')
})

app.use(async (ctx, next) => {
  console.log('middleware 2 start')
  await next()
  console.log('middleware 2 end')
})

app.use(async (ctx, next) => {
  console.log('middleware 3')
  ctx.body = 'Hello World'
})
```

| ------------ | ---------------------- | -------------------- | ------------------ |

**1. 固定窗口算法:**

```javascript
class FixedWindowRateLimiter {
  constructor(limit, window) {
    this.limit = limit // 请求次数限制
    this.window = window // 时间窗口(ms)
    this.counters = new Map()
  }

  isAllowed(key) {
    const now = Date.now()
    const windowStart = Math.floor(now / this.window) * this.window
    const counterKey = `${key}:${windowStart}`

    const count = this.counters.get(counterKey) || 0

    if (count < this.limit) {
      this.counters.set(counterKey, count + 1)
      // 清理过期数据
      setTimeout(() => this.counters.delete(counterKey), this.window)
      return true
    }

    return false
  }
}

// 临界问题示例:
// 窗口: [0s-1s] 限制 100 次
// 0.9s: 100 次请求 ✓
// 1.0s: 新窗口,100 次请求 ✓
// 结果: 0.1s 内处理 200 次请求(2x 超限!)
```

**2. 滑动窗口算法(解决临界问题):**

```javascript
class SlidingWindowRateLimiter {
  constructor(limit, window) {
    this.limit = limit
    this.window = window
    this.requests = new Map()
  }

  isAllowed(key) {
    const now = Date.now()
    const requests = this.requests.get(key) || []

    // 移除过期请求
    const validRequests = requests.filter((time) => now - time < this.window)

    if (validRequests.length < this.limit) {
      validRequests.push(now)
      this.requests.set(key, validRequests)
      return true
    }

    return false
  }

  // 优化: 使用滑动窗口计数(减少内存)
  isAllowedOptimized(key) {
    const now = Date.now()
    const currentWindow = Math.floor(now / this.window)
    const previousWindow = currentWindow - 1

    const currentCount = this.counters.get(`${key}:${currentWindow}`) || 0
    const previousCount = this.counters.get(`${key}:${previousWindow}`) || 0

    // 加权计算
    const previousWeight = 1 - (now % this.window) / this.window
    const estimatedCount = currentCount + previousCount * previousWeight

    if (estimatedCount < this.limit) {
      this.counters.set(`${key}:${currentWindow}`, currentCount + 1)
      return true
    }

    return false
  }
}
```

**3. 令牌桶算法(最常用):**

```javascript
class TokenBucketRateLimiter {
  constructor(capacity, refillRate) {
    this.capacity = capacity // 桶容量(最大突发)
    this.refillRate = refillRate // 每秒补充令牌数(平均速率)
    this.buckets = new Map()
  }

  isAllowed(key, tokens = 1) {
    const now = Date.now()
    let bucket = this.buckets.get(key)

    if (!bucket) {
      bucket = {
        tokens: this.capacity,
        lastRefill: now,
      }
      this.buckets.set(key, bucket)
    }

    // 补充令牌(匀速)
    const timePassed = (now - bucket.lastRefill) / 1000
    const tokensToAdd = timePassed * this.refillRate
    bucket.tokens = Math.min(this.capacity, bucket.tokens + tokensToAdd)
    bucket.lastRefill = now

    // 消费令牌
    if (bucket.tokens >= tokens) {
      bucket.tokens -= tokens
      return true
    }

    return false
  }

  // 获取剩余令牌数
  getTokens(key) {
    const bucket = this.buckets.get(key)
    if (!bucket) return this.capacity

    const now = Date.now()
    const timePassed = (now - bucket.lastRefill) / 1000
    const tokensToAdd = timePassed * this.refillRate
    return Math.min(this.capacity, bucket.tokens + tokensToAdd)
  }
}

// 场景: API 限流,允许短时突发
// capacity: 100(最多积累 100 个令牌)
// refillRate: 10(每秒补充 10 个令牌)
// 结果: 平均 10 req/s,允许突发 100 req(然后降为 10 req/s)
```

```javascript
class LeakyBucketRateLimiter {
  constructor(capacity, leakRate) {
    this.capacity = capacity // 桶容量
    this.leakRate = leakRate // 每秒漏出请求数(恒定速率)
    this.buckets = new Map()
    this.queue = new Map()
  }

  async processRequest(key, handler) {
    const now = Date.now()
    let bucket = this.buckets.get(key)

    if (!bucket) {
      bucket = { water: 0, lastLeak: now }
      this.buckets.set(key, bucket)
    }

    // 漏水(处理请求)
    const timePassed = (now - bucket.lastLeak) / 1000
    const leaked = Math.floor(timePassed * this.leakRate)
    bucket.water = Math.max(0, bucket.water - leaked)
    bucket.lastLeak = now

    // 请求入桶
    if (bucket.water < this.capacity) {
      bucket.water += 1
      // 延迟处理(模拟漏桶恒定速率)
      const delay = (bucket.water / this.leakRate) * 1000
      await new Promise((resolve) => setTimeout(resolve, delay))
      return handler()
    } else {
      throw new Error('Bucket overflow')
    }
  }
}

// 场景: 保护下游服务,匀速处理请求
// capacity: 100
// leakRate: 10
// 结果: 无论请求多快,始终以 10 req/s 恒定速率处理
```

```javascript
    // 自动重启
    cluster.fork()

      const worker = workers[index]
      if (!worker) return

        console.log(`进程 ${worker.process.pid} 已重启`)
        cluster.fork()
        restartWorker(index + 1)

    }
```
