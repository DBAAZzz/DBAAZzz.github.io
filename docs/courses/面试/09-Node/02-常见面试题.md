## 一、Node.js 基础

### 1.1 核心概念

**Q1: 什么是 Node.js?它有哪些特点?**

<details>
<summary>查看答案</summary>

Node.js 是一个基于 Chrome V8 引擎的 JavaScript 运行时环境,主要特点:

- **事件驱动**: 基于事件循环机制处理异步操作
- **非阻塞 I/O**: I/O 操作不会阻塞主线程
- **单线程**: 主线程是单线程,但通过事件循环和线程池处理并发
- **跨平台**: 支持 Windows、Linux、macOS 等
- **高性能**: V8 引擎提供快速的 JavaScript 执行
- **轻量高效**: 适合 I/O 密集型应用

</details>

**Q2: Node.js 为什么采用单线程模型?有什么优缺点?**

<details>
<summary>查看答案</summary>

**原因:**

- 避免多线程上下文切换的开销
- 避免线程同步和死锁问题
- 简化编程模型
- JavaScript 本身是单线程的

**优点:**

- 不需要考虑线程同步
- 降低内存消耗
- 避免死锁
- 实现简单

**缺点:**

- 无法利用多核 CPU(需通过 cluster 或 worker_threads 解决)
- CPU 密集型任务会阻塞事件循环
- 单点故障影响整个应用

</details>

```javascript
  /* ... */

```

**Q5: process.nextTick() 和 setImmediate() 有什么区别?**

<details>
<summary>查看答案</summary>

```javascript
// process.nextTick(): 在当前执行栈结束后立即执行
// setImmediate(): 在事件循环的 check 阶段执行

console.log('1')

setImmediate(() => {
  console.log('2')
})

process.nextTick(() => {
  console.log('3')
})

console.log('4')

// 输出: 1 4 3 2
```

**区别:**

- `process.nextTick()` 在微任务队列,优先级最高
- `setImmediate()` 在宏任务队列的 check 阶段
- `process.nextTick()` 会在同一个阶段内执行完所有回调
- 过度使用 `process.nextTick()` 可能导致事件循环饥饿

</details>

## 二、事件循环机制

**Q6: 详细描述 Node.js 的事件循环机制**

<details>
<summary>查看答案</summary>

Node.js 事件循环分为 6 个阶段:

```
   ┌───────────────────────────┐
┌─>│           timers          │ 执行 setTimeout/setInterval 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │     pending callbacks     │ 执行延迟到下一个循环的 I/O 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
│  │       idle, prepare       │ 内部使用
│  └─────────────┬─────────────┘      ┌───────────────┐
│  ┌─────────────┴─────────────┐      │   incoming:   │
│  │           poll            │<─────┤  connections, │
│  └─────────────┬─────────────┘      │   data, etc.  │
│  ┌─────────────┴─────────────┐      └───────────────┘
│  │           check           │ 执行 setImmediate 回调
│  └─────────────┬─────────────┘
│  ┌─────────────┴─────────────┐
└──┤      close callbacks      │ 执行 close 事件回调
   └───────────────────────────┘
```

**微任务队列:**

- process.nextTick 队列(优先级最高)
- Promise 队列

每个阶段完成后都会检查并清空微任务队列。

</details>

**Q8: 什么是事件循环饥饿?如何避免?**

<details>
<summary>查看答案</summary>

**事件循环饥饿:**
当 process.nextTick() 或其他微任务不断产生新的微任务,导致事件循环无法进入下一个阶段。

```javascript
// 错误示例 - 导致饥饿
function loop() {
  process.nextTick(loop)
}
loop()
// setTimeout 永远不会执行
setTimeout(() => console.log('never run'), 0)
```

**避免方法:**

1. 限制递归调用深度
2. 使用 setImmediate() 替代 process.nextTick()
3. 将任务分批处理
4. 监控事件循环延迟

```javascript
// 正确示例
function loop(count = 0) {
  if (count < 1000) {
    setImmediate(() => loop(count + 1))
  }
}
loop()
```

</details>

## 三、模块系统

**Q9: require() 的加载机制是怎样的?**

<details>
<summary>查看答案</summary>

**加载顺序:**

1. 如果是核心模块(如 fs、http),直接返回
2. 如果以 `/` 开头,按绝对路径查找
3. 如果以 `./` 或 `../` 开头,按相对路径查找
4. 否则,在 node_modules 中查找

**文件查找规则:**

1. 如果有明确后缀,直接加载
2. 否则依次尝试: .js → .json → .node
3. 如果是目录,查找 package.json 的 main 字段
4. 否则查找 index.js → index.json → index.node

**缓存机制:**

- 模块加载后会缓存在 require.cache 中
- 再次 require 直接返回缓存
- 可通过 delete require.cache[path] 清除缓存

```javascript
// 第一次加载
const mod1 = require('./module')
// 第二次直接从缓存返回
const mod2 = require('./module')
console.log(mod1 === mod2) // true
```

</details>

**Q10: module.exports 和 exports 有什么区别?**

<details>
<summary>查看答案</summary>

```javascript
// 初始状态
exports = module.exports = {}

// 正确用法 1
module.exports = {
  name: 'test',
}

// 正确用法 2
exports.name = 'test'

// 错误用法 - exports 指向被改变,不再指向 module.exports
exports = {
  name: 'test',
}

// 正确理解
console.log(exports === module.exports) // 初始时 true
```

**原则:**

- 最终导出的是 `module.exports`
- `exports` 只是 `module.exports` 的引用
- 不要重新赋值 `exports`,否则会断开引用
- 导出单个值时用 `module.exports`,导出多个属性时用 `exports`

</details>

**Q11: 如何处理循环引用?**

```javascript
console.log('a starting')

console.log('in a, b.done =', b.done)
```

### 4.1 Stream 流

**Q12: Stream 有哪几种类型?各自的应用场景是什么?**

<details>
<summary>查看答案</summary>

**四种类型:**

1. **Readable(可读流)**: fs.createReadStream、http.IncomingMessage
2. **Writable(可写流)**: fs.createWriteStream、http.ServerResponse
3. **Duplex(双工流)**: net.Socket、TCP socket
4. **Transform(转换流)**: zlib.createGzip、crypto.createCipher

```javascript

// 读取文件 -> 压缩 -> 写入文件
fs.createReadStream('input.txt')
  .pipe(zlib.createGzip())
  .pipe(fs.createWriteStream('input.txt.gz'))
```

**应用场景:**

- 大文件处理(避免内存溢出)
- 网络传输(视频、音频流)
- 数据转换(压缩、加密)
- 实时数据处理

</details>

**Q13: 什么是背压(Backpressure)?如何处理?**

<details>
<summary>查看答案</summary>

**背压概念:**
当数据生产速度大于消费速度时,数据会在缓冲区堆积,导致内存占用过高。

```javascript
// 错误示例 - 可能导致内存问题
const readable = getReadableStream()
const writable = getWritableStream()

readable.on('data', (chunk) => {
  writable.write(chunk) // 不管是否写入成功
})

// 正确示例 1 - 手动处理背压
readable.on('data', (chunk) => {
  const canContinue = writable.write(chunk)
  if (!canContinue) {
    readable.pause() // 暂停读取
  }
})

writable.on('drain', () => {
  readable.resume() // 恢复读取
})

// 正确示例 2 - 使用 pipe(自动处理背压)
readable.pipe(writable)
```

**pipe 的优势:**

- 自动处理背压
- 自动管理数据流速
- 自动处理错误
- 代码简洁

</details>

**Q14: 实现一个自定义的 Transform 流**

<details>
<summary>查看答案</summary>

```javascript
const { Transform } = require('stream')

// 将输入转为大写的转换流
class UpperCaseTransform extends Transform {
  _transform(chunk, encoding, callback) {
    try {
      const upperChunk = chunk.toString().toUpperCase()
      this.push(upperChunk)
      callback()
    } catch (err) {
      callback(err)
    }
  }
}

// 使用示例
const upperCase = new UpperCaseTransform()

process.stdin.pipe(upperCase).pipe(process.stdout)

// 更复杂的例子 - CSV 解析器
class CSVParser extends Transform {
  constructor(options) {
    super(options)
    this.buffer = ''
  }

  _transform(chunk, encoding, callback) {
    this.buffer += chunk.toString()
    const lines = this.buffer.split('\n')
    this.buffer = lines.pop() // 保留不完整的行

    lines.forEach((line) => {
      const columns = line.split(',')
      this.push(JSON.stringify(columns) + '\n')
    })

    callback()
  }

  _flush(callback) {
    if (this.buffer) {
      const columns = this.buffer.split(',')
      this.push(JSON.stringify(columns) + '\n')
    }
    callback()
  }
}
```

</details>

### 4.2 Buffer

**Q15: Buffer 是什么?为什么需要 Buffer?**

<details>
<summary>查看答案</summary>

**定义:**
Buffer 是 Node.js 提供的用于处理二进制数据的类,本质是一块内存区域。

**为什么需要:**

- JavaScript 原生不擅长处理二进制数据
- 处理 TCP 流、文件流等需要二进制数据
- 处理图片、视频、音频等媒体文件
- 网络协议通信需要二进制格式

```javascript
// 创建 Buffer
const buf1 = Buffer.from('hello')
const buf2 = Buffer.alloc(10) // 分配 10 字节,填充 0
const buf3 = Buffer.allocUnsafe(10) // 不初始化,性能更好但可能包含旧数据

// Buffer 与字符串转换
const buf = Buffer.from('你好', 'utf8')
console.log(buf) // <Buffer e4 bd a0 e5 a5 bd>
console.log(buf.toString('utf8')) // 你好

// Buffer 拼接
const buf4 = Buffer.concat([buf1, buf2])

// Buffer 比较
console.log(buf1.equals(buf2))
console.log(Buffer.compare(buf1, buf2))
```

</details>

**Q16: Buffer 的内存分配机制是怎样的?**

<details>
<summary>查看答案</summary>

**分配策略:**

1. **小对象(<4KB)**: 使用预分配的内存池(8KB)
2. **大对象(≥4KB)**: 直接分配内存

```javascript
// 小对象 - 从内存池分配
const buf1 = Buffer.alloc(100)

// 大对象 - 直接分配
const buf2 = Buffer.alloc(8192)
```

**allocUnsafe vs alloc:**

```javascript
// alloc: 分配后会用 0 填充,安全但稍慢
const safe = Buffer.alloc(10)

// allocUnsafe: 不初始化,可能包含旧数据,快但不安全
const unsafe = Buffer.allocUnsafe(10)
// 使用前应该先填充
unsafe.fill(0)
```

**性能优化:**

- 小数据用内存池(alloc/allocUnsafe)
- 大数据考虑使用 Stream
- 避免频繁创建小 Buffer
- 重用 Buffer 对象

</details>

## 五、进程与线程

**Q17: child_process 的 spawn、exec、execFile、fork 有什么区别?**

<details>
<summary>查看答案</summary>

```javascript
const { spawn, exec, execFile, fork } = require('child_process')

// 1. spawn: 流式处理,适合大量数据
const ls = spawn('ls', ['-lh', '/usr'])
ls.stdout.on('data', (data) => {
  console.log(`stdout: ${data}`)
})

// 2. exec: 缓冲输出,适合小量数据,可直接执行 shell 命令
exec('ls -lh /usr', (error, stdout, stderr) => {
  console.log(`stdout: ${stdout}`)
})

// 3. execFile: 直接执行文件,不经过 shell,更安全高效
execFile('node', ['--version'], (error, stdout, stderr) => {
  console.log(`stdout: ${stdout}`)
})

// 4. fork: 专门用于 Node.js 脚本,有 IPC 通道
const child = fork('child.js')
child.send({ hello: 'world' })
child.on('message', (msg) => {
  console.log('来自子进程:', msg)
})
```

**对比:**
| 方法 | 返回流 | Shell | 数据量 | IPC | 场景 |
|------|--------|-------|--------|-----|------|
| spawn | ✓ | ✗ | 大 | ✗ | 长时间运行的进程 |
| exec | ✗ | ✓ | 小 | ✗ | 简单 shell 命令 |
| execFile | ✗ | ✗ | 小 | ✗ | 执行文件 |
| fork | ✓ | ✗ | 大 | ✓ | Node.js 子进程 |

</details>

**Q18: 如何实现多进程架构?cluster 模块的工作原理是什么?**

<details>
<summary>查看答案</summary>

```javascript
const cluster = require('cluster')
const http = require('http')
const numCPUs = require('os').cpus().length

if (cluster.isMaster) {
  console.log(`主进程 ${process.pid} 正在运行`)

  // 衍生工作进程
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork()
  }

  cluster.on('exit', (worker, code, signal) => {
    console.log(`工作进程 ${worker.process.pid} 已退出`)
    // 自动重启
    cluster.fork()
  })
} else {
  // 工作进程可以共享任何 TCP 连接
  http
    .createServer((req, res) => {
      res.writeHead(200)
      res.end(`由进程 ${process.pid} 处理\n`)
    })
    .listen(8000)

  console.log(`工作进程 ${process.pid} 已启动`)
}
```

**工作原理:**

1. **主进程**: 负责管理工作进程,不处理业务
2. **工作进程**: 处理实际的业务逻辑
3. **负载均衡**: 由主进程分发连接
   - Round-Robin(轮询,非 Windows)
   - 操作系统调度(Windows)

**优势:**

- 充分利用多核 CPU
- 进程隔离,单个进程崩溃不影响其他进程
- 零停机重启(逐个重启工作进程)

</details>

**Q19: Worker Threads 与 child_process 的区别?**

<details>
<summary>查看答案</summary>

```javascript
// Worker Threads 示例
const { Worker, isMainThread, parentPort } = require('worker_threads')

if (isMainThread) {
  // 主线程
  const worker = new Worker(__filename)
  worker.on('message', (msg) => {
    console.log('来自工作线程:', msg)
  })
  worker.postMessage('hello')
} else {
  // 工作线程
  parentPort.on('message', (msg) => {
    console.log('来自主线程:', msg)
    parentPort.postMessage('world')
  })
}
```

**对比:**
| 特性 | Worker Threads | child_process |
|------|----------------|---------------|
| 资源占用 | 轻量(共享内存) | 重(独立进程) |
| 启动速度 | 快 | 慢 |
| 通信方式 | SharedArrayBuffer、postMessage | IPC |
| 隔离性 | 共享内存空间 | 完全隔离 |
| 适用场景 | CPU 密集型任务 | I/O 密集、独立任务 |

**使用场景:**

- **Worker Threads**: 图像处理、数据计算、加密解密
- **child_process**: 执行外部命令、隔离不稳定任务、多语言协作

</details>

## 六、性能优化

**Q20: Node.js 的垃圾回收机制是怎样的?**

<details>
<summary>查看答案</summary>

**内存结构:**

```
├── 新生代(New Space) - 32MB(64位)/16MB(32位)
│   ├── From Space
│   └── To Space
└── 老生代(Old Space) - 默认 1.4GB(64位)/700MB(32位)
```

**垃圾回收算法:**

1. **Scavenge(新生代)**

   - 采用 Cheney 算法
   - From Space → To Space
   - 存活对象复制到 To Space
   - 交换 From 和 To
   - 速度快,适合生命周期短的对象

2. **Mark-Sweep(标记清除,老生代)**

   - 标记阶段:标记活动对象
   - 清除阶段:回收未标记对象
   - 会产生内存碎片

3. **Mark-Compact(标记整理,老生代)**
   - 标记活动对象
   - 移动对象,整理内存
   - 消除碎片,但速度慢

**晋升策略:**
新生代对象在两种情况下晋升到老生代:

- 经历过一次 Scavenge
- To Space 使用超过 25%

**优化建议:**

```javascript
// 1. 避免全局变量
// 不好
global.cache = new Map()

// 好
function createCache() {
  return new Map()
}

// 2. 及时释放引用
let data = loadLargeData()
processData(data)
data = null // 主动释放

// 3. 使用对象池
class ObjectPool {
  constructor(factory, reset) {
    this.factory = factory
    this.reset = reset
    this.pool = []
  }

  acquire() {
    return this.pool.pop() || this.factory()
  }

  release(obj) {
    this.reset(obj)
    this.pool.push(obj)
  }
}

// 4. 监控内存
setInterval(() => {
  const usage = process.memoryUsage()
  console.log(`内存使用: ${Math.round(usage.heapUsed / 1024 / 1024)}MB`)
}, 5000)
```

```javascript
// Express 风格中间件
class ExpressLikeApp {
  constructor() {
    this.middlewares = []
  }

  use(middleware) {
    this.middlewares.push(middleware)
  }

  handle(req, res) {
    let index = 0

    const next = (err) => {
      if (err) {
        // 错误处理
        res.statusCode = 500
        res.end(err.message)
        return
      }

      if (index >= this.middlewares.length) {
        res.statusCode = 404
        res.end('Not Found')
        return
      }

      const middleware = this.middlewares[index++]
      try {
        middleware(req, res, next)
      } catch (err) {
        next(err)
      }
    }

    next()
  }
}

// Koa 风格中间件(洋葱模型)
class KoaLikeApp {
  constructor() {
    this.middlewares = []
  }

  use(middleware) {
    this.middlewares.push(middleware)
  }

  compose(middlewares) {
    return function (context) {
      let index = -1

      function dispatch(i) {
        if (i <= index) {
          return Promise.reject(new Error('next() called multiple times'))
        }
        index = i

        if (i >= middlewares.length) {
          return Promise.resolve()
        }

        const middleware = middlewares[i]
        try {
          return Promise.resolve(middleware(context, () => dispatch(i + 1)))
        } catch (err) {
          return Promise.reject(err)
        }
      }

      return dispatch(0)
    }
  }

  async handle(ctx) {
    const fn = this.compose(this.middlewares)
    try {
      await fn(ctx)
    } catch (err) {
      console.error(err)
      ctx.status = 500
      ctx.body = 'Internal Server Error'
    }
  }
}

// 使用示例
const app = new KoaLikeApp()

app.use(async (ctx, next) => {
  console.log('middleware 1 start')
  await next()
  console.log('middleware 1 end')
})

app.use(async (ctx, next) => {
  console.log('middleware 2 start')
  await next()
  console.log('middleware 2 end')
})

app.use(async (ctx, next) => {
  console.log('middleware 3')
  ctx.body = 'Hello World'
})
```

```javascript
// 1. 固定窗口算法
class FixedWindowRateLimiter {
  constructor(limit, window) {
    this.limit = limit // 请求次数限制
    this.window = window // 时间窗口(ms)
    this.counters = new Map()
  }

  isAllowed(key) {
    const now = Date.now()
    const windowStart = Math.floor(now / this.window) * this.window
    const counterKey = `${key}:${windowStart}`

    const count = this.counters.get(counterKey) || 0

    if (count < this.limit) {
      this.counters.set(counterKey, count + 1)
      // 清理过期数据
      setTimeout(() => this.counters.delete(counterKey), this.window)
      return true
    }

    return false
  }
}

// 2. 滑动窗口算法
class SlidingWindowRateLimiter {
  constructor(limit, window) {
    this.limit = limit
    this.window = window
    this.requests = new Map()
  }

  isAllowed(key) {
    const now = Date.now()
    const requests = this.requests.get(key) || []

    // 移除过期请求
    const validRequests = requests.filter((time) => now - time < this.window)

    if (validRequests.length < this.limit) {
      validRequests.push(now)
      this.requests.set(key, validRequests)
      return true
    }

    return false
  }
}

// 3. 令牌桶算法
class TokenBucketRateLimiter {
  constructor(capacity, refillRate) {
    this.capacity = capacity // 桶容量
    this.refillRate = refillRate // 每秒补充令牌数
    this.buckets = new Map()
  }

  isAllowed(key) {
    const now = Date.now()
    let bucket = this.buckets.get(key)

    if (!bucket) {
      bucket = {
        tokens: this.capacity,
        lastRefill: now,
      }
      this.buckets.set(key, bucket)
    }

    // 补充令牌
    const timePassed = (now - bucket.lastRefill) / 1000
    const tokensToAdd = timePassed * this.refillRate
    bucket.tokens = Math.min(this.capacity, bucket.tokens + tokensToAdd)
    bucket.lastRefill = now

    if (bucket.tokens >= 1) {
      bucket.tokens -= 1
      return true
    }

    return false
  }
}

// 4. 漏桶算法
class LeakyBucketRateLimiter {
  constructor(capacity, leakRate) {
    this.capacity = capacity // 桶容量
    this.leakRate = leakRate // 每秒漏出请求数
    this.buckets = new Map()
  }

  isAllowed(key) {
    const now = Date.now()
    let bucket = this.buckets.get(key)

    if (!bucket) {
      bucket = {
        water: 0,
        lastLeak: now,
      }
      this.buckets.set(key, bucket)
    }

    // 漏水
    const timePassed = (now - bucket.lastLeak) / 1000
    const waterLeaked = timePassed * this.leakRate
    bucket.water = Math.max(0, bucket.water - waterLeaked)
    bucket.lastLeak = now

    if (bucket.water < this.capacity) {
      bucket.water += 1
      return true
    }

    return false
  }
}

function rateLimitMiddleware(limiter) {

  }

async function redisRateLimiter(key, limit, window) {

  pipe.zremrangebyscore(key, 0, now - window)
  pipe.zcard(key)

  const count = results[2][1]

}
```

```javascript
// app.js
const cluster = require('cluster')
const os = require('os')

if (cluster.isMaster) {
  const numCPUs = os.cpus().length

  // 创建工作进程
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork()
  }

  // 监控进程状态
  cluster.on('exit', (worker, code, signal) => {
    console.log(`进程 ${worker.process.pid} 退出`)
    // 自动重启
    cluster.fork()
  })

  // 优雅重启
  process.on('SIGUSR2', () => {
    const workers = Object.values(cluster.workers)
    const restartWorker = (index) => {
      const worker = workers[index]
      if (!worker) return

      worker.on('exit', () => {
        if (!worker.exitedAfterDisconnect) return
        console.log(`进程 ${worker.process.pid} 已重启`)
        cluster.fork()
        restartWorker(index + 1)
      })

      worker.disconnect()
    }

    restartWorker(0)
  })
} else {
  require('./server')
}
```
