## 布局与盒模型：

### absolute 与 fixed 的区别？

两者都会脱离文档流，核心区别在于**定位参照物不同**：

| 特性     | absolute                                        | fixed                    |
| -------- | ----------------------------------------------- | ------------------------ |
| 参照物   | 最近的**非 static 定位祖先**（没有则相对 html） | 始终相对**浏览器视口**   |
| 滚动行为 | 随页面滚动而移动                                | 固定在视口，不随页面滚动 |
| 典型场景 | 下拉菜单、Tooltip、弹层                         | 固定导航栏、返回顶部按钮 |

**背诵版：**

- absolute 相对最近的**非 static 祖先**定位，会跟着页面滚动走
- fixed 相对**浏览器视口**定位，页面怎么滚它都不动
- 有个坑：如果 fixed 的祖先设了 `transform`、`filter` 或 `will-change`，fixed 会**退化成 absolute 的效果**，不再相对视口

### 标准盒模型与怪异盒模型（box-sizing）的区别。

区别在于 `width/height` **包含的范围不同**：

| 模型                             | width/height 包含          | 实际占用宽度             |
| -------------------------------- | -------------------------- | ------------------------ |
| 标准盒模型 `content-box`（默认） | 仅 content                 | width + padding + border |
| 怪异盒模型 `border-box`          | content + padding + border | 就是设置的 width         |

**背诵版：**

- `content-box` 是默认值，width 只包含内容区域，padding 和 border **往外加**，实际占用空间 = width + padding + border
- `border-box` 的 width **包含了** content + padding + border，设多宽就占多宽，padding 和 border **往内缩**
- 实际开发推荐全局设置 `border-box`，尺寸更直观可控

**记忆口诀：content-box 往外加，border-box 往内缩。**

```css
/* 推荐全局设置 border-box */
*,
*::before,
*::after {
  box-sizing: border-box;
}
```

### Flex 布局详解：flex: 1 或 flex: 0 1 200px 代表什么？

`flex` 是 `flex-grow`、`flex-shrink`、`flex-basis` 三个属性的简写：

| 属性          | 含义                     | 默认值 |
| ------------- | ------------------------ | ------ |
| `flex-grow`   | 剩余空间的**放大比例**   | 0      |
| `flex-shrink` | 空间不足时的**缩小比例** | 1      |
| `flex-basis`  | 分配空间前的**初始大小** | auto   |

**背诵版：**

- flex 是三个值的简写：**放大比例、缩小比例、初始大小**
- `flex: 1` 等于 `1 1 0%`——会自动**填满剩余空间**，初始大小为 0，按比例瓜分所有空间
- `flex: 0 1 200px`——**不放大、可缩小**，初始宽 200px，空间不够时会被压缩
- `flex: none` 等于 `0 0 auto`——完全**刚性**，不放大也不缩小，内容多宽就多宽
- `flex: auto` 等于 `1 1 auto`——能放大能缩小，初始按内容大小

**经典布局：左侧固定，右侧自适应：**

```css
.left {
  flex: 0 0 200px;
} /* 固定 200px */
.right {
  flex: 1;
} /* 填满剩余 */
```

### 如何实现水平垂直居中（多种方案）？

**方案一：Flex（最推荐，面试先说这个）**

```css
.parent {
  display: flex;
  justify-content: center;
  align-items: center;
}
```

**方案二：Grid（最简洁）**

```css
.parent {
  display: grid;
  place-items: center;
}
```

**方案三：absolute + transform（不需要知道子元素尺寸）**

```css
.child {
  position: absolute;
  top: 50%;
  left: 50%;
  transform: translate(-50%, -50%);
}
```

**方案四：absolute + margin auto（子元素需有宽高）**

```css
.child {
  position: absolute;
  inset: 0;
  margin: auto;
  width: 200px;
  height: 200px;
}
```

**背诵版：**

- **Flex**：父元素 `display: flex` + `justify-content: center` + `align-items: center`，最常用
- **Grid**：父元素 `display: grid` + `place-items: center`，一行搞定
- **absolute + transform**：子元素绝对定位 `top/left: 50%`，再 `translate(-50%, -50%)` 回退自身的一半，不需要知道尺寸
- **absolute + margin auto**：子元素绝对定位 `inset: 0` + `margin: auto`，但必须设宽高

### 什么是 BFC（块级格式化上下文）？它能解决什么问题？

**背诵版：**

- BFC 就是一个**独立的渲染区域**，里面的元素怎么布局都不会影响外面
- 触发方式有五种：`overflow` 不为 visible（常用 `hidden`）、`display: flow-root`（最推荐）、`display: flex/inline-block`、`float` 不为 none、`position: absolute/fixed`
- BFC 能解决三个经典问题：
  1. **外边距折叠**——相邻元素的上下 margin 会合并，把它们放进不同 BFC 就不会合并了
  2. **浮动高度塌陷**——子元素浮动后父元素高度变 0，给父元素触发 BFC 就能包住浮动子元素
  3. **文字环绕浮动**——给文字容器触发 BFC，可以阻止环绕，实现两栏不重叠

```css
/* 清除浮动最佳实践 */
.clearfix {
  display: flow-root; /* 触发 BFC，包含浮动子元素 */
}
```

## 样式隔离：

### 微前端中如何做样式隔离？

| 方案                  | 原理                                        | 优缺点                                     |
| --------------------- | ------------------------------------------- | ------------------------------------------ |
| **CSS Modules**       | 构建时给类名加 hash 后缀，避免冲突          | 仅隔离类名，全局样式仍会污染               |
| **Shadow DOM**        | 浏览器原生隔离，样式完全封闭                | 隔离最彻底，但弹窗类组件挂载到 body 时失效 |
| **CSS 前缀/命名空间** | 子应用所有样式加统一前缀（如 `.app1 .btn`） | 简单可靠，但需要构建工具配合               |
| **动态样式加卸载**    | 子应用激活时插入样式，卸载时移除            | qiankun 默认方案，切换时有短暂闪烁         |
| **CSS-in-JS**         | 样式写在 JS 中，运行时生成唯一类名          | 天然隔离，但有运行时开销                   |

**背诵版：**

- **CSS Modules**：构建时给类名加 hash，自动隔离。缺点是只隔离了类名，全局样式（如 `body`、`h1`）还是会污染
- **Shadow DOM**：浏览器原生方案，隔离最彻底。缺点是弹窗 / dialog 挂到 body 上时样式就丢了
- **CSS 前缀**：给子应用所有样式自动加前缀（如 `.app1 .btn`），简单可靠，但需要构建工具（如 PostCSS）配合
- **动态加卸载**：子应用激活时插入 `<style>`，卸载时移除。qiankun 默认方案，缺点是切换时可能闪一下
- **CSS-in-JS**：样式写在 JS 里，运行时生成唯一类名，天然隔离。缺点是有运行时性能开销
- qiankun 提供两个选项：`strictStyleIsolation`（Shadow DOM）和 `experimentalStyleIsolation`（加前缀）

### CSS Modules 与 Tailwind CSS 的对比。

| 维度     | CSS Modules                | Tailwind CSS                          |
| -------- | -------------------------- | ------------------------------------- |
| 核心理念 | 组件级样式隔离，类名哈希化 | 原子化工具类，直接在 HTML 中写样式    |
| 样式隔离 | 自动隔离，无全局污染       | 全局工具类，不存在污染问题            |
| 开发体验 | 需在 CSS 文件和组件间切换  | 不用切文件，在模板中直接写            |
| 可维护性 | 传统 CSS 写法，语义化类名  | 类名长但可通过 `@apply` 或组件抽象    |
| 产物体积 | 按使用打包                 | Tree-shaking 只保留用到的类，体积极小 |
| 学习成本 | 几乎无，就是写 CSS         | 需记忆工具类名，但 IDE 提示后很快     |
| 适用场景 | 中小项目、需要精细控制样式 | 快速开发、设计系统一致性强的项目      |

**背诵版：**

- CSS Modules 的思路是**组件级隔离**，构建时自动给类名加 hash，写法跟普通 CSS 一样，学习成本低。缺点是需要在 CSS 文件和组件之间来回切换
- Tailwind 的思路是**原子化工具类**，样式直接写在模板里不用切文件，基于设计令牌天然保证 UI 一致性。缺点是类名堆得很长，可读性差
- 两者产物体积都很小，都是按需打包。Tailwind 用 JIT 模式只生成用到的类
- CSS Modules 适合需要精细控制样式的场景，Tailwind 适合快速开发和设计系统一致性要求高的项目

## 移动端/H5 特性：

### 100vh 在移动端（如 iOS 底部工具栏）的兼容性问题。

**背诵版：**

- 问题：移动端浏览器的地址栏和底部工具栏会**动态显示/隐藏**，但 `100vh` 永远等于包含工具栏的**最大视口高度**。工具栏显示时，实际可见区域比 `100vh` 小，导致页面底部被遮挡
- **推荐方案：用 `100dvh`**（dynamic viewport height），这个值会随着工具栏的显示/隐藏**动态变化**
- 三种新单位要区分：`svh` 是最小视口高度（工具栏显示时），`lvh` 是最大视口高度（工具栏隐藏时），`dvh` 是动态跟着变的
- 兼容老浏览器可以用 JS：监听 `resize` 事件，把 `window.innerHeight` 写入 CSS 变量 `--vh`，然后用 `calc(var(--vh) * 100)` 代替 100vh

**方案一：使用 `dvh`（推荐）**

```css
.full-screen {
  height: 100dvh;
}
```

**方案二：JS 动态计算（兼容老浏览器）**

```js
function setVH() {
  document.documentElement.style.setProperty(
    '--vh',
    `${window.innerHeight * 0.01}px`,
  )
}
window.addEventListener('resize', setVH)
setVH()
```

```css
.full-screen {
  height: calc(var(--vh, 1vh) * 100);
}
```

### 响应式设计的常见适配方案。

| 方案                | 原理                                       | 适用场景                         |
| ------------------- | ------------------------------------------ | -------------------------------- |
| **媒体查询**        | `@media` 根据屏幕宽度应用不同样式          | PC 和移动端差异大的页面          |
| **rem + 根字号**    | 根据屏幕宽度动态设置 `html` 的 `font-size` | 移动端等比缩放（淘宝方案）       |
| **vw/vh**           | 直接用视口百分比单位                       | 移动端，简单直接                 |
| **Flex + 百分比**   | 弹性布局自适应                             | 通用，大部分场景够用             |
| **Container Query** | 根据容器宽度而非视口宽度响应               | 组件化开发，组件需在不同容器复用 |

**背诵版：**

- **媒体查询**：用 `@media` 按屏幕宽度写不同样式，适合 PC 和移动端差异大的页面
- **rem + 根字号**：根据屏幕宽度动态设置 `html` 的 `font-size`，所有用 rem 的元素等比缩放，这就是淘宝方案
- **vw/vh**：直接用视口百分比单位，简单直接，移动端常用
- **Flex + 百分比**：弹性布局自适应，大部分场景够用
- **Container Query**：根据**容器宽度**而不是视口宽度来响应，适合组件需要在不同宽度容器中复用的场景
- 实际项目：PC 端用媒体查询 + Flex/Grid，移动端用 vw 或 rem + postcss-pxtorem，响应式项目用 Tailwind 的 `md:`、`lg:` 断点前缀

## Tailwind CSS (高频)：

### Tailwind 的优缺点？如何解决类名过长、产物体积过大的问题？

**背诵版：**

- 优点四个：**开发效率高**（不用切文件写 CSS、不用想类名）、**天然一致性**（基于设计令牌）、**产物极小**（JIT 按需生成，只包含用到的类）、**无样式冲突**（原子类不会互相覆盖）
- 缺点三个：**模板可读性差**（类名堆叠很长）、**学习曲线**（需要记工具类名）、**复杂样式受限**（复杂选择器和动画需要自定义）
- 类名过长怎么解：**首选抽组件**（把类名封装在组件内部），其次用 `@apply` 提取公共样式（但不要滥用，否则失去原子化的意义）
- 产物体积不是问题：v3+ 默认 JIT 模式，按需生成，产物通常只有几 KB

```vue
<!-- 方案一：抽组件（最推荐） -->
<Button variant="primary">确定</Button>

<!-- 方案二：@apply 提取（适合全局复用的基础样式） -->
<style>
.btn-primary {
  @apply px-4 py-2 bg-blue-500 text-white rounded hover:bg-blue-600;
}
</style>
```

---

### Tailwind 的可维护性如何保障？

**背诵版（五个要点）：**

1. **组件化抽象**：重复的样式组合封装成组件，不要到处复制类名
2. **设计令牌统一**：在 `tailwind.config.js` 里定义项目的 color、spacing、font，团队用统一的 token
3. **Prettier 插件**：用 `prettier-plugin-tailwindcss` 自动排序类名，保持顺序一致
4. **`@apply` 适度用**：只用于全局基础样式（如 `.btn`、`.card`），滥用就失去原子化优势了
5. **前缀顺序规范**：团队约定 `hover:`、`md:` 等修饰符的书写顺序

## 动画与渲染：

### transform 移动与 margin-left 移动在性能上的区别？

| 属性          | 触发阶段                   | 性能                               |
| ------------- | -------------------------- | ---------------------------------- |
| `margin-left` | Layout → Paint → Composite | 触发**重排**，影响周围元素，性能差 |
| `transform`   | 仅 Composite               | 不触发重排重绘，GPU 加速，性能极好 |

**背诵版：**

- `margin-left` 改变了元素在文档流中的位置，浏览器需要**重新计算布局**（重排），然后重绘，再合成，整条渲染流水线都要走一遍，还会影响周围元素
- `transform` 不影响文档流，元素会被提升为独立的**合成层**，变换直接在 **GPU** 上完成，只走 Composite 这一步，性能极好
- 所以做动画**一定用 transform**，可以配合 `will-change: transform` 提前告知浏览器创建合成层

**记忆口诀：动画用 transform，布局用 margin。**

### requestAnimationFrame 的作用与应用场景。

| 特性     | requestAnimationFrame          | setTimeout                     |
| -------- | ------------------------------ | ------------------------------ |
| 执行时机 | 浏览器重绘前，与刷新率同步     | 到时间就执行，可能与刷新不同步 |
| 精度     | 自动适配屏幕刷新率             | 实际间隔不精确，可能掉帧       |
| 后台行为 | 页面不可见时自动暂停，节省性能 | 持续执行，浪费资源             |

**背诵版：**

- rAF 的作用是在浏览器**下一次重绘之前**执行回调，确保动画和屏幕刷新率同步（60fps 就是约 16.6ms 一帧）
- 跟 setTimeout 比有三个优势：**时机精准**（跟刷新率同步不掉帧）、**自动适配**（120Hz 屏幕也能适配）、**后台省电**（页面不可见时自动暂停）
- 应用场景四个：JS 动画、滚动监听优化（在 rAF 中批量处理）、Canvas 逐帧绘制、DOM 批量更新

```js
function animate() {
  element.style.transform = `translateX(${x++}px)`
  if (x < 300) {
    requestAnimationFrame(animate) // 递归调用，形成动画循环
  }
}
requestAnimationFrame(animate)
```

## 标签与属性：

### script 标签中 async 和 defer 的区别？

| 方式             | HTML 解析  | 脚本下载     | 脚本执行时机                             | 执行顺序   |
| ---------------- | ---------- | ------------ | ---------------------------------------- | ---------- |
| `<script>`       | **阻塞**   | 阻塞期间下载 | 下载完立即执行，阻塞解析                 | 按书写顺序 |
| `<script async>` | **不阻塞** | 并行下载     | 下载完**立即执行**，中断解析             | 不保证顺序 |
| `<script defer>` | **不阻塞** | 并行下载     | HTML 解析完后，`DOMContentLoaded` 前执行 | 按书写顺序 |

**背诵版：**

- 普通 `<script>`：遇到就**停下来**，下载完立刻执行，执行完才继续解析 HTML，全程阻塞
- `async`：下载**不阻塞**解析，但下载完会**立刻执行并中断解析**。多个 async 脚本谁先下完谁先执行，**不保证顺序**。适合独立脚本（统计、广告）
- `defer`：下载**不阻塞**解析，等 HTML **全部解析完**后才执行，在 `DOMContentLoaded` 之前。多个 defer 脚本**按书写顺序**执行。大多数场景首选
- 选择策略：业务代码用 defer，独立脚本（不依赖 DOM 和其他脚本）用 async

**记忆口诀：defer 延迟有序，async 异步无序。**

---

### 解析 HTML 时遇到 script 会发生什么？

**背诵版：**

- 浏览器解析 HTML 是**自上而下**的，遇到 `<script>` 会**暂停解析**
- 四个步骤：暂停 DOM 构建 → 下载脚本（外部脚本）→ 执行脚本 → 继续解析
- 执行时只能访问到**前面已解析的 DOM**，后面的还没构建所以访问不到
- 传统做法是把 script 放在 `</body>` 前面，确保 DOM 构建完再执行
- **现代推荐做法**：放在 `<head>` 里加 `defer`，效果一样，而且浏览器能更早发现并开始下载脚本
- 还有个冷知识：CSS 也会间接阻塞脚本——如果 `<script>` 前面有没加载完的 CSS，浏览器会等 CSS 加载完再执行脚本，因为脚本可能要读样式信息

```html
<head>
  <!-- 推荐：defer 不阻塞解析，且保证 DOM 就绪后执行 -->
  <script defer src="app.js"></script>
</head>
```

## 多媒体与渲染：

### 前端实现 2D/3D 渲染的技术有哪些（Canvas, WebGL, WebGPU）？

| 技术          | 维度  | 原理                          | 适用场景                      |
| ------------- | ----- | ----------------------------- | ----------------------------- |
| **Canvas 2D** | 2D    | JS 调用 2D 绑图 API，逐帧绘制 | 图表、简单游戏、图片处理      |
| **SVG**       | 2D    | 基于 XML 的矢量图，DOM 节点   | 图标、数据可视化、可交互图形  |
| **WebGL**     | 2D/3D | 基于 OpenGL ES，GPU 渲染      | 3D 场景、复杂游戏、地图       |
| **WebGPU**    | 2D/3D | 新一代 GPU API，性能更强      | 高性能渲染、GPU 计算、AI 推理 |

**背诵版：**

- **Canvas 2D**：JS 调用 2D API 逐帧绘制像素，适合图表、简单游戏、图片处理。**不保留绘制对象**，画完就是像素了，适合元素量大（>1000）的场景
- **SVG**：基于 XML 的矢量图，每个图形都是 **DOM 节点**，天然支持事件绑定和 CSS 动画，适合图标、数据可视化等**少量元素且需要交互**的场景
- **WebGL**：基于 OpenGL ES，利用 **GPU 渲染**，能做 3D 场景，性能强大。常用 Three.js 封装
- **WebGPU**：新一代 GPU API，比 WebGL 更底层更强，不仅能渲染还能做 **GPU 通用计算**（如 AI 推理）
- Canvas vs SVG 怎么选：元素多、需要频繁重绘用 Canvas；元素少、需要交互和动画用 SVG
- 常用库：ECharts（Canvas）、D3.js（SVG）、Three.js（WebGL）、Babylon.js（WebGL/WebGPU）

## H5 原生方法：

### requestIdleCallback 的原理？如何判断浏览器是否空闲？

**背诵版：**

- 浏览器每帧（16.6ms）要做一堆事：处理输入 → 执行 JS → rAF → Layout → Paint。如果这些做完了还**有剩余时间**，就执行 `requestIdleCallback` 的回调
- 回调会收到一个 `deadline` 参数，调用 `deadline.timeRemaining()` 可以知道**当前帧还剩多少空闲时间**
- 可以传 `{ timeout: 2000 }` 设超时时间，超过后即使没有空闲时间也会**强制执行**，防止任务被饿死
- 适用场景：**非紧急任务**（埋点上报、预加载）、**大计算分片**（拆成小块在空闲时逐步执行）
- React Fiber 的时间切片就是受这个 API 启发的
- 注意：Safari 不支持，需要用 `setTimeout` 做 polyfill

```js
requestIdleCallback(
  (deadline) => {
    while (deadline.timeRemaining() > 0 && tasks.length > 0) {
      doTask(tasks.shift())
    }
    if (tasks.length > 0) {
      requestIdleCallback(callback)
    }
  },
  { timeout: 2000 },
)
```

### 文件切片上传的实现原理（原生 API）。

**背诵版：**

- 核心思路：把大文件**切成小块**（chunk），**并发上传**，全部上传完后通知服务端**合并**
- 三个步骤：
  1. **切片**：用 `File.prototype.slice()` 按固定大小（如 5MB）切割文件，得到多个 Blob
  2. **算 hash**：用 `crypto.subtle.digest('SHA-256', buffer)` 计算整个文件的哈希值，用于标识文件
  3. **并发上传**：每个切片用 FormData 包装，配合并发控制器（如最多 3 个同时传），全部完成后请求合并接口
- 三个关键特性：
  - **断点续传**：上传前先问服务端哪些切片已经有了，跳过已上传的
  - **秒传**：先把文件 hash 发给服务端，如果已经有这个文件就直接返回成功，根本不用传
  - **进度显示**：用 `XMLHttpRequest` 的 `upload.onprogress` 监听每个切片的上传进度

```js
// 切片
function createChunks(file, chunkSize = 5 * 1024 * 1024) {
  const chunks = []
  let cur = 0
  while (cur < file.size) {
    chunks.push(file.slice(cur, cur + chunkSize))
    cur += chunkSize
  }
  return chunks
}

// 算 hash
async function calcHash(file) {
  const buffer = await file.arrayBuffer()
  const hashBuffer = await crypto.subtle.digest('SHA-256', buffer)
  return Array.from(new Uint8Array(hashBuffer))
    .map((b) => b.toString(16).padStart(2, '0'))
    .join('')
}

// 并发上传 + 合并
async function uploadChunks(chunks, fileHash) {
  const requests = chunks.map((chunk, index) => {
    const formData = new FormData()
    formData.append('chunk', chunk)
    formData.append('hash', fileHash)
    formData.append('index', index)
    return () => fetch('/upload', { method: 'POST', body: formData })
  })

  await concurrencyControl(requests, 3) // 最多 3 个同时上传

  await fetch('/merge', {
    method: 'POST',
    body: JSON.stringify({ hash: fileHash, total: chunks.length }),
  })
}
```
