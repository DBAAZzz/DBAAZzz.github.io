# Vue 3 高频面试题 Top 10

## 1. Vue 3 响应式原理是什么？与 Vue 2 有什么区别？

Vue 3 用 `Proxy` 替代了 Vue 2 的 `Object.defineProperty` 来实现响应式。

Vue 2 的 `Object.defineProperty` 只能劫持对象已有属性的 `get/set`，无法检测属性新增和删除，所以需要 `Vue.set`、`Vue.delete` 这些额外 API。

对数组也只能通过重写原型方法（`push`、`splice` 等）来拦截，无法通过索引直接触发更新。而且初始化时需要递归遍历整个对象，性能开销比较大。

Vue 3 的 `Proxy` 可以拦截整个对象的 13 种操作，包括 `get`、`set`、`has`、`deleteProperty` 等，天然支持属性新增删除和数组索引修改，不再需要任何 hack API。

同时采用**懒代理**策略，只有访问到嵌套对象时才递归创建 Proxy，初始化性能更好。

另外响应式模块 `@vue/reactivity` 是独立可用的，不依赖 Vue 组件系统。

核心流程就是：`reactive()` 创建 Proxy，`get` 中通过 `track()` 收集依赖，`set` 中通过 `trigger()` 触发更新。

## 2. `ref` 和 `reactive` 的区别？为什么 `reactive` 解构会丢失响应式？

`ref` 可以包装任意类型，包括基本类型和对象，访问时需要 `.value`。

对于基本类型它内部用 getter/setter 包装，对于对象类型其实内部也是调用 `reactive`。`ref` 可以直接通过 `ref.value = newObj` 替换整个值。

`reactive` 只能用于对象、数组、Map、Set 等引用类型，直接访问属性不需要 `.value`，底层直接使用 `Proxy`。但不能直接替换整个对象，否则会丢失响应式。

**解构丢失响应式的原因：** `reactive` 返回的是一个 Proxy 对象，响应式依赖于对这个 Proxy 的属性访问拦截。解构操作 `const { name } = state` 本质上就是 `const name = state.name`，把属性值赋给了一个普通变量，这个变量已经脱离了 Proxy，后续修改它不会经过 Proxy 的 `set` 拦截，所以失去响应式。

解决方案是使用 `toRefs(state)` 将每个属性转为 `ref`，解构后依然保持响应式。

## 3. `computed` 和 `watch` / `watchEffect` 的区别和使用场景？

`computed` 是声明式的派生状态，有**缓存机制**，依赖不变就不重新计算。必须有返回值，适合"由 A 推导出 B"的场景，在模板中使用时只读取不执行副作用。

`watch` 是侦听特定数据源的变化，支持获取新旧值。默认是懒执行的，首次不触发，可以设 `immediate: true`。适合在数据变化时执行副作用，比如发 API 请求、路由跳转、同步本地存储等。

`watchEffect` 会自动收集依赖，立即执行一次回调，不需要显式指定监听源，回调中访问的响应式数据都会被追踪。适合"依赖多个数据，变了就重新执行"的场景，但无法获取新旧值。

```ts
// computed：有缓存的派生值
const fullName = computed(() => `${first.value} ${last.value}`)

// watch：精确监听 + 获取新旧值
watch(userId, async (newId, oldId) => {
  const data = await fetchUser(newId)
})

// watchEffect：自动追踪依赖
watchEffect(() => {
  console.log(`user: ${user.name}, age: ${user.age}`)
})
```

## 4. Composition API 相比 Options API 的优势？

首先是**逻辑复用**。Options API 复用逻辑主要靠 Mixins，但 Mixins 存在命名冲突、来源不清晰、隐式依赖等问题。Composition API 通过 composables 实现逻辑复用，数据来源明确，支持参数传递和类型推导。

其次是**代码组织**。Options API 按选项类型分割代码，`data`、`methods`、`computed` 分散在不同位置，一个功能的逻辑被打散了。Composition API 允许按功能关注点组织代码，同一功能的响应式数据、计算属性、方法可以集中放在一起。

第三是**TypeScript 支持**。Composition API 对 TS 的支持远优于 Options API，`ref<string>()`、`computed` 的返回类型都可以自动推断，不需要像 Options API 那样依赖 `ThisType` 等复杂类型体操。

最后是 **Tree-shaking**。Composition API 的函数式写法使得未使用的 API 比如 `watch`、`computed` 可以被构建工具 tree-shake 掉，减小打包体积。

## 5. 自定义 Hooks（Composables）的设计原则？

命名以 `use` 开头，如 `useCounter`、`useFetch`，表明这是一个组合式函数。

入参设计上要支持 `ref` 和普通值（用 `unref()` 或 `toValue()` 兼容）。返回值推荐用**对象**而非数组，跟 React Hooks 不同，对象方便按需解构且语义清晰。返回的状态推荐使用 `ref` 而非 `reactive`，让调用方解构后仍保持响应式。

副作用管理方面，内部用到的定时器、事件监听等必须在 `onUnmounted` 中清理。如果依赖 DOM，逻辑应放在 `onMounted` 中。

跟 React Hooks 的核心区别在于：Vue composables 只在 `setup()` 中**调用一次**，建立响应式关联后持续生效；而 React Hooks 每次渲染都重新执行。所以 Vue 中不存在"闭包陷旧值"问题，也不需要 `useCallback`、`useMemo` 这类优化手段。

```ts
export function useFetch<T>(url: MaybeRef<string>) {
  const data = ref<T | null>(null)
  const loading = ref(false)
  const error = ref<Error | null>(null)

  const fetchData = async () => {
    loading.value = true
    try {
      const res = await fetch(unref(url))
      data.value = await res.json()
    } catch (e) {
      error.value = e as Error
    } finally {
      loading.value = false
    }
  }

  onMounted(fetchData)
  watch(() => unref(url), fetchData)

  return { data, loading, error, refetch: fetchData }
}
```

## 6. Vue 3 的 Diff 算法相比 Vue 2 做了哪些优化？

Vue 3 在编译阶段引入了多项优化，大幅减少运行时 diff 的开销。

**PatchFlag（补丁标记）：** 编译器在生成 VNode 时会标记动态内容的类型，比如 `TEXT = 1`、`CLASS = 2`、`PROPS = 4`。运行时 diff 只需检查对应标记位，跳过静态内容的比对。

**静态提升（Static Hoisting）：** 纯静态节点在编译时被提升到渲染函数外部，只创建一次，后续渲染直接复用引用，避免重复创建和 diff。

**Block Tree：** Vue 3 引入 Block 概念，将模板中的动态节点收集到一个扁平数组 `dynamicChildren` 中。更新时只需遍历这个数组，而不是递归整棵虚拟 DOM 树，将 diff 复杂度从"与模板大小相关"降低为"与动态内容数量相关"。

**缓存事件处理函数：** 内联事件处理器（如 `@click="count++"`）会被缓存，避免每次渲染创建新函数导致子组件不必要的更新。

Vue 2 的 diff 采用双端对比算法（头头、尾尾、头尾、尾头四种比较），虽然有效但需要逐层递归整棵树。Vue 3 在此基础上引入**最长递增子序列**算法优化节点移动，进一步减少 DOM 操作次数。

## 7. 什么是 Vapor Mode？它解决什么问题？

Vapor Mode 是 Vue 团队正在开发的一种**无虚拟 DOM** 的编译策略，灵感来源于 Svelte 和 Solid.js。

核心思路是编译器直接将模板编译为命令式的 DOM 操作代码，而不是生成虚拟 DOM 节点再通过 diff 更新。响应式数据变化时，精确地只更新对应的 DOM 节点，跳过虚拟 DOM 的创建和比对。

它解决的问题主要是消除虚拟 DOM 的运行时开销（创建 VNode、diff、patch），减小打包体积（不需要虚拟 DOM 运行时），在性能敏感场景下提供更好的更新性能。

跟现有模式的关系是：Vapor Mode 是可选的，不会替代现有的虚拟 DOM 模式。计划支持在同一应用中混用两种模式，可以在组件级别选择使用哪种。现有的 Composition API、SFC 语法等不受影响。

> 面试中需要说明 Vapor Mode 目前仍在实验阶段，尚未正式发布。

## 8. Vue 3 生命周期钩子有哪些变化？`setup` 中如何使用？

最大的变化是 `setup()` 本身就在 `beforeCreate` 和 `created` 之间执行，所以这两个钩子不再需要，对应的逻辑直接写在 `setup` 顶层就行。

其他钩子在 Composition API 中都加了 `on` 前缀：`beforeMount` 变成 `onBeforeMount`，`mounted` 变成 `onMounted`，`beforeUpdate` 变成 `onBeforeUpdate`，`updated` 变成 `onUpdated`。Vue 2 中的 `beforeDestroy` 和 `destroyed` 在 Vue 3 中重命名为 `onBeforeUnmount` 和 `onUnmounted`，语义更准确。

Vue 3 还新增了两个调试用钩子：`onRenderTracked` 用于追踪依赖收集，`onRenderTriggered` 用于追踪触发更新的依赖，开发阶段排查问题很有用。

组合式钩子的一个重要特性是支持**多次调用**，不同的 composable 可以各自注册自己的生命周期逻辑而不会冲突。在 `<script setup>` 语法糖中直接 import 使用即可。

```ts
<script setup>
import { onMounted, onUnmounted } from 'vue'

// 相当于 created 阶段
console.log('组件初始化逻辑')

onMounted(() => {
  window.addEventListener('resize', handleResize)
})

onUnmounted(() => {
  window.removeEventListener('resize', handleResize)
})
</script>
```

## 9. `nextTick` 的原理和使用场景？

Vue 的 DOM 更新是**异步批量**执行的。当响应式数据变化时，Vue 不会立即更新 DOM，而是将更新操作推入一个队列，在当前同步任务执行完毕后，利用微任务（`Promise.then`）批量刷新队列、执行 DOM 更新。

`nextTick()` 返回一个 Promise，在 DOM 更新完成后 resolve，确保回调中能拿到更新后的 DOM。

使用场景主要有三个：一是修改数据后需要**立即操作更新后的 DOM**，比如获取元素尺寸、滚动位置；二是动态添加元素后需要立即获取其 DOM 引用；三是在 `onMounted` 中确保子组件都已渲染完成。

```ts
const count = ref(0)

async function increment() {
  count.value++
  // 此时 DOM 尚未更新
  console.log(document.getElementById('count')?.textContent) // 旧值

  await nextTick()
  // DOM 已更新
  console.log(document.getElementById('count')?.textContent) // 新值
}
```

简化版原理可以这样理解：

```ts
const queue: Function[] = []
let pending = false

function queueJob(job: Function) {
  queue.push(job)
  if (!pending) {
    pending = true
    Promise.resolve().then(flushJobs)
  }
}
```

## 10. Pinia 相比 Vuex 的优势？

首先是**去除了 Mutations**。Vuex 强制区分 Mutations（同步）和 Actions（异步），增加了很多样板代码。Pinia 中直接在 Actions 里修改 state，同步异步统一处理，代码更简洁。

其次是**更好的 TypeScript 支持**。Vuex 对 TS 的支持需要大量类型声明和 `useStore` 的类型体操。Pinia 在定义 Store 时自动推导 state、getters、actions 的类型，调用处也有完整的类型提示。

第三是**模块化设计**。Vuex 使用 `modules` + `namespaced` 的嵌套结构，访问跨模块状态比较复杂。Pinia 每个 Store 都是独立的，按需 import，天然扁平化，Store 之间可以直接互相引用。

体积上 Pinia 约 1KB gzip，Vuex 约 10KB+，差距也很明显。

Pinia 支持两种定义风格：Options Store 类似 Vuex 的写法，适合从 Vuex 迁移；Setup Store 是组合式风格，更灵活，跟 Composition API 一脉相承。

```ts
// Options Store（类似 Vuex 风格）
export const useCounterStore = defineStore('counter', {
  state: () => ({ count: 0 }),
  getters: { double: (state) => state.count * 2 },
  actions: {
    increment() {
      this.count++
    },
  },
})

// Setup Store（组合式风格，更灵活）
export const useCounterStore = defineStore('counter', () => {
  const count = ref(0)
  const double = computed(() => count.value * 2)
  function increment() {
    count.value++
  }
  return { count, double, increment }
})
```

其他优势还包括：支持 SSR、开箱即用的 DevTools 集成、插件机制（比如持久化插件）、支持 `$patch` 批量更新和 `$reset` 重置状态。
