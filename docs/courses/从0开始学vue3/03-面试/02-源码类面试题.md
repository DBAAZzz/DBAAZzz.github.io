# Vue 3 源码系列高频面试题 Top 11

## 1. Vue 3 的响应式原理是怎样的？为什么要用 Proxy 替代 Object.defineProperty？

- 考察点：响应式底层的核心API替换原因及基本架构。
- 核心解析：
  - Vue3 使用 Proxy 拦截对象操作，配合 Reflect 保证上下文 this 的正确性。
  - 替代原因：defineProperty 只能拦截对象属性的 get/set，无法拦截属性的添加/删除、无法监听数组索引的修改和 length 变化；并且需要深度遍历整个对象，性能差。Proxy 是代理整个对象，支持多达13种拦截操作，且是惰性代理（只有访问到嵌套对象时才将其转化为响应式）。

**结论：** Vue 3 用 `Proxy` 代理整个对象来实现响应式，替代了 Vue 2 的 `Object.defineProperty` 逐属性劫持方案，从根本上解决了无法拦截属性新增/删除、数组索引修改等问题。

**核心逻辑步骤：** 调用 `reactive(obj)` → 内部执行 `createReactiveObject()` → 创建 `new Proxy(target, mutableHandlers)` → 在 `mutableHandlers` 的 `get` trap 中调用 `track()` 收集当前正在运行的 `activeEffect` 作为依赖 → 在 `set` trap 中调用 `trigger()` 通知所有收集到的 effect 重新执行。嵌套对象采用惰性代理，只有在 `get` 中访问到子对象时才递归调用 `reactive()`，而非初始化时深度遍历。

**关键 API：** `Proxy` 提供 13 种拦截操作（`get`、`set`、`has`、`deleteProperty`、`ownKeys` 等）。配合 `Reflect` 调用保证 `this` 上下文正确性。核心函数为 `track(target, key)` 和 `trigger(target, key)`。

**设计优缺点：** 优点是天然支持属性增删、数组索引、Map/Set 等操作，不需要 `Vue.set` 这类 hack API；惰性代理策略让初始化性能大幅提升；`@vue/reactivity` 独立可用，不依赖组件系统。缺点是 `Proxy` 不支持 IE11，这也是 Vue 3 放弃 IE 的原因之一；对基本类型无法代理，需要 `ref` 包装。

## 2. 详细说说依赖收集和派发更新的数据结构？（3.3 → 3.5 的演进）

**结论：** 依赖收集经历了从"三层 Map + Set"到"版本计数 + 双向链表"的演进，3.5 版本在内存占用和计算效率上有质的提升。

**核心逻辑步骤（3.5）：** 响应式数据被读取时触发 `track()`，在当前 dep 和当前 activeEffect 之间建立一个 `Link` 节点，插入双向链表。数据变更时触发 `trigger()`，dep 的 `version` 自增，然后沿 subscriber 链表通知所有订阅者。订阅者（effect/computed）在执行前遍历自己的 deps 链表，通过 `link.dep.version !== link.version` 判断哪些依赖真正变化了。

**数据结构：**

- 3.3 及之前：`targetMap: WeakMap<target, Map<key, Set<ReactiveEffect>>>`，三层嵌套。用 `WeakMap` 是为了让 target 被销毁时键值对自动回收，防止内存泄漏。
- 3.5：dep 和 subscriber 之间通过 `Link` 节点组成双向链表，每个 Link 包含 `nextDep`、`prevDep`、`nextSub`、`prevSub` 四个指针。每个 dep 维护 `version` 数字用于版本比对。

**设计优缺点：** 3.5 方案优点是不需要为每个 dep 维护 Set 对象，内存占用显著降低；双向链表减少了递归调用的调用栈开销；版本计数避免了不必要的重计算。缺点是链表结构在代码层面比 Set 更复杂，调试和理解成本更高。

## 3. `ref` 和 `reactive` 在源码实现上有什么区别？

- 考察点：响应式 API 的底层差异。
- 核心解析：
  - reactive 内部直接使用 Proxy 包装对象。
  - ref 内部实例化了一个 RefImpl 类。由于基本数据类型无法被 Proxy 拦截，RefImpl 通过类的 getter 和 setter (get value() / set value()) 来拦截对 .value 的访问和修改。
    当 ref 传入一个对象时，底层还是会调用 reactive 进行转化。

**结论：** `reactive` 直接用 `Proxy` 代理对象，`ref` 用 `RefImpl` 类的 getter/setter 拦截 `.value` 访问。两者是互补关系，`ref` 解决了 `Proxy` 无法代理基本类型的问题。

**核心逻辑步骤：** `reactive(obj)` → `createReactiveObject()` → `new Proxy(target, handlers)`，返回代理对象。`ref(val)` → `new RefImpl(val)` → 如果 val 是对象则内部调用 `toReactive(val)` 走 Proxy 逻辑 → `get value()` 中调用 `trackRefValue()` 收集依赖 → `set value(newVal)` 中对比新旧值，不同则调用 `triggerRefValue()` 派发更新。

**关键 API：** `RefImpl` 类通过 ES6 class 的 `get value()` / `set value()` 访问器实现拦截。`toReactive()` 内部判断如果是对象就调用 `reactive()`。`trackRefValue()` 和 `triggerRefValue()` 是 ref 专用的依赖收集/派发函数。

**设计优缺点：** `ref` 的优点是支持任意类型、可以通过 `ref.value = newObj` 直接替换整个值且不丢失响应式。缺点是需要 `.value` 访问，增加了心智负担（模板中自动解包可以缓解）。`reactive` 的优点是使用时不需要 `.value`，更接近原生对象操作。缺点是不能直接替换整个对象、解构会丢失响应式（需要 `toRefs` 转换）。

## 4. `computed` 在源码中的实现原理？（3.3 → 3.5 的演进）

**结论：** computed 的核心是缓存 + 懒计算。实现机制从 3.3 的 dirty 布尔标记，演进到 3.4 的 dirtyLevel 多级状态，再到 3.5 的版本计数，每一代都在解决"不必要的重计算"问题。

**核心逻辑步骤（3.5 最新）：** `computed(getter)` → 内部创建 `ComputedRefImpl` 实例，关联一个 `ReactiveEffect` → 首次读取 `.value` 时执行 getter 计算结果并缓存 → 后续读取时遍历 deps 双向链表，逐个对比 `link.dep.version !== link.version` → 版本号全部一致则直接返回缓存值（命中缓存）→ 有任一版本号不一致则重新执行 getter，更新缓存和自身 version → 如果新值与旧值不同（`Object.is` 比较），才通知下游 subscriber。

**关键 API / 数据结构：**

- 3.3：`_dirty` 布尔标记 + `scheduler` 回调置脏。
- 3.4：`dirtyLevel`（`NotDirty`、`MaybeDirty`、`Dirty`）多级状态 + `Object.is` 值比较。
- 3.5：`dep.version` 版本号 + 双向链表遍历比对。computed 懒订阅——有第一个 subscriber 时才订阅上游，失去所有 subscriber 时自动取消。

**设计优缺点：** 3.3 方案简单但存在"值没变下游也重算"的问题（computed A 依赖 computed B，B 依赖变了但值没变，A 也被标脏）。3.4 用值比较解决了这个问题，但 dirtyLevel 逻辑复杂且有边界 bug（3.4.x 多个 patch 都在修 dirty 相关问题）。3.5 用版本计数彻底简化了判断逻辑，懒订阅机制还解决了 SSR 场景下 computed 不易被垃圾回收的问题。

## 5. Vue 3 的 Diff 算法相比 Vue 2 有什么改进？说说最长递增子序列的作用。

- 考察点：核心算法的演进。
- 核心解析：
  - Vue2 使用的是双端比较算法。
  - Vue3 使用了快速 Diff 算法。过程：先处理头尾相同节点（预处理）；如果有多余节点则挂载，缺少节点则卸载；如果是复杂的中间乱序部分，会基于新节点的 key 建立一个 keyToNewIndexMap。
  - 最长递增子序列（LIS）：在处理中间乱序节点时，Vue3 会计算出一个最长递增子序列。它的作用是找到不需要移动的 DOM 节点序列，从而使 DOM 的移动操作次数降到最低，极致优化性能。

**结论：** Vue 3 使用快速 Diff 算法替代 Vue 2 的双端比较算法，通过预处理 + 最长递增子序列（LIS）将 DOM 移动操作降到最少。

**核心逻辑步骤（`patchKeyedChildren`）：** 第一步，从头部开始比较相同节点（`isSameVNodeType` 判断），相同则 patch 并右移指针。第二步，从尾部开始比较相同节点，相同则 patch 并左移指针。第三步，旧节点遍历完、新节点有剩余，说明有新增，直接挂载。第四步，新节点遍历完、旧节点有剩余，说明有删除，直接卸载。第五步，处理中间乱序部分：基于新节点的 key 建立 `keyToNewIndexMap`，遍历旧节点找对应关系构建 `newIndexToOldIndexMap`，对该数组求 LIS。

**关键 API / 数据结构：** `keyToNewIndexMap: Map<key, index>` 存储新节点 key 到索引的映射。`newIndexToOldIndexMap: number[]` 记录新节点对应的旧索引。`getSequence()` 函数求 LIS，返回的是不需要移动的节点索引序列。`isSameVNodeType(n1, n2)` 通过 `type === type && key === key` 判断节点身份。

**设计优缺点：** 优点是头尾预处理能快速命中"仅尾部追加"或"仅头部删除"这类高频场景，效率极高。LIS 确保了中间乱序部分的 DOM 移动次数最少。缺点是 LIS 算法本身有一定复杂度（`O(n log n)`），在列表极短时相比 Vue 2 的双端比较没有明显优势，但在大列表场景下优势显著。

## 6. Vue 3 的编译期优化做了哪些事情？

- 考察点：Vue3 性能碾压 Vue2 的重要原因之一（不仅是 Proxy，还有编译优化）。
- 核心解析：
  - PatchFlag（靶向更新）：给动态节点打上不同的标志（如文本动态、class动态），Diff时只比较带有标志的属性，跳过静态节点。
  - Block Tree：将动态节点收集到根级 Block 中，更新时直接扁平化遍历动态节点数组（dynamicChildren），忽略DOM树层级的深度遍历。
  - 静态提升（hoistStatic）：将静态的节点和属性在 render 函数之外提升为常量，避免每次渲染都重新创建对象的开销。
  - 事件监听缓存（cacheHandlers）：缓存内联事件处理函数，避免每次渲染因为函数引用改变而导致组件重复渲染。

**结论：** Vue 3 的性能提升不仅来自 Proxy，编译期优化同样关键。核心是把运行时 diff 的工作量尽可能在编译期就确定下来，让运行时只关注真正动态的部分。

**核心逻辑步骤：** 模板编译分三个阶段。`parse` 将模板字符串解析为 AST。`transform` 对 AST 进行优化标记（打 PatchFlag、标记静态节点、收集 Block 的 dynamicChildren）。`generate` 将优化后的 AST 生成 render 函数代码字符串。四大优化都发生在 `transform` 和 `generate` 阶段。

**关键 API / 数据结构：**

- PatchFlag：数字枚举（`TEXT = 1`、`CLASS = 2`、`PROPS = 4` 等），运行时用位运算 `flag & PatchFlags.TEXT` 判断 diff 类型。
- Block Tree：`openBlock()` 创建当前 Block 的 `dynamicChildren` 数组，`createBlock()` 将动态子节点（跨层级）收集进去。`patchBlockChildren` 直接遍历扁平数组而非递归 DOM 树。
- 静态提升：静态 VNode 和属性对象提升为模块级常量，渲染时复用同一引用。
- 事件缓存：内联事件处理器编译为缓存变量，避免每次渲染创建新函数。

**设计优缺点：** 优点是把 diff 复杂度从"与模板大小相关"降低为"与动态节点数量相关"，对大模板中只有少量动态内容的场景提升巨大。缺点是 Block Tree 在 `v-if`/`v-for` 等结构化指令处需要建立新的 Block，增加了编译器实现的复杂度；且只有模板写法能享受这些优化，手写 render 函数（JSX）无法被编译器优化。

## 7. `nextTick` 原理是什么？Vue 3 的异步更新队列是怎么工作的？

- 考察点：事件循环（Event Loop）与 DOM 批量更新机制。
- 核心解析：
  - 响应式数据改变触发 trigger，接着触发 effect 的 scheduler。
  - Vue 不会立即更新 DOM，而是将更新任务（Job）推入一个异步队列（queueJob），并进行去重。
  - 然后通过 nextTick（底层是一个 Promise.resolve().then() 微任务）在当前同步代码执行完毕后，统一清空队列（flushJobs）去执行真正的 DOM 更新。

**结论：** Vue 3 的 DOM 更新是异步批量执行的。数据变更后不会立即更新 DOM，而是将更新任务推入队列，在微任务中统一刷新。`nextTick` 本质就是在这个刷新之后再执行回调。

**核心逻辑步骤：** 响应式数据变更 → `trigger()` 触发 effect 的 `scheduler` → `scheduler` 调用 `queueJob(componentUpdateFn)` 将组件更新任务推入队列 → `queueJob` 内部通过 Job 的 `id` 去重（同一组件多次数据变更只入队一次）→ 首次入队时调用 `queueFlush()`，通过 `Promise.resolve().then(flushJobs)` 注册微任务 → 当前同步代码执行完毕后微任务触发 → `flushJobs()` 按 `id` 从小到大排序队列（父组件先于子组件更新）→ 逐个执行 Job 完成 DOM 更新。

**关键 API / 数据结构：** `queue: SchedulerJob[]` 存储待执行的更新任务。`SchedulerJob` 带有 `id` 属性用于排序和去重。`queueFlush()` 内部用 `isFlushing` 布尔值防止重复注册微任务。`nextTick()` 返回一个 Promise，在 `flushJobs` 完成后 resolve。

```ts
// 简化版核心逻辑
const queue: SchedulerJob[] = []
let isFlushing = false

function queueJob(job: SchedulerJob) {
  if (!queue.includes(job)) {
    queue.push(job)
    queueFlush()
  }
}

function queueFlush() {
  if (!isFlushing) {
    isFlushing = true
    Promise.resolve().then(flushJobs)
  }
}

function flushJobs() {
  queue.sort((a, b) => a.id! - b.id!)
  for (const job of queue) job()
  queue.length = 0
  isFlushing = false
}
```

**设计优缺点：** 优点是批量更新避免了同一个 tick 内多次数据变更导致的重复渲染，性能极好。排序保证了父子组件的更新顺序正确性。缺点是异步更新意味着修改数据后不能立即拿到更新后的 DOM，必须通过 `nextTick` 等待，增加了一层心智负担。

## 8. 讲讲 Vue 3 的初始化流程（从 `createApp` 到 `mount` 经历了什么）？

- 考察点：宏观框架流程和运行机制。
- 核心解析：
  - createApp 底层调用 ensureRenderer() 获取渲染器。
  - 通过 createAppAPI 创建 app 实例，并重写 mount 方法以支持跨平台。
  - 调用 mount 时，内部会先调用 createVNode 创建根组件的虚拟节点（vnode）。
  - 然后调用渲染器的 render 方法，进而触发 patch。
  - 执行组件的 setup，编译模板，建立响应式副作用函数 effect（Vue3 叫 ReactiveEffect），最终执行真实的 DOM 挂载。

**结论：** 整体流程是"创建渲染器 → 创建应用实例 → 创建根 VNode → patch 挂载组件 → 建立响应式渲染循环"，每一步都为后续的响应式更新打基础。

**核心逻辑步骤：**

1. `createApp(App)` 底层调用 `ensureRenderer()` 获取渲染器单例。渲染器通过 `createRenderer(rendererOptions)` 创建。然后调用 `createAppAPI(render)` 生成 `app` 实例，挂载 `use`、`component`、`directive`、`mount` 等方法。
2. `app.mount('#app')` 时，通过 `normalizeContainer()` 获取真实 DOM 容器，调用 `createVNode(rootComponent)` 创建根组件 VNode，再调用 `render(vnode, container)`。
3. `render` 内部调用 `patch()`，根据 `shapeFlag` 判断走 `processComponent` → `mountComponent()`。内部依次执行 `createComponentInstance()` 创建组件实例 → `setupComponent()` 初始化 props/slots 并执行 `setup()` → `setupRenderEffect()` 创建响应式副作用。
4. `setupRenderEffect()` 用 `ReactiveEffect` 包装组件 render 函数。首次执行 render 生成 VNode 树并 `patch` 成真实 DOM。后续数据变化时 effect 的 scheduler 将更新 Job 推入异步队列。

**关键 API：** `ensureRenderer()` → `createRenderer()` → `createAppAPI()` → `createVNode()` → `render()` → `patch()` → `mountComponent()` → `setupComponent()` → `setupRenderEffect()` → `new ReactiveEffect(componentUpdateFn, scheduler)`。

**设计优缺点：** 优点是渲染器通过参数注入平台操作，实现了跨平台架构解耦。`setupRenderEffect` 将组件渲染和响应式系统桥接起来，数据驱动视图的机制非常优雅。缺点是初始化链路较长，源码阅读时跳转层次较深，调试复杂度较高。

## 9. `KeepAlive` 组件的源码实现原理是什么？

- 考察点：内置组件的底层逻辑与缓存算法。
- 核心解析：
  - KeepAlive 本质是一个抽象组件，它本身不渲染真实的 DOM。
  - 内部维护了一个缓存对象 cache（存 vnode）和一个 keys 集合。
  - 缓存策略：当组件切换时，将内部组件的 DOM 节点移入一个隐藏的容器中（不卸载），并在返回时恢复。
  - LRU 算法（最近最少使用）：通过 max 属性限制最大缓存数。当缓存超限时，会将 keys 里面最前面（最久未使用）的组件销毁掉。

**结论：** `KeepAlive` 是一个抽象组件，不渲染真实 DOM。它通过将被缓存组件的 DOM 节点移入隐藏容器（而非卸载）来实现缓存，并用 LRU 算法管理缓存淘汰。

**核心逻辑步骤：** 子组件首次渲染时正常挂载，同时将其 VNode 存入 `cache`，key 存入 `keys`。子组件切换离开时，不执行 `unmount`，而是通过渲染器的 `move` 操作将 DOM 节点移入一个隐藏的 `storageContainer`。子组件重新激活时，从 `storageContainer` 把 DOM 移回页面，触发 `activated` 钩子。缓存数量超过 `max` 时，淘汰 `keys` 中最前面的 key（最久未使用），销毁其组件实例并清理 DOM。每次访问已缓存组件时，将其 key 从 `keys` 中删除再添加到末尾，保证最近使用的始终在尾部。

**关键 API / 数据结构：** `cache: Map<CacheKey, VNode>` 存储缓存的 VNode。`keys: Set<CacheKey>` 记录缓存顺序（Set 保持插入顺序）。`storageContainer` 是通过 `createElement('div')` 创建的一个不挂载到页面的隐藏容器。`include`/`exclude` 通过 `watch` 监听变化，变化后遍历 `cache` 清除不匹配的项。

**设计优缺点：** 优点是"移动 DOM 而非销毁重建"的策略避免了重复的组件初始化和 DOM 创建开销，对频繁切换的 Tab、路由页面效果显著。LRU 策略通过 `max` 控制内存上限。缺点是被缓存的组件实例和 DOM 常驻内存，如果缓存数量大或组件状态重，内存占用会明显增加。另外 `activated`/`deactivated` 是 KeepAlive 专属钩子，增加了开发者的心智负担。

## 10. Vue 3 是如何实现跨平台渲染的（自定义渲染器原理）？

- 考察点：架构解耦设计（runtime-core 和 runtime-dom 的关系）。
- 核心解析：
  - Vue3 把核心逻辑（虚拟DOM计算、生命周期、响应式等）抽离在 runtime-core 中，与平台无关。
  - 平台相关操作（如浏览器的 DOM 增删改查、属性设置等）抽离在 runtime-dom 中。
  - runtime-dom 通过 createRenderer API，将浏览器特有的 nodeOps（如 createElement, insert, setElementText 等）作为参数传入 runtime-core。开发者也可以传入 Canvas 或小程序的操作 API，从而实现跨平台渲染。

**结论：** Vue 3 将核心逻辑（`runtime-core`）和平台操作（`runtime-dom`）彻底解耦。通过 `createRenderer` API 注入平台相关操作，实现"一套核心逻辑、多平台渲染"。

**核心逻辑步骤：** `runtime-dom` 定义一组 `nodeOps`（`createElement`、`insert`、`remove`、`setElementText`、`parentNode` 等）和 `patchProp`（处理 `class`、`style`、事件、`attrs`）→ 调用 `createRenderer(nodeOps)` 传入 `runtime-core` → `runtime-core` 内部的 `patch`、`mount`、`unmount` 等流程中所有 DOM 操作全部通过参数调用，而非直接使用 `document.createElement` → 返回包含 `render` 和 `createApp` 的渲染器对象。

**关键 API：** `createRenderer(options)` 是 `runtime-core` 暴露的核心工厂函数。`options` 包含 `createElement`、`insert`、`remove`、`setElementText`、`patchProp` 等平台方法。开发者传入 Canvas 操作 API 就能渲染到 Canvas，传入小程序操作 API 就能渲染到小程序。

```ts
import { createRenderer } from '@vue/runtime-core'

const { render, createApp } = createRenderer({
  createElement(type) {
    /* 平台创建元素 */
  },
  insert(el, parent) {
    /* 平台插入节点 */
  },
  remove(el) {
    /* 平台删除节点 */
  },
  setElementText(el, text) {
    /* 平台设置文本 */
  },
  patchProp(el, key, prev, next) {
    /* 平台处理属性 */
  },
})
```

**设计优缺点：** 优点是架构上实现了真正的关注点分离，`runtime-core` 完全不依赖浏览器环境，可测试性极强。社区可以基于此扩展到任意渲染目标（Canvas、PDF、终端 UI 等）。缺点是抽象层增加了一层函数调用开销（实际影响极小），且自定义渲染器需要实现的接口较多，开发门槛相对较高。

## 11. `v-for` 中 `key` 的作用是什么？从源码角度解释为什么不能用 index？

**结论：** `key` 是虚拟 DOM diff 算法中节点的身份标识。它决定了 Vue 是复用更新已有 DOM，还是销毁重建。用 `index` 作为 key 在列表发生插入/删除/排序时会导致大量不必要的 DOM 更新甚至状态错乱。

**核心逻辑步骤：** `patch` 函数中通过 `isSameVNodeType(n1, n2)` 判断新旧节点是否是"同一个节点"，判断条件为 `n1.type === n2.type && n1.key === n2.key`。两者都相同则走 `patchElement`/`patchComponent` 复用更新；否则卸载旧节点、挂载新节点。在 `patchKeyedChildren` 处理乱序部分时，基于新节点的 key 建立 `keyToNewIndexMap: Map<key, index>`，遍历旧节点查找对应关系，构建 `newIndexToOldIndexMap`，再对其求 LIS 确定哪些节点不需要移动。

**关键 API / 数据结构：** `isSameVNodeType(n1, n2)` 是节点身份判断的核心。`keyToNewIndexMap` 是 key 到新索引的映射，O(1) 查找。`newIndexToOldIndexMap` 记录新节点在旧列表中的位置，值为 0 表示新增节点。`getSequence()` 求 LIS 用于最小化 DOM 移动。

```ts
// 源码关键判断
function isSameVNodeType(n1: VNode, n2: VNode): boolean {
  return n1.type === n2.type && n1.key === n2.key
}

// patchKeyedChildren 中建立映射
const keyToNewIndexMap = new Map()
for (let i = s2; i <= e2; i++) {
  const nextChild = c2[i]
  if (nextChild.key != null) {
    keyToNewIndexMap.set(nextChild.key, i)
  }
}
```

**设计优缺点：**

- 不写 key（`key` 全为 `undefined`）：所有同类型节点 `isSameVNodeType` 都返回 `true`，按顺序就地复用。纯展示场景没问题，但列表项有组件状态、输入框内容时会出现状态错乱。
- 用 `index` 作为 key：列表头部插入一个元素后，所有元素的 index 都移位了，Vue 会认为 key=0 的旧节点和 key=0 的新节点是同一个，走 patch 更新而非移动。结果几乎所有节点都做了无效的 prop diff，性能跟没写 key 一样差，有状态组件同样错乱。
- 用稳定唯一 id 作为 key：id 不随列表操作改变，Vue 通过 `keyToNewIndexMap` 精确定位每个节点的新位置，配合 LIS 只移动真正需要移动的 DOM，性能最优。

额外技巧：给组件绑定一个变化的 `key` 可以强制销毁重建（key 变了 → `isSameVNodeType` 返回 `false` → 卸载旧组件、创建新组件），适用于需要重置组件状态的场景。
